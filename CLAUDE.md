# Pandacorp — 100% AI software factory

This repo is **the factory**: the company's know-how. Product code NEVER lives here.

## What's here

- `factory/constitution.md` — non-negotiable principles and standards. **Read it before any factory work.**
- `factory/standards/` — engineering standards that are injected into every project (conventions, structure, patterns, quality, **build orchestration** —how a build is planned/run: states, coarse work orders, the Build Plan, how a run stops; DR-050—, recommended stack, external services/accounts/secrets). The stack is a suggestion; it is chosen and approved in the blueprint. Default services, the account model, secrets (SOPS+age) and payments (Polar) live in `external-services.md`.
- `factory/ideas/` — idea base (one .md card per idea, frontmatter with `status`). **Your cards are personal (gitignored)**; the `_idea-template.md` template is versioned.
- `factory/profile.md` — the owner's profile (name, goals, GitHub, project path). **Personal (gitignored)**; generated by `/pandacorp:onboarding` from `profile.example.md`. The factory reads it to personalize itself.
- `factory/portfolio.md` — index of created projects and their state (pointers, not content). **Personal (gitignored)**; seed in `portfolio.example.md`.
- `factory/decisions/registry.yaml` — decision registry with pre-approved defaults
- `factory/memory/` — the factory's cross-project engineering memory: durable lessons (problem→solution, library verdicts, patterns, gotchas, anti-patterns) harvested from projects and retrieved to make future builds smarter/faster. Committed know-how like `standards/` (NOT personal data); policy in `DR-047`, design in `docs/proposals/09-self-learning-factory.md`. The self-learning loop graduates lessons into standards/rules/skills via `learn`; operate it with `/pandacorp:memory` (harvest/review/status).
- `plugin/` — the `pandacorp` plugin (skills, agents, hooks, templates). Installed via the local marketplace (see *Plugin maintenance*).
- `docs/` — vision, research and proposals
- `DECISION-LOG.md` — index of decision logs (one per area). See the **Decision log** section.
- `ideas.base` — Obsidian kanban view over the idea base

## How the factory is operated

| Action | Skill |
|---|---|
| **Configure the factory (first boot, after cloning)** | `/pandacorp:onboarding` |
| Explore/clarify a fuzzy idea in conversation (discovery) | `/pandacorp:explore [tema]` |
| Capture/crystallize an owner's idea (one-shot or from the conversation) | `/pandacorp:new-idea` |
| Search the internet for monetizable pains | `/pandacorp:discover` |
| Ask for an idea ranking/recommendation | `/pandacorp:recommend` |
| Create the project and document an idea's MVP (handoff) | `/pandacorp:spec <idea>` |
| Adopt an EXISTING external project (brownfield) — run inside it | `/pandacorp:adopt` |
| Sync the portfolio and detect moved cards | `/pandacorp:sync-portfolio` |

`/pandacorp:spec <idea>` is the **handoff**: it is run FROM the factory with the idea's name, creates the project folder/repo and documents the MVP (research + PRD + FRDs).

The other phases are run **inside the project folder**, without a name:
- `/pandacorp:design` — mockups and design system (iterate in the conversation).
- `/pandacorp:architecture` — creates the blueprint **+ work orders**.
- `/pandacorp:implement` — kicks off construction with a dynamic workflow (Dynamic Workflows) that orchestrates the subagents, followed in Mission Control. Its **last step is the hardening pass** — security audit + quality close-out + telemetry/metrics — that used to live in `release` (DR-085).
- `/pandacorp:release` — deploy/launch the product, **internal or external** (`deploy_target`), with the return-aware launch plan + the human gate for an external production deploy; sets `phase: release`, the launched/terminal phase (DR-085). The audit is no longer here — it's the last step of construction.
- `/pandacorp:review-launch` — after launch, reads real metrics vs the value hypothesis and recommends kill / hold / double-down (DR-043); runs on demand or as a `/loop` job.
- `/pandacorp:change` — **the single owner front door for ANY change** to a project (a new feature, an adjustment, or a bug). You describe it in plain language; it classifies (feature/change vs bug), gives it a class of service and files it in the project's change queue (`.pandacorp/inbox/changes/`); the running build (or the next `/pandacorp:implement`) drains it at its next safe point (DR-069). You don't have to remember which command to use — `change` routes to the right engine: `/pandacorp:iterate` (feature/change) or `/pandacorp:bug` (a bug found while testing). `/pandacorp:decide` answers a pending decision the build left in `.pandacorp/inbox/decisions.md`.
- `/pandacorp:iterate` — the feature/change engine (one work order, a new FRD, or a mini design step); reachable directly or via `change`.
- `/pandacorp:sync` — the **reverse** flow: reconcile the docs *backwards* from changes you made directly in the code (the inverse of `iterate`). Classifies the diff with the canonical-doc table and propagates reality to the owning docs + the area's decision log, behind an intent gate (DR-081).

Optional/internal: `:bug` (the bug-case engine behind `change`), `:new-version` (large milestone with a mini-PRD), `:scaffold` and `:work-orders` (steps normally invoked by `spec`/`blueprint`), `:upgrade` (re-sync a project's `.pandacorp/` overlay to the latest factory version — usually automatic, DR-048).

**Iterate without advancing (DR-032).** No manual phase (`explore`, `new-idea`, `spec`, `design`, `blueprint`) auto-advances a column: it produces its output, sets `advance_pending: true` and waits for your "ok, advance." **Re-running the same phase = keep polishing** (it refines, it doesn't regenerate or repeat what was discarded). The essence of the back-and-forth is persisted in the project's `.pandacorp/comms/iteration.md` (for ideas without a project: `factory/ideas/_drafts/<slug>.md`), so you can pick it up again even if you lose the conversation. It does not apply to `implement` (autonomous, already resumable) and does not change the production gate.

## Plugin maintenance

The plugin is installed from this repo's local marketplace (`claude plugin install pandacorp@panda-corp`, user scope). **After editing anything in `plugin/`**: (1) **bump the version** in `plugin/.claude-plugin/plugin.json` per semver (see below); (2) commit; (3) run `claude plugin update pandacorp@panda-corp` (the changes apply on session restart). Validate with `claude plugin validate plugin/`.

**Skill naming convention.** Every skill lives at `plugin/skills/<kebab-slug>/SKILL.md` (slug in English, kebab-case, **NO `pandacorp:` prefix**) and is invoked as `/pandacorp:<slug>` — the namespace prefix is automatic. The `SKILL.md` body H1 is `# /pandacorp:<slug>`. Never hardcode the prefix in the directory name or in a `name:` frontmatter field; every skill is reached uniformly as `/pandacorp:<slug>` (e.g. `plugin/skills/review-launch/` → `/pandacorp:review-launch`). The rule is also enforced in the `learn` skill (skill-authoring).

**Plugin versioning (semver, DR-034).** The plugin carries a `version` in `plugin/.claude-plugin/plugin.json` (source of truth) and **is bumped on every change** to `plugin/`:
- **PATCH** (`x.y.Z`) — fix, adjustment or doc that doesn't change the behavior of skills/agents.
- **MINOR** (`x.Y.0`) — new compatible capability (new skill or agent, new option).
- **MAJOR** (`X.0.0`) — breaking change (renaming/removing a skill, changing a flow in a way incompatible with existing projects).

Note the reason for the change in `plugin/docs/decision-log.md`. The local installation keeps tracking the commit SHA to detect drift (see below); the `version` is the release's semantic label.

Mission Control **warns about drift** (FRD-15): if there are uncommitted changes in `plugin/`, or the installed SHA (`~/.claude/plugins/installed_plugins.json`) has fallen behind the plugin's latest commit, it shows a banner with the update command. That way it's not forgotten.

## Decision log — document everything

Documenting a change is **two things, always, and in the right place**:

1. **Update the canonical doc** (the current truth) — the document that *owns* that fact, so it describes the reality as of now.
2. **Record the decision in the area's decision log** (the history) — date, *what*, *why*, linking the doc that was touched.

The canonical doc answers *"what is true now?"*; the decision log, *"how did we get here and why?"*. An FRD says what the app does today; the decision log, what changed and why. You need **both**: the FRD alone loses the why; the decision log alone leaves the FRD lying.

**Canonical doc — where each type of change is updated:**

| Change | Owning doc |
|---|---|
| App behavior/feature (what it does, acceptance criteria) | the corresponding **FRD** (`docs/frds/`); new feature → new FRD |
| Product scope/goal, metrics | the **PRD** (`docs/product/prd.md`) |
| Architecture, stack, data model, technical decision | the **blueprint** + an **ADR** |
| Visual design, tokens, components | **DESIGN.md** / design tokens |
| Plugin skill/agent/hook | the **file in `plugin/`** |
| An idea (content/state) | its **card** `factory/ideas/<slug>.md` |
| Engineering convention/standard | `factory/standards/` |
| Build orchestration (build state, WO granularity, Build Plan, how a run stops) | `factory/standards/build-orchestration.md` (policy: DR-050) |
| Recurring rule with a default | `factory/decisions/registry.yaml` |

**Decision log — one per area (the history, with the why):** index in `DECISION-LOG.md` (root).

| Area | File |
|---|---|
| Mission Control (Next.js app) | `mission-control/docs/decision-log.md` |
| Pandacorp plugin | `plugin/docs/decision-log.md` |
| Ideas (the base and its process) | `factory/ideas/decision-log.md` |
| Factory (constitution, standards, operation) | `factory/decision-log.md` |

**Rule:** on any relevant decision or change, update the **canonical doc** *and* record the entry in the **area's decision log** **before closing the turn**; link them (in the entry, the *Impact* field → which doc was updated). Most recent on top. Don't record trivial changes already obvious in the commit. Each project's `status.yaml` is written by the skills/CI, not by hand.

**Three distinct things, don't mix them:** the **canonical doc** (FRD/PRD/blueprint/DESIGN/card) is the **current truth**; the **decision log** is **history**; `factory/decisions/registry.yaml` is **policy** (rules with a default). A decision that creates a rule goes to the registry *and* to the decision log; an app change goes to its FRD *and* to the decision log. For **product projects** this discipline is injected as a propagable standard (`factory/standards/documentation.md`): every project is born with its `docs/decision-log.md` and the rule in its `CLAUDE.md`/`AGENTS.md`.

**Mission Control's Manual reflects the factory (DR-046).** Beyond the owning doc above, every change to the factory's **operable surface** — a new/renamed skill, agent, flow, gate, decision rule or standard — must also surface in **Mission Control's Manual** (the "Códice del gremio", [FRD-08](mission-control/docs/frds/frd-08-documentation/frd.md)), the navigable face of this know-how. Two mechanisms keep it honest: the **Reference** catalogs (commands/agents/decision-rules/standards) are **derived** from the canonical source (plugin skill+agent frontmatter, `factory/decisions/registry.yaml`, `factory/standards/`) so a new or renamed item appears automatically — never hand-copied; the hand-authored **Guides/Concepts** (the how-to and the why) are kept in sync by this same discipline — a change to how the factory is operated updates the affected Manual page in the same change. Don't let the Manual drift from the factory. (Today it's the HTML prototype, reconciled by hand; the Next.js app derives the Reference at build time.)

## Rules for this folder

1. Language — **the git state decides the language** (the *committed = English / gitignored = Spanish* rule). Everything **committed** is in English: code, commits, **file and folder names**, and the product/technical documents (PRD, FRD, blueprint, ADR, README, tests, and the project's `docs/decision-log.md`). Everything **gitignored** is in Spanish: communication with Pandacorp (the project summary, `.pandacorp/inbox/decisions.md`, logs, activity, the Mission Control feed, `.pandacorp/comms/iteration.md`) and the owner's personal data. `.pandacorp/status.yaml` is committed with machine state only in English (keys/enums/counters/SHAs); its human-readable prose lives in the gitignored Spanish layer. That way, whoever clones the repo sees everything in English and the owner operates Pandacorp in Spanish. **The interaction is always in Spanish:** everything the agent says to the owner — in chat and inside any skill (questions, explanations, progress, recommendations, summaries) — is in Spanish, whatever the artifact's language (the agent writes English into committed artifacts but always addresses the owner in Spanish). Details in `factory/standards/conventions.md` (Language) and `DR-009`.
2. Every recurring decision is resolved by consulting `factory/decisions/registry.yaml`. If it's not covered: escalate to the owner ONCE and codify their answer as a new rule in the registry.
3. Human gates (the owner): idea selection (run `/pandacorp:scaffold` on the chosen one; discard the rest from Mission Control), design choice, release to production, spending money, external communications, deleting data. The idea board is read-only: the states are written by the skills.
4. Agents never check off their own checks: verification is done by scripts/CI.
5. **One source of truth per state, with an explicit bridge.** A card's `status` lives ONLY in its frontmatter and covers the **pre-project lifecycle + terminal**: `discovered → recommended → in-pipeline → shipped | discarded`. Once `in-pipeline`, the card **freezes as a pointer** and does NOT duplicate the project's phase; each project's detailed state lives ONLY in its `.pandacorp/status.yaml` (`phase`). The board derives a card's **column** from both axes (pre-project → from the card status; `in-pipeline` → from the project `phase`: product→Documentada, design→Diseño, architecture→Arquitectura, implementation→building, release→shipped — DR-085: `release` is the launched/terminal phase, the old `operation` phase no longer exists; see Mission Control FRD-01/FRD-02). It never expects `design`/`architecture`/`building` to be a card status. The portfolio only keeps pointers and summaries.
6. Document everything: every relevant decision is recorded in the area's decision log (see the **Decision log** section) before closing the turn. The decision log is history; `registry.yaml` is policy.
7. **Framework vs owner's data (DR-033).** Always distinguish three planes and NEVER mix personal data with the versioned framework:
   - **The framework** — versioned and shared, the same for anyone who clones the repo: `plugin/`, `factory/constitution.md`, `factory/standards/`, `factory/decisions/registry.yaml`, the **Mission Control** app, templates and the `*.example.md` seeds.
   - **The owner's data** — personal and **gitignored**, NEVER to the public repo: `factory/profile.md` (the profile generated by `/pandacorp:onboarding`; from it the factory draws the name, goals, interests, assets, GitHub account and project path — it is the **single source** of user configuration), `factory/portfolio.md` and `factory/ideas/*.md`. Secrets/credentials don't go to the repo either (DR-037).
   - **The product projects** — in **sibling** folders/repos, outside the factory, with their own documentation in their `docs/`.
8. **Capture lessons as you work (self-learning, DR-047).** Whenever you notice something durable and reusable — a fix worth remembering, a library that worked or failed, a gotcha, a recurring pattern — in ANY skill or in plain conversation, jot a one-line candidate to `factory/memory/_inbox.md` (gitignored scratch; tag `(owner-stated)` if the owner said it, else `(agent-inferred)`). Don't polish it inline — just capture it. The `librarian` later refines the inbox into clean, deduplicated lessons in `factory/memory/` (via `/pandacorp:memory`); only the owner promotes a lesson to a standard/rule/skill. Capture freely; promotion is gated.
9. **Keep the owner's profile alive (personalization, DR-053).** When the owner reveals something durable and personal **about themselves** in conversation — an interest, a hobby, a taste/dislike, a goal, an asset/lever, or how they like to work — capture it into `factory/profile.md` (personal, gitignored, Spanish): read it first, add only what's genuinely new or update what changed, never duplicate. This is the **owner profile** (who the owner is — it personalizes `/discover`, `/recommend` and the conversation itself), a **third plane** distinct from the engineering memory (`factory/memory/`, rule 8 / DR-047) and from the decision log (history). `/pandacorp:onboarding` *bootstraps* it from past conversations; from there it grows continuously. In a product project the same rule writes to the factory's profile via the path stamped in its `.pandacorp/guide.md`. Don't ask permission to note a personal fact — just keep it current.
