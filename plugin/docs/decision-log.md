## 2026-06-16 — Coarser work orders + no same-file wave-siblings · v8.4.0

## 2026-06-16 — Freno por salud+gasto, repair-antes-de-bloquear, BLOCKED con motivo, sin ntfy · v8.2.0
**What:** The owner reviewed the v8.1 guardrail and refined how a build STOPS. (1) The build now **runs to completion** by default — `maxFrds` is no longer a per-run cap of 6; it defaults to **no cap** (Infinity) and is **opt-in for supervised TEST runs only**. A run stops only by: nothing left, the **budget ceiling** (the real token guardrail, set via the run's budget directive), a **health breaker** (3 non-`external` blocks in a row → something is systemically wrong), or work that **needs the owner**. Rationale: "stop after N features" protects neither tokens nor progress — one feature can cost 10x another. (2) **Repair before block** (the owner's rule): a failing work order or FRD gate first triggers a **repair pass** (a strong-model agent diagnoses + fixes, re-verifying with `verify.sh`); only if the fix is out of reach does the feature go `BLOCKED`. (3) **`BLOCKED` carries a `blocked_reason`**: `needs-owner` (a human must act — env var/secret, external account, product decision; also logged to `.pandacorp/inbox/decisions.md`), `external` (internet/upstream, retried later, doesn't trip the breaker), or `error` (unresolved fault). (4) **ntfy removed** (the app is poor): notifications are now macOS desktop (`osascript`) from the engine + `PushNotification` to the phone from the supervising agent. MINOR bump to **8.2.0**; `OVERLAY_VERSION` → 8.2.0.
**Why:** The owner is about to run unattended builds and wants the build to *finish the work*, not stop at an arbitrary count, while staying safe against runaway token spend and silent stalls. Health+budget stops + repair-before-block + a typed block reason give "run to completion unless something is genuinely wrong or needs me." Replaces the v8.1 hard `maxFrds=6` cap. Decided with the owner this session.
**Impact:** `plugin/templates/shared/.claude/workflows/pandacorp-build.js` (`maxFrds`→Infinity default, `attemptRepair`, `consecutiveBlocks` health breaker, `blocked_reason`, `NOTIFY` desktop-only), `plugin/skills/implement/SKILL.md`, `plugin/templates/docs/work-order-template.md` (`blocked_reason` field), `factory/standards/build-orchestration.md` (§6 "How a run stops"), `factory/decisions/registry.yaml` (DR-050 refined), `plugin/.claude-plugin/plugin.json` + `plugin/templates/OVERLAY_VERSION` → 8.2.0. Reaches a project via `/upgrade` or the part-2 MC migration (the engine runs from the PROJECT's copy). Activation: commit + `claude plugin update` + restart.

## 2026-06-16 — Faster build: high parallelism + selective verify + native notifications · v7.3.0
**What:** Three changes to the build engine (`pandacorp-build.js`), driven by the Mission Control build running ~6h at ~5 WO/h with the owner blind to stalls. (1) **Higher parallelism**: `powerful` → `wave: 8` with `split: false` (one full-stack implementer per WO, faster than test→backend→frontend), `balanced` → `wave: 4`. (2) **Selective verify** (the big win): per-WO safe points run biome + tsc + ONLY the WO's own tests, not the whole suite; the FULL `verify.sh` runs once per WAVE — a barrier that repairs cross-WO regressions — instead of once per WO. Cost drops from O(WOs × suite) to O(WOs × WO-tests + waves × suite). (3) **Native owner notifications**: the engine fires a macOS `osascript` desktop notification on the freeze, baseline-unrepairable, wave-verify-unrepairable and end-of-run paths, so the owner learns when the build blocks, can't self-repair, or finishes — no more silent stalls. The end-of-run notify ALWAYS fires (built/blocked/pending counts). MINOR bump to **7.3.0**; `OVERLAY_VERSION` → **7.3.0**.
**Why:** Owner feedback after the MC dogfood: the build was far too slow (the global per-WO verify dominated; only 3 WO ran in parallel) and gave no signal when something needed attention (a blocked WO or pending decision never reached them). Selective verify + higher wave target ~3-4× throughput; the notifications close the visibility gap until Mission Control's live view exists. Decided with the owner ("máximo" speed + "notificación nativa + yo vigilo").
**Impact:** `plugin/templates/shared/.claude/workflows/pandacorp-build.js` (PROFILES waves/split, `verifyWO` selective, per-wave full-suite barrier, osascript notifications; re-copied to `mission-control/.claude/workflows/`), `plugin/templates/OVERLAY_VERSION` → 7.3.0, `plugin/.claude-plugin/plugin.json` → 7.3.0, `mission-control/.pandacorp/status.yaml` `overlay_version` → 7.3.0. Trade-off: a selective per-WO verify can miss a cross-WO regression until the wave barrier — acceptable, the barrier catches it cheaply. Activation: commit + `claude plugin update pandacorp@panda-corp` + restart.

## 2026-06-16 — pandacorp-build self-heals a red baseline (deadlock breaker) · v7.2.0
**What:** Added a **Baseline self-heal phase** to the build engine (`pandacorp-build.js`), running BEFORE the Plan. `verify.sh` is global, so a single test left red by a prior run — an adversarial test the reviewer added after a WO closed/was-blocked, broken code from a WO that exhausted its 2 review cycles, or a committed regression — freezes the WHOLE build: every new WO fails its own verify gate on someone else's broken test, the ready-queue drains, and the run stalls silently as a phantom "empty queue". The new phase runs `verify.sh` first; if green it changes nothing; if red it spawns an `implementer` (judge model) to fix the PRODUCTION code (never weaken tests) until the suite is green and commit — or, if it genuinely can't, return red and STOP for the owner instead of stalling. MINOR bump to **7.2.0**; `OVERLAY_VERSION` → **7.2.0** (machinery changed compatibly; existing projects pick it up on `/upgrade`).
**Why:** Building Mission Control hit this deadlock 3× (WO-17-001 `parseProjects`, WO-05-001 work-order status parse) — each time the owner's session had to repair the baseline by hand before the build could move. The engine should do that itself. Root cause is structural (a global verify gate + the reviewer adding adversarial tests in a step separate from the WO), captured in `factory/memory/_inbox.md`; this is the engine-side mitigation. Decided with the owner ("arreglo el motor primero").
**Impact:** `plugin/templates/shared/.claude/workflows/pandacorp-build.js` (new `phase('Baseline')` + repair agent; re-copied to `mission-control/.claude/workflows/`), `plugin/templates/OVERLAY_VERSION` → 7.2.0, `plugin/.claude-plugin/plugin.json` → 7.2.0, `mission-control/.pandacorp/status.yaml` `overlay_version` → 7.2.0. Activation: commit + `claude plugin update pandacorp@panda-corp` + restart.

## 2026-06-16 — Feature-centric docs structure (the FRD as a self-contained module) · v7.0.0 (MAJOR)
**What:** Rewrote the whole flow to the **feature-centric** documentation model (DR-049): each FRD is a self-contained module folder `docs/frds/frd-NN-<slug>/` (`frd.md` + conditional `fdd.md`/`mocks/` + per-feature `blueprint.md` + `work-orders/`), under a thin product layer (`docs/product/{prd.md,architecture.md,research.md}`) and a global design system (`docs/design/` = the PDD). **Why:** The owner observed the flat model caps at scale (a single blueprint monolith; growth forced a mid-project flat→nested migration) and adopted the JobLeap arrangement after a benchmark (JobLeap vs PandaTrack). Feature-centric gives locality of change — a new feature is a new folder, nothing else moves. See `factory/decision-log.md` (2026-06-16) + `docs/proposals/10-feature-centric-doc-structure.md`.
