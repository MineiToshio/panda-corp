# DESIGN — Replace with the product title

> The **frozen design contract** for this product. One design system per project; features consume
> it, never fork it. The canonical tokens live in `docs/design/design-tokens.json` — this document
> explains the *intent* and the *rules*, the JSON is the source of values.

## Identity & mood

The personality of this product (3–5 adjectives) and who it's for. What it should feel like — and
what it must **not** feel like. Each app has its own identity; never reuse another project's look.

## Tokens

Pointer to `docs/design/design-tokens.json` (palette, typography, spacing, radii, elevation). Note
the default theme (light/dark) and any brand anchor colors. **Never hardcode a visual value** that
isn't a token.

## Usage rules

How tokens combine: semantic colors (background/foreground/primary/destructive…), type scale,
spacing rhythm, radii, elevation. The do's that keep screens coherent.

## Components

The allowed component base (e.g. shadcn/ui as the accessible base, restyled to this identity) and a
pointer to the living inventory `docs/design/components.md`. **Reuse before creating** (DR-057).

## Prohibitions

The explicit don'ts: arbitrary values, off-token colors, ad-hoc components that duplicate the
inventory, fixed `bg-white`/`text-black`, etc.

## Theming & accessibility

Light/dark as first-class, and the baseline a11y contract (WCAG 2.2 AA: contrast, focus, touch
targets). Feature-specific a11y notes go in that feature's `fdd.md`.

## Voice

Pointer to `docs/design/voice-and-tone.md` (the product's writing voice), if present.
