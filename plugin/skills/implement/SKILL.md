# /pandacorp:implement

**This is the command that starts (and resumes) the build.** It launches a **dynamic workflow** (`pandacorp-build`, a native Claude Code JS script that runs in the background) that builds **FRD by FRD** (DR-050): for each feature it reads its blueprint's **Build Plan** (the work-order DAG) and each work order's frontmatter **`implementation_status`** (the source of truth), builds the `PLANNED`/`IN_PROGRESS` ones, and runs **one review + integration gate per FRD** → `VERIFIED`. It runs IN the project. On start: `.pandacorp/status.yaml → phase: implementation`, `running: true`; when it stops/finishes, `running: false`.

> **Preflight (DR-045) — is this a Pandacorp project?** This skill mutates the project, so first confirm the marker `.pandacorp/status.yaml` exists. If it's missing, STOP and tell the owner (in Spanish) that this folder isn't a factory project yet — `/pandacorp:adopt` brings an existing one in, `/pandacorp:spec` creates a new one. Then, if `overlay_version` is behind the plugin's `OVERLAY_VERSION`, run `/pandacorp:upgrade` first (DR-048) so the project's `.claude/workflows/pandacorp-build.js` and structure are current. Don't proceed over a missing/stale structure.
>
> **Concurrent-run guard (DR-050 §9) — only one build per project at a time.** After confirming `status.yaml` exists, read `running`, `run_started_at` and `supervisor_heartbeat`:
> - `running: false` (or field absent) → proceed.
> - `running: true` AND `supervisor_heartbeat` is **less than 10 minutes old** → **ABORT**. Tell the owner in Spanish that there is already an active build (include `run_started_at` so they know when it started). Do NOT launch the workflow.
> - `running: true` AND `supervisor_heartbeat` is **10 minutes old or older** (or the field is absent) → **stale lock**: the supervisor died (internet cut, crash, etc.). Warn the owner, reset `running: false` and clear `supervisor_heartbeat` in `status.yaml`, then proceed normally — the frontmatter state is still valid and the engine resumes without re-doing `VERIFIED` work.

`$ARGUMENTS` optional: a **mode** (`pro` | `balanced` | `powerful` | `deep`; default `powerful`); `maxFrds` (**opt-in** cap on features **processed** per run — built + blocked + **reopened**, so reopens count — for **supervised TEST runs**); **`maxAgents`** (a hard cap on subagents spawned this run — **the real guardrail for overnight runs**: counted inside the engine, it brakes reliably even if `budget.spent()` under-counts subagent work or the supervisor dies); and `maxSpend` (an output-token ceiling via `budget.spent()` — a **secondary** ceiling, since it under-counts the heavy subagent work).

> **Engine = Dynamic Workflows, not Agent Teams** (DR-013). The per-FRD loop lives in the **script's code**, not in messages between peer agents. **Resumable by construction**: state lives in the work-order frontmatter + commits, so a re-launch reads `implementation_status` and **never rebuilds a `VERIFIED` work order** (DR-050) — no re-work.

## Execution modes (consumption/quality control)

Control the **concurrency and models** of the workflow (DR-014), not "team size":
- **pro**: minimum consumption. Up to 2 work orders at a time, economical models. A single full-stack `implementer` (no split). The `reviewer` still gates each FRD.
- **balanced** (Max 5x): up to 4 work orders in parallel within a feature; judge (review) on opus, workers on sonnet.
- **powerful** (**default**, Max 20x): up to 8 in parallel — the default; finishes sooner.
- **deep**: best model everywhere, split team, extra adversarial review. Slowest/most expensive (it intentionally narrows the wave — opus + split is heavier per slice, so fewer run at once).

**How a run stops (DR-050, owner decision 2026-06-16).** By default the build runs to **completion** — it does NOT stop after N features ("stop after N" protects neither tokens nor progress: one feature can cost 10x another). A run stops only when: (a) **nothing is left** to build, (b) the **budget ceiling** is reached — `maxAgents` (a hard cap on subagents spawned, **the real overnight guardrail**) or a `+Nk` turn directive / `maxSpend`; **for overnight runs always pass `maxAgents`** (counted inside the engine, it brakes even if `budget.spent()` under-counts or the supervisor dies), (c) **too many features block in a row** (a health breaker), or (d) what remains **needs you**. `maxFrds` bounds a supervised **test** run and counts features **processed** — built + blocked + **reopened** — so a reopen counts and chained reopens can't slip past the cap (a real bug the 2026-06-16 overnight test caught). `maxFrds` is never the overnight guardrail; `maxAgents` is.

## Unattended operation — the build supervisor (run and walk away SAFELY)

The owner runs `implement` and leaves, even overnight. For that to be SAFE — **no silent stalls, no burning the week's tokens** — launching the build is **never** just firing the workflow. It is **two things, always**:

1. **Launch + set the ceiling**: write to `.pandacorp/status.yaml`: `running: true`, `run_started_at: <ISO timestamp of now>`, and clear any previous `supervisor_heartbeat` (so the concurrent-run guard starts with a fresh baseline). **Also drop the phase-aware build lock (DR-063): `mkdir -p .pandacorp/run && touch .pandacorp/run/build.lock`** — its freshness tells the Stop gate (`verify-before-stop.sh`) a build is active so it stops running the strict whole-project suite on every turn (the engine owns enforcement per-FRD + at close-out; `.pandacorp/run/` is gitignored runtime state). Then `Workflow({ name: "pandacorp-build", args: { mode, maxAgents } })` — pass `args` as a real **JSON object**, never a string. **VERIFY the args landed (DR-072 R2): the engine's FIRST log line echoes `· maxAgents <N> ·`. If it reads `maxAgents OFF` when you passed one, or `args arrived as a <type>, NOT an object`, the args were DROPPED (a Workflow-tool serialization bug seen 2026-06-19) and the run is UNBOUNDED → `TaskStop` it immediately and relaunch (re-pass args as an object, or persist the script and relaunch via `{scriptPath}` with the scope hardcoded).** For an **overnight run always pass `maxAgents`** (a hard cap on subagents spawned) — that is the real guardrail: counted inside the engine, it survives a dead supervisor and does not depend on `budget.spent()`. Optionally add `maxSpend` as a secondary `budget.spent()` ceiling; add `maxFrds` only for a short supervised test. Each FRD verified is an immutable commit (the safe point is a commit, never an agent state).
2. **Mount the supervisor** (this is the part that was missing and let a build run >1h unwatched):
   - **Live watch (`Monitor`)** — a bash poll (~2 min; **costs no tokens** — only an emitted *event* wakes the agent) over `.pandacorp/status.yaml` (FRD progress), the run's journal mtime + active agents (frozen?), and the agent count (spend proxy), **and the working tree's git health** (`git status --porcelain` — any `UU`/unmerged path or a `<<<<<<<` conflict marker means a BROKEN tree, a first-class failure signal, DR-067). It emits an event on: a FRD verified, **frozen** (no activity ~15 min), a new block, **a broken working tree** (unmerged paths / conflict markers, DR-067), **pace below target**, or run end. Cover the failure signatures, not just success ("silence is not success"). **On every tick, write `supervisor_heartbeat: <ISO timestamp>` to `status.yaml` AND refresh the build lock (`touch .pandacorp/run/build.lock`, DR-063)** — both are liveness signals: as long as the supervisor is alive the heartbeat stays fresh (< 10 min) and the Stop gate keeps standing aside; if the supervisor dies, the heartbeat AND the lock go stale together and the next `implement` auto-clears the lock while the Stop gate auto-re-engages (same 10-min TTL).
   - **Positive heartbeat to the event stream (observability fidelity, DR-066).** A frozen `running: true` is **not** proof of life — a monitor reads liveness as `running AND fresh`, so the supervisor must keep the freshness signal genuinely alive, not just flip a flag. On **every tick** (and on the time-driven beat below), besides writing `supervisor_heartbeat`, **append a positive heartbeat event to the stream** so "sin señal" (no events at all) truly means *hung*: `printf '{"event":"SupervisorTick","at":"%s","project":"%s","data":{"running":true}}\n' "$(date -u +%FT%TZ)" "$(basename "$PWD")" >> ~/.claude/dashboard-events.ndjson` (fire-and-forget). The **engine** already advances `last_event_at` in `status.yaml` at each safe point (FRD gate, sync) and appends `AgentWorking` events per agent — together they are the producer's positive heartbeat. The consumer (Mission Control) crosses `running` with this recency and declares its own freshness (en vivo / datos de hace X / sin señal). See `factory/standards/observability.md` "Liveness & freshness fidelity".
   - **Health heartbeat — a VISIBLE message to the owner every ~20–30 min, even with NO anomaly.** The Monitor only wakes on events, so the supervisor must schedule this latido SEPARATELY (a `ScheduleWakeup` timer) — never rely on events for it, or a healthy-but-quiet build looks dead. **The same time-driven tick also advances `supervisor_heartbeat` + appends a `SupervisorTick` event (DR-066)** so the freshness signal never freezes between long, quiet agents (the exact audit failure: the build advanced while the heartbeat stayed frozen). Each heartbeat states: WOs done/total, the FRD in progress, rough spend so far, and "all green". **Silence reads as "stuck" to the owner** — a periodic "still going, X/Y done, ~Nk output spent, all green" is the entire point.
   - **Notifications** — the engine fires a macOS desktop notification (`osascript`) on every alarm; the supervising agent ALSO sends a `PushNotification` to the phone (when Remote Control is on). No third-party push app (owner decision 2026-06-16). The owner is asleep, so reach BOTH the desktop and (if available) the phone.
   - **Announce each FRD (milestone)** — on every FRD **verified**, send the owner a visible message (+ push): the FRD name, one line on WHAT it does, the **route/page to try it**, and X/Y done. Don't make the owner ask "is it done yet?" — tell them at each milestone, and point at the review worktree (below) so they can actually open it.
   - **React to every event, repairing before giving up** — a stall → unstick it; a **broken working tree** (unmerged paths / `<<<<<<<` markers) → **restore it to `last_green_sha`** (a clean reset to the green commit — bounded auto-repair; NEVER hand-resolve a stash-pop, DR-067) and let the run rebuild the lost between-safe-point work; a failing feature → the engine itself runs a **repair pass** (diagnose + fix) before it ever blocks. Only when a fix is genuinely out of reach does the feature go **BLOCKED with a reason** and the build moves on to independent features: `needs-owner` (a human must act — env var, secret, external account, product decision; also logged to `.pandacorp/inbox/decisions.md`), `external` (internet/upstream — retried later), or `error` (unresolved technical fault). The supervisor escalates `needs-owner` to you immediately; for a budget/health stop it **stops the build and notifies** with the detail ("blocked at FRD-X because Y, come decide").
   - **Self-learning of avoidable stalls (DR-047)** — when the supervisor unsticks something, **classify it**: uncontrollable (internet, server 500) → nothing to learn, don't capture; **avoidable** (a recurring pattern, an engine bug, a config) → capture the lesson to `factory/memory/_inbox.md` and, if fixable factory-wide, **fix it on the spot** so it never recurs. Every avoidable stall makes the factory better.
   - **Budget ceiling** — the engine stops at `maxAgents` (counting the subagents it spawns, checked at every FRD **and build-wave** boundary — DR-070, the fine-grained brake) and `maxSpend` is a secondary `budget.spent()` ceiling. **The in-engine counter is NOT trusted alone (DR-070):** it once silently under-counted and let a `maxAgents=40` run reach **68 agents / ~5M tokens** (MC Phase-2 pass-1, 2026-06-21). So the supervisor enforces a **reliable EXTERNAL brake**: each Monitor tick it counts the actual `agent-*.jsonl` files in the run's transcript dir (`ls .../subagents/workflows/<run-id>/agent-*.jsonl | grep -v meta | wc -l`) and, past `maxAgents` (allow a small overshoot for the in-flight wave), it `TaskStop`s the workflow + runs the guaranteed shutdown — independent of the engine counter and of `budget.spent()`. The supervisor also treats `maxAgents` as the **cumulative-across-passes** walk-away budget: it does **not** relaunch the next pass once cumulative agents reach the ceiling — it stops and notifies the owner with the balance. Always pass `maxAgents` for an unattended run — and remember one pass over many FRDs can legitimately cost ~3–5 agents/FRD, so size the cap to the FRDs you expect to process, not to a single number that one pass blows through.
   - **Relaunch for multiple passes** — a run makes ONE pass over the pending FRDs; WOs a gate **reopens are deferred to the NEXT run** (never retried in a loop). To finish the project the supervisor **relaunches after each run** (it reads the frontmatter, so it never rebuilds a `VERIFIED`) until nothing pending/reopened remains or the `maxAgents` ceiling is hit. **Relaunch the WORKFLOW directly** (`Workflow({ name: 'pandacorp-build', args: { mode, maxAgents } })`), NOT `/implement` — you already hold the run lock (DR-050 §9), so re-invoking the skill would trip its own concurrent-run guard and abort. Expect **several passes** when gates reopen often — that's by design, not a fault.
   - **Stop the LOOP vs stop a run (DR-068).** Stopping a single workflow **run** (TaskStop / the run ends) does NOT stop the build — the supervisor's job is to **relaunch the next pass**, so a lone stop just triggers another pass (the exact confusion the owner hit: "I stopped it but it came back"). To stop the **whole build**, the owner sets the stop signal — **`rethink_pending: true` in `status.yaml`** (or drops `.pandacorp/run/stop`); the supervisor checks it **at each safe point AND before every relaunch**, and when set it **halts the loop** (does not relaunch) and runs the guaranteed shutdown below. Surface this to the owner: "to stop the build, set `rethink_pending` / run the stop signal — not just stop a run."
   - **Guaranteed shutdown — `running: false` on EVERY exit, never a lying flag (DR-068).** Whenever the supervisor's loop ends FOR ANY REASON (nothing pending/reopened, budget/health/needs-owner stop, owner stop signal, or the supervisor itself wrapping up/being interrupted), its **guaranteed LAST act** is to clear state: write `status.yaml: running: false` + **remove the build lock (`rm -f .pandacorp/run/build.lock`)**. This is NOT optional or "only on a clean end" — a stopped build that still says `running: true` is a **lying flag** (the owner saw a stuck spinner over a build dead ~1h). Belt-and-suspenders: the engine spawns a tiny **close-out agent** on normal completion that sets `running: false`; and **a stale `running: true` is NEVER trusted as alive** — any reader (the concurrent-run guard, the supervisor's own health tick, Mission Control's display) crosses `running` with `supervisor_heartbeat` recency (DR-066) and reads a stale flag as STOPPED, and the next state-writer auto-corrects it to `false`. (If the supervisor is hard-killed without cleaning up, the lock + heartbeat go stale after 10 min and the next `implement` auto-clears them — the explicit shutdown is the tidy path, the TTL + recency-reconciliation are the safety nets.)
- **Auto mode (permissions), not babysitting**: the owner activates Claude Code's *auto mode* (Shift+Tab, or `defaultMode: "auto"` in THEIR `~/.claude/settings.json`). NEVER `--dangerously-skip-permissions`.
- **Review worktree — let the owner SEE the latest green version without touching the build.** Keep a worktree pinned at the last green under the **canonical review-worktrees root** (DR-090 — `/Users/Shared/review-worktrees/<project>`, a `Shared`-level sibling of `Proyectos/`, named after the project, NOT inside `Proyectos/` and NOT inside `.claude/`): `git worktree add /Users/Shared/review-worktrees/<project> <last_green_sha>` with its OWN `node_modules` (`pnpm install` there — NOT a symlink, which breaks Next's HMR/watcher), and update it (`git -C /Users/Shared/review-worktrees/<project> checkout <new last_green>`) each time an FRD verifies. The build keeps running untouched on the main checkout. To view it, the owner opens a **Claude Desktop session in that worktree** (its Preview button serves it natively — the build session's Preview can't point at an external worktree, confirmed) OR asks the supervisor to bring up its dev server on a fixed port **on-demand** and hands back the URL. Do NOT keep a permanent dev server that restarts on every FRD (Next's file watcher is flaky across `git checkout`). Subagents that must mutate in parallel can use `isolation: 'worktree'`.

## How it runs (per-FRD)

The script: **baseline self-heal** (FIRST: if the working tree is dirty/conflicted — unmerged paths or `<<<<<<<` markers, e.g. an app-restart or a kill left a run mid-write — **restore to `last_green_sha`** and clear stale build stashes + leftover temp `preview-wo*` pages, DR-067; THEN if `verify.sh` is still red, repair it before planning) → **plan** (read every FRD's frontmatter state + Build Plan; return the non-`VERIFIED` FRDs in cross-FRD order) → for each FRD (to completion, unless budget/health stops the run): build its `PLANNED`/`IN_PROGRESS` work orders in waves per the Build Plan (each: implement → fast self-test on its OWN tests → `IN_REVIEW` + `## Status Note` hand-off) → **FRD gate** (one review + integration test + **focused** `verify.sh --since <last_green>` —biome+tsc global, only the tests affected this cycle; the full suite runs once at close-out— → all `VERIFIED`; or reopen specific WOs to `PLANNED` for the next run; or, after a failed **repair pass**, `BLOCKED` with a reason) → **close-out + notify**. App-agnostic: it depends only on the docs' state and the project's `verify.sh`.

## Per-FRD loop (what each feature goes through)

1. **Build** each work order (coarse — a whole view/capability): set its frontmatter `IN_PROGRESS`, implement with TDD, run its **own** fast self-test (not the whole suite), set `IN_REVIEW`, write its `## Status Note` hand-off (interfaces exposed, integration seams) so the next agent reads that, not all the code.
2. **FRD gate** (`reviewer`, a different model from the generator): review the WHOLE feature with the 3 lenses + write adversarial tests the implementers didn't see (DR-015), exercising the work orders **together** (integration); run the **focused** `verify.sh --since <last_green>` (biome+tsc global + only the tests affected this cycle; the full suite runs once at close-out). Green → every WO + the FRD frontmatter → `VERIFIED`, commit. A specific WO wrong → **DR-073 PATCH-FIRST: the engine attempts ONE in-place opus patch of the reviewer's specific finding(s) + RED-proven failing test on the EXISTING build, re-gated WHOLE-PROJECT (full FRD tests + a whole-project knip+biome+tsc) — VERIFY on clean; only if the patch can't green it does it revert + reopen those WOs to `PLANNED` (retry next run from a clean base) and bump `reopen_count` (the single attempt budget, capped by `MAX_REOPENS` → BLOCK needs-owner).** Otherwise the engine runs a **repair pass** (diagnose + fix); only if that fails does the feature go `BLOCKED` with a `blocked_reason` (`needs-owner` | `external` | `error`) + notify, and the build continues with independent features. **Model escalation (DR-073):** a WO marked `difficulty: high` builds on opus a-priori, and any WO with `reopen_count >= 1` escalates to opus empirically (it already failed once) — never below the mode's floor; `maxAgents` counts each opus spawn as ~3 cost-units so escalation can't silently blow the budget.
3. **Drain the change queue at the safe point (DR-069) — how the owner talks to a running build.** Read `.pandacorp/inbox/changes/` (the unified queue; the owner files items with `/pandacorp:change`, already classified by `type` + `class`). The change was written by a detection-free door, so it's been waiting safely — now the build (the only thing that touches the docs) integrates it **at this safe point, never mid-feature**. **Take ONLY `status: ready` items; SKIP `status: draft`** (the owner is still working it out — leave the file untouched, the build never acts on a draft, DR-069). For each `ready` change, route **work-conserving** (never stall the whole build for one item):
   - **`expedite`** first, then **`standard`** FIFO.
   - **autonomous-resolvable** (a clear scoped change) → integrate it via the **`iterate`** engine (feature/change → FRD/work-orders/blueprint, calling `work-orders`/`design`) or the **`bug`** engine (regression test → fix), then it passes the **FRD review/test gate** like planned work (same quality).
   - **needs-owner** (product/money/irreversible/design decision) → write it to `.pandacorp/inbox/decisions.md`, notify, and **DEFER + continue with independent FRDs**; only fully STOP if it blocks everything.
   - **structural / fundamental / new-design** → pause (`rethink_pending`) + tell the owner the command to run (e.g. `/design` for a visual — the design gate is theirs); don't silently rebuild half the app.
   - **Complete = verify the durable record, then ARCHIVE (never hand-delete, DR-069).** When a change lands, **independently** (generator ≠ verifier, constitution §22) confirm its durable record exists: the landing commit **touched the canonical doc** (the FRD/PRD/blueprint) AND a `docs/decision-log.md` entry references the change. Only then stamp `status: done` + `shipped_sha` + `shipped_at` and **`git mv`/move the file into `changes/done/`** (the active `changes/` stays pending-only — the owner's clean queue; the file is **moved, never `rm`'d** — the folder is gitignored, so a delete is irreversible). If the durable record is missing → **do NOT archive**: leave it in place, flag the doc gap (the missing decision-log/FRD update is the real defect to fix). Decrement `pending_changes`. **Never auto-delete** `done/`; any prune is a separate owner-gated, reversible action.
   Also honor `rethink_pending: true` (stop cleanly + notify) and apply answered `.pandacorp/inbox/decisions.md`. (The former `inbox/bugs/` folds into `changes/` as `type: bug`.)

## Real-time documentation (Mission Control reads it live)

Keep updated: each work order's frontmatter `implementation_status`; `.pandacorp/status.yaml` (`progress`, `running`, per-status counts, `last_green_sha`, `safe_to_test`); `.pandacorp/comms/progress.md` (append-only log — what was done, decisions, problems; allows resuming with no prior context). If something deviates from plan, document it; don't hide it.

## Decision points (escalation, visible in Mission Control)

When something unplanned appears: **first investigate** (delegate to `researcher`); if you can take the coherent decision yourself, do so and document it. **Only escalate genuinely human decisions** (product scope, irreversible, spending money, or what the registry marks human): note it in `.pandacorp/inbox/decisions.md` as `pending` (what's happening, options, **your recommendation**), notify the owner, and continue with independent FRDs. The owner answers with `/pandacorp:decide`.

## Final hardening step — the audit moves INTO construction (DR-085)

Construction's **last step**, once every FRD is `VERIFIED`, is the hardening pass that used to live in `/pandacorp:release`. The build is not "done" until it is **secure, quality-gated, audited and instrumented** — that is now part of building, not a later release concern:

1. **Security audit** (`security-auditor` agent): essential web OWASP + **OWASP Top 10 for Agentic Applications** (ASI01–ASI10, DR-017) if the product has agents/LLMs with tools. Critical/high findings are fixed (via fast work orders through the FRD gate) **before construction is declared complete** — never deferred to release.
2. **Telemetry / metrics verification** (`analytics` agent): verify the events in `docs/analytics/events.md` actually fire in the critical flow (evidence, not "they should") and leave the value-hypothesis funnel documented. The product is instrumented as part of the build (DR-085) — metrics are added here, during construction, not as a post-launch operation step.
3. **Quality close-out**: the full suite + e2e green, lint + typecheck clean, the responsive / shell-presence / visual gates green (the per-FRD gates already enforced them; this is the whole-project confirmation).

Only when this hardening is green is construction complete.

## When all are done
Full suite + e2e green **and the final hardening step above complete (audited, instrumented, quality-gated — DR-085)**, then `.pandacorp/status.yaml → phase: release`. (There is no `operation` phase — reaching `release` means the build is ready to be launched by `/pandacorp:release`, internal or external.) **Harvest the lessons now (DR-047) — close-out is when they're freshest:** run a `librarian` harvest of this project (drain `.pandacorp/run/lessons.md` + the build's capture points — reviewer findings seen ≥2×, fixed bugs, library verdicts chosen in the blueprint — into `factory/memory/` as evidence-anchored candidates, A.U.D.N.-deduped via `/pandacorp:memory harvest <project>`), so the NEXT build inherits what this one learned instead of relearning it; the periodic review sweep reconciles them and proposes promotions to you. This closes the self-learning loop at its highest-signal moment (the always-on capture is the supervisor's job during the run; this is the refine step). Summary to the owner + next step `/pandacorp:release` (the deploy/launch — internal or external).

## Rules
- Never advance with red tests. A WO that can't pass: the engine first attempts a **repair**; only if that fails = **freeze-on-red** (don't commit broken, leave HEAD at `last_green_sha`, mark `BLOCKED` + `blocked_reason` in the frontmatter, notify, continue with independent features). The build runs to completion; it stops only by budget/health or when something `needs-owner`.
- **State is the frontmatter** (`implementation_status`), never body prose or git (DR-050). `VERIFIED` is never rebuilt.
- **Never `git stash` to preserve in-flight work across a kill / relaunch / stop (DR-067).** The build is resumable from the frontmatter + commits, so killing or stopping a run discards only the uncommitted work between two safe points — the relaunch rebuilds it from `PLANNED`/`IN_PROGRESS` state, never re-doing `VERIFIED`. A `git stash` of in-flight work, popped later onto a tree that has moved on, **conflicts** (the MC Phase-2 incident: a stale `frd02-lacampana` stash applied onto an already-`VERIFIED` FRD-02 left `<<<<<<<` markers in `Button.tsx`, broke the gate with 130 errors, and stalled the build ~1h while the supervisor — blind to `git status` — kept reporting "healthy"). The only safe recovery from a dirty/conflicted tree is to **restore it to `last_green_sha`** (a clean reset to the green commit), never a stash-pop. **Drop any pre-existing build stashes on startup** and clean leftover temp `preview-wo*` pages — they're stale; the work is resumable.
- Quota limits: lower the mode (fewer parallel agents / cheaper models). Not a code error.
- If a work order reveals the blueprint/FRD was wrong, document the conflict, fix the source doc (ADR if architectural), then continue.
- **Don't trust the worker's honesty** (constitution §22): the `reviewer` re-verifies everything; the environment is fail-closed. Never relax verification because "the agent said it passed".
