# /pandacorp:implement

**This is the command that starts (and resumes) the build.** It launches a **dynamic workflow** (`pandacorp-build`, a native Claude Code JS script that runs in the background) that builds in **GLOBAL WAVES** (DR-050 + BL-0021): each wave takes the READY work orders of **all** FRDs — `dependsOn` satisfied, artifacts disjoint (DR-060), capped at the mode's wave size — so independent features build **in parallel** instead of single-file; it reads each blueprint's **Build Plan** (the work-order DAG) and each work order's frontmatter **`implementation_status`** (the source of truth), builds the `PLANNED`/`IN_PROGRESS` ones, and runs **one review + integration gate per FRD** → `VERIFIED`, gates **serialized at wave boundaries** (they always see a quiet tree). It runs IN the project. On start the launch SOP (§Unattended operation, step 1) writes `.pandacorp/status.yaml → phase: implementation` (the ONE place the architecture→implementation transition is set — no manual skill touches `phase`, DR-032) and `running: true`; when it stops/finishes, `running: false`.

> **Preflight (DR-045) — is this a Pandacorp project?** This skill mutates the project, so first confirm the marker `.pandacorp/status.yaml` exists. If it's missing, STOP and tell the owner (in Spanish) that this folder isn't a factory project yet — `/pandacorp:adopt` brings an existing one in, `/pandacorp:spec` creates a new one. Then, if `overlay_version` is behind the plugin's `OVERLAY_VERSION`, run `/pandacorp:upgrade` first (DR-048) so the project's `.claude/engines/pandacorp-build.js` and structure are current. Don't proceed over a missing/stale structure.
>
> **Readiness re-check (DR-100/DR-102) — the gates' evidence, asserted for real (audit-20 P1-1).** Before launching, verify the plan actually passed its gates — this used to be a claim four docs made with no mechanism behind it; now it IS the mechanism: (a) every per-FRD `blueprint.md` with `status: ACTIVE` carries the **`readiness_gate: passed …`, `grounding_gate: passed …` and `consistency_gate: passed …` stamps** (written by `architecture` step 9b2 when its fresh-agent gates passed — an ACTIVE blueprint without them never passed the gates: STOP and route back to `/pandacorp:architecture`); (b) `grep -rn "NEEDS CLARIFICATION" docs/` finds no marker in an ACTIVE doc (the DR-100 blocker — `verify.sh` also REDs on it, this is the cheap pre-launch echo); (c) work orders to build have `status: ACTIVE` (a DRAFT WO is un-gated — don't build it). A legacy project whose ACTIVE blueprints predate the stamps: warn the owner once and offer to run the gates retroactively via `/pandacorp:architecture` — don't silently build.
>
> **Concurrent-run guard (DR-050 §11) — only one build per project at a time.** `preflight-implement.sh` (Launch checklist step 1) mechanizes this; the rule it applies, crossing the PRODUCER's own clock (DR-066) — after confirming `status.yaml` exists, read `running`, `run_started_at`, `supervisor_heartbeat` AND `last_event_at`:
> - `running: false` (or field absent) → proceed.
> - `running: true` AND **EITHER `supervisor_heartbeat` OR `last_event_at` is less than 10 minutes old** → **ABORT**. Tell the owner in Spanish that there is already an active build (include `run_started_at` so they know when it started). Do NOT launch the workflow. (Crossing `last_event_at` in is what stops a build with a quiet supervisor but a live engine from being misread as dead.)
> - `running: true` AND **BOTH `supervisor_heartbeat` AND `last_event_at` are 10 minutes old or older** (or both absent) → **stale lock**: the supervisor died (internet cut, crash, etc.). Warn the owner, reset `running: false` and clear `supervisor_heartbeat` in `status.yaml`, then proceed normally — the frontmatter state is still valid and the engine resumes without re-doing `VERIFIED` work.

`$ARGUMENTS` optional:
- **mode** (`pro` | `balanced` | `powerful` | `deep`; default `powerful`) — controls concurrency and model quality.
- **`frds`** — one or more FRD identifiers to target. The workflow normalizes the input automatically, so all these formats are valid: `'frd-05-settings'` (folder name), `'docs/frds/frd-05-settings'` (relative path), `'docs/frds/frd-05-settings/frd.md'` (path to file). When set, the engine builds **only those FRDs** and skips everything else. If any requested FRD has cross-FRD deps that are not yet `VERIFIED`, the build **refuses to start** and tells you which deps to implement first. **Mutually exclusive with `change`** — if `change` is set, `frds` is derived from it automatically. The launcher-printed `Workflow()` call carries this field plus `projectDir`, `project`, `leaseToken` and `leaseEpoch`; never hand-construct the call.
- **`change`** — a change from `.pandacorp/inbox/changes/` to process and build in one targeted run. All these formats are valid and normalized automatically: `'mc-observability-consumer-dr066'` (slug), `'mc-observability-consumer-dr066.md'` (with extension), `'.pandacorp/inbox/changes/mc-observability-consumer-dr066.md'` (full relative path). The engine: (1) opens the file exactly (no fuzzy search — if it doesn't exist it lists available ones), (2) verifies `status: ready` (the queue enum is `draft | ready | building | done`, DR-069 — a `draft` is skipped; an owner decision is NOT a queue status, it routes through `.pandacorp/inbox/decisions.md`), (3) creates/updates FRDs + WOs via the iterate/bug engine inline, (4) sets `ONLY` = the affected FRDs, then (5) runs the normal plan + dep-check + build loop for those FRDs. Wins over `frds` if both are passed. The launcher-printed `Workflow()` call carries this field plus the fenced lease receipt; never hand-construct it.
- `maxFrds` (**opt-in** cap on features **processed** per run — built + blocked + **reopened**, so reopens count — for **supervised TEST runs**).
- **`maxAgents`** (a hard cap on subagents spawned this run — **the real guardrail for overnight runs**: counted inside the engine, it brakes reliably even if `budget.spent()` under-counts subagent work or the supervisor dies).
- `maxSpend` (an output-token ceiling via `budget.spent()` — a **secondary** ceiling, since it under-counts the heavy subagent work).

**Targeted build (partial implementation) — when the owner asks to implement only one FRD or a subset:** parse the FRD name(s) from `$ARGUMENTS` (accept both the short slug like `frd-05-settings` and a plain name like `settings` — resolve to the full folder name by scanning `docs/frds/`), pass them as `args.frds`, and follow the usual launch + supervisor flow. If the dep gate fires (unsatisfied deps), **stop immediately** before launching the workflow and tell the owner in Spanish which FRDs must be implemented first — do not launch. A targeted build still runs the full FRD gate and hardening for the requested FRDs; it does NOT skip review or tests.

**Targeted change build — when the owner asks to implement a specific pending change from the queue:** parse the change name from `$ARGUMENTS` (the filename without `.md` or a recognizable substring) and pass it as `args.change`. The workflow processes the change FIRST (creating/updating FRDs+WOs), then builds the resulting FRDs. If the change is `draft` (the only pre-`ready` queue status, DR-069), the engine stops immediately and tells the owner to flip it to `ready` first. If the resulting FRDs have unsatisfied deps, the dep gate fires as usual.

> **Engine = Dynamic Workflows, not Agent Teams** (DR-013). The per-FRD loop lives in the **script's code**, not in messages between peer agents. **Resumable by construction**: state lives in the work-order frontmatter + commits, so a re-launch reads `implementation_status` and **never rebuilds a `VERIFIED` work order** (DR-050) — no re-work.

## Execution modes (consumption/quality control)

Control the **concurrency and models** of the workflow (DR-014), not "team size":
- **pro**: minimum consumption. Up to 2 work orders at a time, economical worker (sonnet). A single full-stack `implementer` (no split). The `reviewer` still gates each FRD **on opus** — the judge is ALWAYS a different model from the worker (DR-015): pro economizes on throughput, never on the trust boundary (the diversity is one weighted judge spawn per FRD gate, not per WO).
- **balanced** (Max 5x): up to 4 work orders in parallel within a feature; judge (review) on opus, workers on sonnet.
- **powerful** (**default**, Max 20x): up to 8 in parallel — the default; finishes sooner.
- **deep**: best model everywhere, split team, extra adversarial review. Slowest/most expensive (it intentionally narrows the wave — opus + split is heavier per slice, so fewer run at once).

**How a run stops (DR-050, owner decision 2026-06-16).** By default the build runs to **completion** — it does NOT stop after N features ("stop after N" protects neither tokens nor progress: one feature can cost 10x another). A run stops only when: (a) **nothing is left** to build, (b) the **budget ceiling** is reached — `maxAgents` (a hard cap on subagents spawned, **the real overnight guardrail**) or a `+Nk` turn directive / `maxSpend`; **for overnight runs always pass `maxAgents`** (counted inside the engine, it brakes even if `budget.spent()` under-counts or the supervisor dies), (c) **too many features block in a row** (a health breaker), or (d) what remains **needs you**. `maxFrds` bounds a supervised **test** run and counts features **processed** — built + blocked + **reopened** — so a reopen counts and chained reopens can't slip past the cap (a real bug the 2026-06-16 overnight test caught). `maxFrds` is never the overnight guardrail; `maxAgents` is.

## Unattended operation — the build supervisor (run and walk away SAFELY)

### Launch checklist (run in order — mechanized, not prose)
1. **Preflight (read-only):** `bash "${CLAUDE_PLUGIN_ROOT}/scripts/preflight-implement.sh" <projectDir> --target-runtime claude --run-mode auto`. Every line must read `PASS` (exit 0), including the logical build-run intent classification from the shared resolver; it never asks the owner for an ID. A `FAIL` STOPS the launch: *another build ACTIVE* → abort; *STALE LOCK* → it prints the reset commands (it never auto-resets — the next step's `launch-implement.sh` takes a fresh lock); a readiness/overlay/`NEEDS CLARIFICATION` fail routes back to `/pandacorp:architecture` or `/pandacorp:upgrade`.
2. **Resolve the logical run + take the lock:** `bash "${CLAUDE_PLUGIN_ROOT}/scripts/launch-implement.sh" <projectDir> <mode> <maxAgents>` defaults to `auto`. The shared resolver automatically reuses the canonical `build_run_id` only for a safe cross-runtime cold continuation: `phase: implementation`, `running: false`, no lease, a valid prior ID, and `build_runtime` different from the runtime being launched. The owner never copies or chooses that ID. Otherwise the launcher creates a new governed run. A same-runtime next pass and a project already in `phase: release` are new runs; when the owner's request explicitly says to start a separate/reset budget run, pass `new` as the fourth argument. An explicit prior ID is reserved for a runtime supervisor's mechanical restart receipt, not owner interaction. The launcher then writes `phase: implementation`, `running: true`, `run_started_at`, touches `.pandacorp/run/build.lock`, and PRINTS the exact `Workflow()` call to run next.
3. **Launch the workflow:** run the exact printed `Workflow({ scriptPath: '<projectDir>/.claude/engines/pandacorp-build.js', args: { mode, maxAgents, projectDir, project, leaseToken, leaseEpoch } })` — `args` a real **JSON object**, never a string. **CAPTURE the run id it returns** (the `wf_…` id) and keep it for the whole run — the external maxAgents brake needs it to locate the run's transcript dir (§external brake below).
4. **ARG-ECHO VERIFICATION — its own step, never skipped:** the engine's FIRST log line must read `· maxAgents <N> ·`. If it says `maxAgents OFF` (when you passed one) or `args arrived as a <type>, NOT an object`, the args were DROPPED and the run is UNBOUNDED → `TaskStop` it immediately and relaunch (re-pass args; hardcode the scope into `args` if needed).
5. **Mount the supervisor:** Monitor + the **dedicated ~2-min `ScheduleWakeup` heartbeat** + `PushNotification` — per the detailed steps 1–2 below.
   The dedicated heartbeat MUST renew the atomic lease on every tick using the exact `leaseToken` and `leaseEpoch` printed inside the launcher's Workflow args: `node "${CLAUDE_PLUGIN_ROOT}/scripts/pandacorp-build-state.mjs" renew --project <projectDir> --token <leaseToken> --epoch <leaseEpoch>`. A failed renewal means ownership was lost: `TaskStop` the workflow and do not mutate anything else. Never copy either credential to events, commits or owner-facing summaries.

### Operative constants (the numbers the engine + supervisor run on)
| Constant | Value |
|---|---|
| Heartbeat / lock TTL (stale) | **10 min** (`supervisor_heartbeat` AND `last_event_at` both older ⇒ stale) |
| Supervisor liveness tick | **~2 min** (dedicated `ScheduleWakeup`, writes heartbeat + touches lock) |
| Frozen threshold (no activity) | **~15 min** |
| Health breaker | **3 consecutive non-`external` BLOCKED** features ⇒ stop the run |
| `MAX_REOPENS` per WO | **3** ⇒ then `BLOCKED: needs-owner` |
| Foundation repair cap | **2** internal cycles |
| `maxFrds` default | **unlimited** (opt-in cap, supervised TEST runs only) |
| `maxAgents` unit | **cost-weighted** (opus ≈ 3 units), bounded overshoot ≈ one expensive WO (**~11 units**) |

The owner runs `implement` and leaves, even overnight. For that to be SAFE — **no silent stalls, no burning the week's tokens** — launching the build is **never** just firing the workflow. It is **two things, always**:

1. **Launch + set the ceiling** — the **Launch checklist above** already mechanizes this; don't hand-build any of it. `preflight-implement.sh` asserts the conditions, then `launch-implement.sh` writes `phase: implementation` (the ONE place the architecture→implementation transition is set — DR-032; idempotent on a resume), `running: true`, `run_started_at`, clears `supervisor_heartbeat`, drops the phase-aware build lock (`.pandacorp/run/build.lock`, DR-063 — its freshness tells the Stop gate a build is active), pins `projectDir`/`project` (BL-0022 — the engine cds every subagent there and stamps events/`track.jsonl` with it, so launching from ANY cwd is safe), and **prints the exact `Workflow()` call**. **Run that printed call verbatim — never hand-construct it** (no `$(pwd -P)` juggling): the engine's shim `JSON.parse`s the `args` string a `scriptPath` launch delivers, failing LOUD on an unparseable one. Then complete the **ARG-ECHO verification (Launch checklist step 4, a hard gate): the engine's first log line must echo `· maxAgents <N> ·`; if it reads `maxAgents OFF` or `args arrived as a <type>, NOT an object`, the args were DROPPED and the run is UNBOUNDED → `TaskStop` immediately and relaunch.** For an **overnight run always pass `maxAgents`** — the real guardrail (counted in-engine, survives a dead supervisor, independent of `budget.spent()`); `maxSpend` is a secondary ceiling, `maxFrds` only for a short supervised test. Each FRD verified is an immutable commit (the safe point is a commit, never an agent state).
2. **Mount the supervisor** (this is the part that was missing and let a build run >1h unwatched):
   - **Live watch (`Monitor`)** — a bash poll (~2 min; **costs no tokens** — only an emitted *event* wakes the agent) over `.pandacorp/status.yaml` (FRD progress), the run's journal mtime + active agents (frozen?), and the agent count (spend proxy), **and the working tree's git health** (`git status --porcelain` — any `UU`/unmerged path or a `<<<<<<<` conflict marker means a BROKEN tree, a first-class failure signal, DR-067). It emits an event on: a FRD verified, **frozen** (no activity ~15 min), a new block, **a broken working tree** (unmerged paths / conflict markers, DR-067), **pace below target**, or run end. Cover the failure signatures, not just success ("silence is not success"). **Write `supervisor_heartbeat: <ISO timestamp>` to `status.yaml` AND refresh the build lock (`touch .pandacorp/run/build.lock`, DR-063) on every event tick — AND on a DEDICATED ~2-min `ScheduleWakeup` cadence, never ONLY when a Monitor event fires** (a healthy-but-quiet build emits no events for long stretches, so an event-only heartbeat freezes and the build looks dead — the exact audit failure): the heartbeat/lock-touch is a timer, not a side effect of activity. Both are liveness signals: as long as the supervisor is alive the heartbeat stays fresh (< 10 min) and the Stop gate keeps standing aside; if the supervisor dies, the heartbeat AND the lock go stale together and the next `implement` auto-clears the lock while the Stop gate auto-re-engages (same 10-min TTL). **Every staleness reader crosses `supervisor_heartbeat` WITH `last_event_at` (the producer's own clock, DR-066): the build is LIVE if EITHER is < 10 min old, and STALE only when BOTH are older** — so a quiet supervisor whose engine is still advancing `last_event_at` at safe points is never misread as dead (this is what `preflight-implement.sh` and `verify-before-stop.sh` both enforce).
   - **Positive heartbeat to the event stream (observability fidelity, DR-066).** A frozen `running: true` is **not** proof of life — a monitor reads liveness as `running AND fresh`, so the supervisor must keep the freshness signal genuinely alive, not just flip a flag. On **every tick** (and on the time-driven beat below), besides writing `supervisor_heartbeat`, **append a positive heartbeat event to the stream** so "sin señal" (no events at all) truly means *hung*: `printf '{"event":"SupervisorTick","at":"%s","project":"%s","data":{"running":true}}\n' "$(date -u +%FT%TZ)" "$(basename "$PWD")" >> ~/.claude/dashboard-events.ndjson` (fire-and-forget). The **engine** already advances `last_event_at` in `status.yaml` at each safe point (FRD gate, sync) and appends `AgentWorking` events per agent — together they are the producer's positive heartbeat. The consumer (Mission Control) crosses `running` with this recency and declares its own freshness (en vivo / datos de hace X / sin señal). See `factory/standards/observability.md` "Liveness & freshness fidelity".
   - **Health heartbeat — a VISIBLE message to the owner every ~20–30 min, even with NO anomaly.** The Monitor only wakes on events, so the supervisor must schedule this latido SEPARATELY (a `ScheduleWakeup` timer) — never rely on events for it, or a healthy-but-quiet build looks dead. **The same time-driven tick also advances `supervisor_heartbeat` + appends a `SupervisorTick` event (DR-066)** so the freshness signal never freezes between long, quiet agents (the exact audit failure: the build advanced while the heartbeat stayed frozen). Each heartbeat states: WOs done/total, the FRD in progress, rough spend so far, and "all green". **Silence reads as "stuck" to the owner** — a periodic "still going, X/Y done, ~Nk output spent, all green" is the entire point.
   - **Notifications** — the engine fires a macOS desktop notification (`osascript`) on every alarm; the supervising agent ALSO sends a `PushNotification` to the phone (when Remote Control is on). No third-party push app (owner decision 2026-06-16). The owner is asleep, so reach BOTH the desktop and (if available) the phone.
   - **Announce each FRD (milestone)** — on every FRD **verified**, send the owner a visible message (+ push): the FRD name, one line on WHAT it does, the **route/page to try it**, and X/Y done. Don't make the owner ask "is it done yet?" — tell them at each milestone, and point at the review worktree (below) so they can actually open it.
   - **React to every event, repairing before giving up** — a stall → unstick it; a **broken working tree** (unmerged paths / `<<<<<<<` markers) → **restore it to `last_green_sha`** (a clean reset to the green commit — bounded auto-repair; NEVER hand-resolve a stash-pop, DR-067) and let the run rebuild the lost between-safe-point work; a failing feature → the engine itself runs a **repair pass** (diagnose + fix) before it ever blocks. Only when a fix is genuinely out of reach does the feature go **BLOCKED with a reason** and the build moves on to independent features: `needs-owner` (a human must act — env var, secret, external account, product decision; also logged to `.pandacorp/inbox/decisions.md`), `external` (internet/upstream — retried later), or `error` (unresolved technical fault). The supervisor escalates `needs-owner` to you immediately; for a budget/health stop it **stops the build and notifies** with the detail ("blocked at FRD-X because Y, come decide").
   - **Self-learning of avoidable stalls (DR-047)** — when the supervisor unsticks something, **classify it**: uncontrollable (internet, server 500) → nothing to learn, don't capture; **avoidable** (a recurring pattern, an engine bug, a config) → capture the lesson to `factory/memory/_inbox.md` and, if fixable factory-wide, **fix it on the spot** so it never recurs. Every avoidable stall makes the factory better.
   - **Budget ceiling** — the engine stops at `maxAgents` (counting the subagents it spawns, checked at every FRD **and build-wave** boundary — DR-070, the fine-grained brake) and `maxSpend` is a secondary `budget.spent()` ceiling. **The in-engine counter is NOT trusted alone (DR-070)** — it once silently under-counted and blew past the cap. So the supervisor enforces a **reliable EXTERNAL brake**: each tick it counts the actual `agent-*.jsonl` transcript files the Workflow run wrote. **The REAL path (verified 2026-07-07 against live runs):** `~/.claude/projects/<project-slug>/<session-id>/subagents/workflows/<run-id>/agent-*.jsonl` — where `<project-slug>` is the LAUNCHING session's cwd with every `/` replaced by `-` (leading dash included: `/Users/Shared/Proyectos/personal-page-v2` → `-Users-Shared-Proyectos-personal-page-v2`), `<session-id>` is this Claude Code session's id, and `<run-id>` is the `wf_…` id the `Workflow()` launch returns — **capture that run id at launch (Launch checklist step 3) and keep it for the whole run.** If you didn't capture it, resolve it at tick time as the NEWEST `subagents/workflows/wf_*` dir under the session transcript root. Count with `ls ~/.claude/projects/<project-slug>/<session-id>/subagents/workflows/<run-id>/agent-*.jsonl 2>/dev/null | grep -v '\.meta\.' | wc -l` and, past `maxAgents` (allow a small overshoot for the in-flight wave), it `TaskStop`s the workflow + runs the guaranteed shutdown — independent of the engine counter and of `budget.spent()`. **DIVERGENCE to know (DR-070): inside the engine `maxAgents` is COST-WEIGHTED (an opus spawn counts ≈ 3 units, so an escalation can't silently blow the budget), but this EXTERNAL brake counts RAW transcript files (1 per spawn, unweighted).** So the external file count is a conservative *over*-estimate-proof floor that trips a bit later than the weighted engine cap — that's intended: the external brake is the fail-safe backstop for a dead/under-counting engine, not a second precise meter. Size `maxAgents` to the weighted cost; the external brake will still catch a runaway. The supervisor also treats `maxAgents` as the **cumulative-across-passes** walk-away budget: it does **not** relaunch the next pass once cumulative agents reach the ceiling — it stops and notifies the owner with the balance. Always pass `maxAgents` for an unattended run — and remember one pass over many FRDs can legitimately cost ~3–5 agents/FRD, so size the cap to the FRDs you expect to process, not to a single number that one pass blows through.
   - **Relaunch for multiple passes** — a run makes ONE pass over the pending FRDs; WOs a gate **reopens are deferred to the NEXT run**. Every terminal path releases its lease, so a same-runtime next pass goes through the full preflight + `launch-implement.sh` again and receives a NEW logical run/receipt. A cross-runtime launch at that released implementation safe point is instead auto-classified as a cold continuation and inherits the prior logical run ledger. Never relaunch the Workflow directly and never reuse a released token/epoch. `VERIFIED` frontmatter prevents rework across passes.
   - **Stop the LOOP vs stop a run (DR-068).** Stopping a single workflow **run** (TaskStop / the run ends) does NOT stop the build — the supervisor's job is to **relaunch the next pass**, so a lone stop just triggers another pass (the exact confusion the owner hit: "I stopped it but it came back"). To stop the **whole build**, the owner sets the stop signal — **`rethink_pending: true` in `status.yaml`** (or drops `.pandacorp/run/stop`); the supervisor checks it **at each safe point AND before every relaunch**, and when set it **halts the loop** (does not relaunch) and runs the guaranteed shutdown below. Surface this to the owner: "to stop the build, set `rethink_pending` / run the stop signal — not just stop a run."
   - **Guaranteed shutdown — `running: false` on EVERY exit, never a lying flag (DR-068).** Whenever the supervisor's loop ends FOR ANY REASON (nothing pending/reopened, budget/health/needs-owner stop, owner stop signal, or the supervisor itself wrapping up/being interrupted), its **guaranteed LAST act** is to clear state: write `status.yaml: running: false` + **remove the build lock (`rm -f .pandacorp/run/build.lock`)**. This is NOT optional or "only on a clean end" — a stopped build that still says `running: true` is a **lying flag** (DR-068). Belt-and-suspenders: the engine spawns a tiny **close-out agent** on normal completion that sets `running: false`; and **a stale `running: true` is NEVER trusted as alive** — any reader (the concurrent-run guard, the supervisor's own health tick, Mission Control's display) crosses `running` with `supervisor_heartbeat` recency (DR-066) and reads a stale flag as STOPPED, and the next state-writer auto-corrects it to `false`. (If the supervisor is hard-killed without cleaning up, the lock + heartbeat go stale after 10 min and the next `implement` auto-clears them — the explicit shutdown is the tidy path, the TTL + recency-reconciliation are the safety nets.)
- **Auto mode (permissions), not babysitting**: the owner activates Claude Code's *auto mode* (Shift+Tab, or `defaultMode: "auto"` in THEIR `~/.claude/settings.json`). NEVER `--dangerously-skip-permissions`.
- **Review worktree — let the owner SEE the latest green version without touching the build.** Keep a worktree pinned at the last green under the **canonical review-worktrees root** (DR-090 — `/Users/Shared/review-worktrees/<project>`, a `Shared`-level sibling of `Proyectos/`, named after the project, NOT inside `Proyectos/` and NOT inside `.claude/`): `git worktree add /Users/Shared/review-worktrees/<project> <last_green_sha>` with its OWN `node_modules` (`pnpm install` there — NOT a symlink, which breaks Next's HMR/watcher), and update it (`git -C /Users/Shared/review-worktrees/<project> checkout <new last_green>`) each time an FRD verifies. The build keeps running untouched on the main checkout. To view it, the owner opens a **Claude Desktop session in that worktree** (its Preview button serves it natively — the build session's Preview can't point at an external worktree, confirmed) OR asks the supervisor to bring up its dev server on a fixed port **on-demand** and hands back the URL. Do NOT keep a permanent dev server that restarts on every FRD (Next's file watcher is flaky across `git checkout`). Subagents that must mutate in parallel can use `isolation: 'worktree'`.

## How it runs

`pandacorp-build` is a native Claude Code JS engine (plain script code in `.claude/engines/`, launched only by `scriptPath` from this skill — never an owner command) that orchestrates each run deterministically: **baseline self-heal** (restore a dirty/conflicted tree to `last_green_sha` + drop stale stashes/temp pages, DR-067; then repair a red `verify.sh` before planning) → **plan** (read every FRD's frontmatter + Build Plan; return non-`VERIFIED` FRDs in cross-FRD order) → **global-wave build** (§Execution modes above) with **one review + integration gate per FRD** and the patch-first → self-repair → revert+reopen recovery ladder. It is app-agnostic — it depends only on the docs' state and the project's `verify.sh`. The step-by-step is §Per-FRD loop below; the depth is `factory/standards/build-orchestration.md`. Two facts the runtime needs that live only here:

**`reviewSplit` — the FRD gate, in four parallel lenses.** In `powerful`/`deep` mode the per-FRD gate fans out into four read-only finder lenses (correctness vs the EARS AC, security, quality/duplication, runtime-visual) running in parallel; their findings are deduped, each surviving CORRECTION is adversarially verified by a skeptic that tries to refute it (a refuted finding dies), and a single Opus closer reviews only what survived — independently confirming each correction, running `verify.sh --since`, and applying the same DR-072 correction-vs-nit split verdict as the serial gate. `pro`/`balanced` keep the plain serial reviewer, unchanged. If the split's estimated cost doesn't fit the remaining `maxAgents` budget, or all four finders die, the engine falls back to the serial gate — the gate itself is never skipped.

**Resume is frontmatter-driven, not run-id-driven.** A relaunch reads `implementation_status` off disk: a `VERIFIED` work order is never rebuilt, and one left `IN_REVIEW` goes straight to its FRD gate instead of being rebuilt from scratch. There is no in-memory "where was I" — the tree IS the memory. A run makes ONE pass over the pending FRDs; the supervisor (Monitor + ScheduleWakeup + PushNotification, §Unattended operation above) relaunches pass after pass — reading the same frontmatter each time — until nothing pending/reopened remains or a budget/health ceiling stops it.

## Per-FRD loop (what each feature goes through)

1. **Build** each work order (coarse — a whole view/capability): the ENGINE stamps `IN_PROGRESS` at wave dispatch (BL-0002 — atomic, so the board never under-reports a busy wave); the builder receives its **context pack (DR-108)** — the WO file path + the verbatim EARS AC lines it owns, extracted once by the planner — implements with TDD, runs its **own** fast self-test (not the whole suite; merged into the builder in solo mode — the FRD gate is the trust boundary, not the self-test), sets `IN_REVIEW`, writes its `## Status Note` hand-off (interfaces exposed, integration seams) so the next agent reads that, not all the code. Mechanical steps (the per-WO commit, the dispatch stamp, safe-point, rollup sync, archive, notify) run on the cheap tier (`MECH`, default haiku — DR-108).
2. **FRD gate** (`reviewer`, a different model from the generator): review the WHOLE feature with the 4 lenses + write adversarial tests the implementers didn't see (DR-015), exercising the work orders **together** (integration); run the **focused** `verify.sh --since <last_green>` (biome+tsc global + only the tests affected this cycle; the browser layer scoped to smoke+shell — DR-106; the full suite runs once at close-out). Green → every WO + the FRD frontmatter → `VERIFIED`, commit. A specific WO wrong → the engine recovers in this order:
   - **Patch-first (DR-073):** attempt an in-place opus patch of the reviewer's specific finding(s) + RED-proven failing test on the EXISTING build, re-gated WHOLE-PROJECT (full FRD tests + a whole-project knip+biome+tsc); an independent verifier stamps `VERIFIED` on clean.
   - **Self-repair budget (DR-107):** a red the patch's own edits introduced — e.g. a type error in its own new test — is fixed in place, up to 2 internal cycles, instead of ending the patch.
   - **Cause-discrimination (BL-0001)** when a patch fails: `cause: 'gate-test-defective'` (the reviewer's adversarial test is internally inconsistent/unsatisfiable) routes to a gate-test repair — an independent reviewer fixes the TEST, never discarding a correct build; `cause: 'code'` falls back to revert + reopen (`PLANNED`, `reopen_count`+1; reviewer-authored / Status-Note-referenced test files are PRESERVED to `.pandacorp/run/preserved-tests/`, never deleted).
   - **Retry budget:** a `cause: 'code'` reopen then gets ONE **in-run retry (DR-107)** — rebuild the reopened WOs NOW from the clean base (opus) + re-gate, instead of paying a whole extra pass; only if that also fails is the WO deferred to the next run. `reopen_count` stays the single attempt budget, capped by `MAX_REOPENS` → BLOCK needs-owner.
   - **Model-escalation (DR-073):** a WO marked `difficulty: high` builds on opus a-priori, and any WO with `reopen_count >= 1` escalates to opus empirically (it already failed once) — never below the mode's floor; `maxAgents` counts each opus spawn as ~3 cost-units so escalation can't silently blow the budget.

   Otherwise (no specific WO to patch) the engine runs a **repair pass** (diagnose + fix); only if that fails does the feature go `BLOCKED` with a `blocked_reason` (`needs-owner` | `external` | `error`) + notify, and the build continues with independent features.
3. **Drain the change queue at the safe point (DR-069) — how the owner talks to a running build. THE ENGINE DOES THIS ITSELF** (audit-20 P0-3: it is cabled in `pandacorp-build.js` as the **safe-point check at every FRD boundary** — it reads `rethink_pending` (stop), the queue's `ready` items (drain via the process-change step) and answered `decisions.md` entries (flip their BLOCKED WOs back to PLANNED); the supervisor no longer carries the drain, it only monitors/notifies). The change was written by a detection-free door, so it's been waiting safely — the build integrates it **at a safe point, never mid-feature**. **Only `status: ready` items are taken; `status: draft` is SKIPPED** (the owner is still working it out — the file is left untouched, DR-069). **TARGETED-BUILD SCOPE (owner incident 2026-07-06): the queue drain runs ONLY on a BARE `/implement` (no `change`, no `frds`). A TARGETED build — launched with a specific `change` or `frds` — implements ONLY its target and does NOT drain other `ready` changes** (they wait for a later bare `/implement`); the engine's `TARGETED` flag suppresses the drain at every safe point (hard JS guard, proven by test-pandacorp-build scenarios 10a/10b, with 10c guarding the bare path). "Implementa solo este cambio" means only that. Routing (encoded in the engine's process-change prompt), work-conserving — never stall the whole build for one item:
   - **`expedite`** first, then **`standard`** FIFO.
   - **autonomous-resolvable** (a clear scoped change) → integrate it via the **`iterate`** engine (feature/change → FRD/work-orders/blueprint, calling `work-orders`/`design`) or the **`bug`** engine (regression test → fix), then it passes the **FRD review/test gate** like planned work (same quality).
   - **needs-owner** (product/money/irreversible/design decision) → write it to `.pandacorp/inbox/decisions.md`, notify, and **DEFER + continue with independent FRDs**; only fully STOP if it blocks everything.
   - **structural / fundamental / new-design** → pause (`rethink_pending`) + tell the owner the command to run (e.g. `/design` for a visual — the design gate is theirs); don't silently rebuild half the app.
   - **Complete = verify the durable record, then ARCHIVE (never hand-delete, DR-069).** The **engine's close-out archive step** does this for every change it integrated whose affected FRDs all VERIFIED: it confirms the durable record exists (the landing commits **touched the canonical docs**), stamps `status: done` + `shipped_sha` + `shipped_at` and **moves the file into `changes/done/`** (the active `changes/` stays pending-only; the file is **moved, never `rm`'d** — the folder is gitignored, so a delete is irreversible), updates the queue index. A change whose FRDs did NOT all verify stays `ready` and is drained again next run. **Never auto-delete** `done/`; any prune is a separate owner-gated, reversible action.
   The engine also honors `rethink_pending: true` at every safe point (stop cleanly + notify; the next run's baseline clears the flag as it starts from the re-planned docs) and applies answered `.pandacorp/inbox/decisions.md` (BLOCKED needs-owner WOs flip back to PLANNED). (The former `inbox/bugs/` folds into `changes/` as `type: bug`.)

## Real-time documentation (Mission Control reads it live)

Keep updated: each work order's frontmatter `implementation_status`; `.pandacorp/status.yaml` (`progress`, `running`, per-status counts, `last_green_sha`, `safe_to_test`); `.pandacorp/comms/progress.md` (append-only log — what was done, decisions, problems; allows resuming with no prior context). If something deviates from plan, document it; don't hide it.

**`progress.md` is the owner's milestone feed, not a gate-tool transcript (BL-0014).** Mission Control's Summary tab renders this file verbatim (`readActivityLog`) — every entry appended by the engine, the supervisor or an agent MUST be a milestone: an FRD/WO built or changed, a decision taken, an item blocked needing the owner, a deviation from plan. **Never append a raw gate-tool exit line** (`**Biome:** limpio. Exit 0.`, `**tsc:** limpio. Exit 0.`, `**knip:** limpio. Exit 0.`, `**madge:** sin dependencias circulares. Exit 0.`, `**Vitest:** 327 ficheros… Exit 0.`, `**Smoke (Playwright):** 14/14 pasados. Exit 0.`, `**Visual (Playwright):** 14/14 baselines coincidentes. Exit 0.` or anything shaped like a `verify.sh` recap) — `verify.sh`'s per-tool results are evidence the reviewer/engine consume internally to decide PASS/FAIL, not owner narrative. When an FRD gate passes, write what it means ("FRD-05 Settings verified — WOs 01–04 built the preferences panel + persistence"), not the tool checklist that proved it. Full contract: `factory/standards/build-orchestration.md` §7.

## Decision points (escalation, visible in Mission Control)

When something unplanned appears: **first investigate** (delegate to `researcher`); if you can take the coherent decision yourself, do so and document it. **Only escalate genuinely human decisions** (product scope, irreversible, spending money, or what the registry marks human): note it in `.pandacorp/inbox/decisions.md` as `pending` (what's happening, options, **your recommendation**), notify the owner, and continue with independent FRDs. The owner answers with `/pandacorp:decide`.

**Closing out a decision — edit the ORIGINAL block in place, never just append (DR-069; owner incident, 2026-06-30).** When the build resolves a logged decision **autonomously mid-build** (a repair pass makes it moot, a later fix supersedes it, the blocking work order finishes another way — the owner-answered path runs through `/pandacorp:decide`), you MUST close it the same way the owner path does: edit that decision's OWN heading + `- **Estado:**` line to `RESUELTO: <what/why> (YYYY-MM-DD)` or `OBSOLETO: <what changed> (YYYY-MM-DD)`; a separate narrating entry is welcome as EXTRA context but never a substitute (Mission Control and `/pandacorp:decide` parse each block's OWN heading, so an untouched heading reads "pending" forever). The full shared protocol — RESUELTO vs OBSOLETO, closing a decision a newer sibling already resolved, the never-decide-obsolete-for-them gate — lives in `plugin/skills/decide/SKILL.md` step 4.

## Final hardening step — the audit moves INTO construction (DR-085, CABLED in the engine — BL-0012)

Construction's **last step**, once every FRD is `VERIFIED`, is the hardening pass that used to live in `/pandacorp:release`. **This is no longer prose — the ENGINE runs it** (`pandacorp-build.js`, the `Hardening` phase before close-out) and **`phase: release` is fail-closed on its evidence**: the close-out asserts a `docs/reviews/security-*.md` dated this run AND the `## Verification` section in `docs/analytics/events.md` exist before it may set `phase: release`; if either hardening stage fails, the engine keeps `phase: implementation`, files a needs-owner decision and notifies — it NEVER self-declares release on the FRD loop alone (the fail-safe close can't either):

1. **Security audit + fix** (two spawns, audit-then-fix — RFC-30 N4): essential web OWASP + **OWASP Top 10 for Agentic Applications** (ASI01–ASI10, DR-017) if the product has agents/LLMs with tools. First the **read-only `security-auditor`** agent audits and writes the evidence report `docs/reviews/security-<date>.md` (each finding + severity + remediation) — it never edits code (that independence is the point). Then a **Write/Edit-capable `implementer`** agent applies every Critical/high fix (TDD) and re-runs `verify.sh --since` until green, **before construction is declared complete** — never deferred to release. Evidence: `docs/reviews/security-<date>.md` (the same report, annotated fixed | accepted).
2. **Telemetry / metrics verification** (`analytics` agent, spawned by the engine): verify the events in `docs/analytics/events.md` actually fire in the critical flow (evidence, not "they should"). Evidence: the `## Verification <date>` section in `events.md`. The product is instrumented as part of the build (DR-085).
3. **Quality close-out**: the full suite + e2e green, lint + typecheck clean, the shell-presence gate green (the per-FRD gates already enforced correctness + shell), and — running here for the FIRST time (DR-106) — the whole-project **visual + responsive** gates (the per-FRD gate scopes its browser layer to smoke + shell, so the full visual/responsive suite runs once, here at close-out) + the cross-feature integration review. **README drift check (DR-112/DOC-3, advisory):** work orders can add env vars or change scripts mid-build — confirm the root README's "Getting started" commands and env-var pointer still match the current `package.json` scripts and `.env.example`; fix inline if they drifted. Never blocks the gate on its own.

Only when this hardening is green — evidence on disk, asserted by the close-out — is construction complete.

## When all are done
Full suite + e2e green **and the hardening evidence present (audited, instrumented, quality-gated — DR-085/BL-0012)**, then the engine's close-out sets `.pandacorp/status.yaml → phase: release`. (There is no `operation` phase — reaching `release` means the build is ready to be launched by `/pandacorp:release`, internal or external.) **Harvest the lessons now (DR-047) — a REQUIRED close-out step, not a suggestion; close-out is when they're freshest.** Three sub-steps, in order:
1. **Harvest**: run a `librarian` harvest of this project (drain `.pandacorp/run/lessons.md` + the build's capture points — reviewer findings seen ≥2×, fixed bugs, library verdicts chosen in the blueprint — into `factory/memory/` as evidence-anchored candidates, A.U.D.N.-deduped via `/pandacorp:memory harvest <project>`), so the NEXT build inherits what this one learned instead of relearning it.
2. **Measure retrieval**: run `bash "${CLAUDE_PLUGIN_ROOT}/scripts/count-lesson-citations.sh" <project-dir> <factory-path>/factory/memory` — it greps the build's artifacts (blueprints, reviews, WOs, progress log) for `LESSON-NNNN` citations and deterministically updates each cited lesson's `applied_in`/`times_applied` (and auto-queues `promotion: proposed` at ≥3 distinct citing projects). Scripts measure retrieval, not agent self-report.
3. **Stamp it**: write `last_harvest: <ISO timestamp of now>` into `.pandacorp/status.yaml`. **A close-out without the stamp is an INCOMPLETE close-out** — do not report the build finished without it. The scheduled memory sweep treats a `phase: release` project with a missing/stale `last_harvest` as un-harvested and drains it as a backstop, so a skipped step is caught, not lost.

The periodic review sweep reconciles the harvested candidates and proposes promotions to the owner. This closes the self-learning loop at its highest-signal moment (the always-on capture is the supervisor's job during the run; this is the refine step). Summary to the owner + next step `/pandacorp:release` (the deploy/launch — internal or external).

## Rules
- Never advance with red tests. A WO that can't pass: the engine first attempts a **repair**; only if that fails = **freeze-on-red** (don't commit broken, leave HEAD at `last_green_sha`, mark `BLOCKED` + `blocked_reason` in the frontmatter, notify, continue with independent features). The build runs to completion; it stops only by budget/health or when something `needs-owner`.
- **State is the frontmatter** (`implementation_status`), never body prose or git (DR-050). `VERIFIED` is never rebuilt.
- **Never `git stash` to preserve in-flight work across a kill / relaunch / stop (DR-067).** The build is resumable from the frontmatter + commits, so killing or stopping a run discards only the uncommitted work between two safe points — the relaunch rebuilds it from `PLANNED`/`IN_PROGRESS` state, never re-doing `VERIFIED`. A `git stash` of in-flight work, popped later onto a tree that has moved on, **conflicts** (the MC Phase-2 incident). The only safe recovery from a dirty/conflicted tree is to **restore it to `last_green_sha`** (a clean reset to the green commit), never a stash-pop. **Drop any pre-existing build stashes on startup** and clean leftover temp `preview-wo*` pages — they're stale; the work is resumable.
- Quota limits: lower the mode (fewer parallel agents / cheaper models). Not a code error.
- If a work order reveals the blueprint/FRD was wrong, document the conflict, fix the source doc (ADR if architectural), then continue.
- **Don't trust the worker's honesty** (constitution §22): the `reviewer` re-verifies everything; the environment is fail-closed. Never relax verification because "the agent said it passed".
