# /pandacorp:memory

The self-learning loop's operator surface. Runs IN panda-corp. It accumulates durable engineering know-how in `factory/memory/` (read its README) so future builds are smarter and faster — the **harvest → store → retrieve → propose → gate → promote → prune** loop (DR-047). This skill is the loop's front-end (harvest, review); `/pandacorp:learn` is the back-end (promotion). It NEVER promotes a lesson to a standard/rule/skill by itself.

`$ARGUMENTS`: `status` (default, read-only) · `harvest [<project>]` · `review`. A mode-less `/pandacorp:memory` (no `$ARGUMENTS`) NEVER auto-harvests — it lands on **status** (step 11): non-destructive, informative, and it points at the next action (including the exact `harvest`/`review` command to run). If the owner's intent is ambiguous between modes, ask instead of guessing — never default to a write.

## Mode: harvest (Phase 1) — `/pandacorp:memory harvest [<project>]`

1. **Pick the target.** Explicit `<project>` argument wins if given. Otherwise: if the cwd is itself a Pandacorp project (`.pandacorp/status.yaml` present, via `scripts/is-pandacorp-project.sh` run against `.`), default the target to **the current project** (the natural in-project gesture — "harvest THIS project" — needs no argument). Otherwise (running from the factory root, no project underfoot), read `factory/portfolio.md` and offer recent / just-shipped projects to pick from. This preserves the factory-run sweep, an explicit `harvest <project>` from anywhere, and the `/loop` portfolio sweep (which always passes explicit targets) exactly as they are — only the no-argument, in-project case changes. Preflight on the resolved target: it must be a Pandacorp project (`.pandacorp/status.yaml` present, via `scripts/is-pandacorp-project.sh`); if its `overlay_version` < 6.0.0, suggest `/pandacorp:upgrade` first (legacy paths differ).
2. **Harvest** (delegate to the `librarian` agent): FIRST **drain the raw capture inboxes** (`.pandacorp/run/lessons.md` in the project, `factory/memory/_inbox.md` in the factory — the always-on notes), applying the **durability routing rule (DR-103)** as it drains — a note that is a **defect/change to the factory's own tooling** (an actionable to-fix item, e.g. a `gap ·` note) does NOT become a lesson; it is filed in **`factory/backlog/`** as a `BL-*` item (memory is durable retrievable knowledge, not an action queue). A note that is BOTH a durable lesson AND a required fix is **split** (defect → backlog, generalizable lesson → memory, linked). THEN read the capture points (`.pandacorp/inbox/changes/done/` — the unified queue's archive, DR-069 —, `.pandacorp/comms/progress.md`, `.pandacorp/build-journal.jsonl` — the durable progressive-learning build journal (attempts/verdicts/diagnoses/resolutions); its GOLD entries — WOs reopened ≥2× before resolving, and any `architectural`/`deadlocked-contract` diagnosis — are the richest rework-lesson source —, `docs/reviews/`, the per-feature `docs/frds/frd-NN-<slug>/blueprint.md` + `docs/product/architecture.md` + `docs/adr/`, `.pandacorp/inbox/decisions.md`, the launch verdict) and write evidence-anchored **candidate** lessons (with `provenance`) to `factory/memory/`, deduping with A.U.D.N. (ADD/UPDATE/MERGE/NO-OP) against what is already there.
3. **Measure retrieval (loop v2)** — run `bash "${CLAUDE_PLUGIN_ROOT}/scripts/count-lesson-citations.sh" <project-dir> factory/memory`: it greps the project's artifacts for `LESSON-NNNN` citations and deterministically updates `applied_in`/`times_applied` on the cited lessons (and auto-queues `promotion: proposed` at ≥3 citing projects). This is the ONLY way those counters move — never hand-edit them, never trust an agent's claim of having applied a lesson without the citation in a durable artifact.
4. **Eval-gate (DR-047, low-risk tier).** A candidate auto-promotes to `status: active` only if schema-valid (run `bash "${CLAUDE_PLUGIN_ROOT}/scripts/validate-memory.sh" factory/memory` — deterministic, fail-closed) AND **cross-corroborated** (confirmed by a DIFFERENT build/project than the one that produced it, OR `provenance: owner-stated`/`ci-verified`) AND it does not contradict a higher-confidence lesson. A single-trajectory `agent-inferred` lesson can NEVER self-activate (EDV/anti-poisoning: a polluted store measurably hurts builds — a small clean store beats a large noisy one). Everything else stays `candidate` and is surfaced to the owner (Mission Control Propuestas inbox / this report).
5. **Stamp the project** — write `last_harvest: <ISO now>` into the project's `.pandacorp/status.yaml` (the scheduled sweep uses it to catch projects that reached `phase: release` without a harvest).
6. **Report** to the owner (in Spanish): what was harvested, what auto-activated, what awaits their call, and any `library-verdict` worth remembering. Nothing was promoted to a standard/rule/skill — that is `/pandacorp:learn`.

## Mode: review / prune (Phase 4) — `/pandacorp:memory review`

7. **Audit** (delegate to `librarian`): scan `factory/memory/` and produce a proposal report — `deprecate` (never retrieved + old — **but see the prune freeze below**), `reconcile` (contradicted by newer evidence), `merge` (near-duplicates), `drop` (cold uncorroborated candidates), and **`promote`** (a lesson recurring across ≥2 projects is a candidate for a standard/DR/skill → set **`promotion: proposed`** on the lesson, with the target + a one-line rationale in its body, so it queues durably for your approval — you decide now or later, never under pressure; also refine the target/rationale of any lesson the citation script auto-queued).
8. **Prune freeze (loop v2):** the "never retrieved" deprecation criterion is FROZEN until `count-lesson-citations.sh` has measured ≥3 builds (i.e. ≥3 distinct projects appear across `applied_in` fields in the store) — before that, `times_applied: 0` means "retrieval was never measured", NOT "the lesson is useless", and pruning on it would condemn the whole store. Once measurement exists, prune uses `times_applied` + how long since the lesson was last cited.
9. **Reflection pass (capped):** when ≥3 low-level lessons cluster around the same domain/root cause, propose ONE higher-level `pattern` lesson that synthesizes them (linking the originals in `links:`) — the store's compression mechanism. Reflection synthesizes from evidence-anchored lessons only, NEVER from other reflections (LESSON-0001), and goes through the same A.U.D.N. dedup + candidate gate.
10. **Apply only the safe, reversible parts** yourself (set `status: deprecated`, merge clear duplicates) — these never delete a file (DR-011/DR-007). Promotions and anything high-risk → propose to the owner; on approval, route to `/pandacorp:learn` (the apply stage).

## Mode: status — `/pandacorp:memory status`

11. Summarize the store: counts by `type` and `status`; **the promotions queue** — every lesson with **`promotion: proposed`** (the rules awaiting your approval, with their target + rationale), so you can review the full list and decide whenever; the most-applied lessons; the oldest never-retrieved; and the pending `candidate`s awaiting corroboration. **Also report the factory backlog's health** (run `bash "${CLAUDE_PLUGIN_ROOT}/scripts/validate-backlog.sh"`): open/doing/done counts + the top 3 oldest OPEN `BL-*` items, one line each — the plane-3 queue drains through visibility (audit-20; DR-103 built the queue, this surfaces it).

## Rules
- **Memory is not an action queue (DR-103).** A defect/change to the factory's own tooling is an actionable,
  closeable item → `factory/backlog/` (a `BL-*` item), NOT a lesson. Only durable, retrievable knowledge
  lands in `factory/memory/`. A dual-nature item is split (defect → backlog, generalizable lesson → memory,
  linked). A product-project defect → that project's `.pandacorp/inbox/changes/` (via `/change`), never here.
- **Propose, don't promote.** Turning a lesson into a standard/DR/skill is HIGH-RISK (DR-047) → always `/pandacorp:learn` + the owner gate. This skill never edits `factory/standards/`, `registry.yaml` or `plugin/`.
- **Evidence or nothing** (LESSON-0001): no candidate without a concrete anchor; never harvest reflections.
- **Never delete** — deprecate (DR-047/DR-011/DR-007). Contradictions are reconciled, not erased.
- **Committed English know-how** (DR-033): the store is company know-how like standards, NOT owner personal data; the agent still talks to the owner in Spanish (DR-009).
- **Cadence (DR-047, loop v2 — three concentric triggers, none of them the owner's memory):**
  (1) **at build close-out** — `/pandacorp:implement`'s close-out RUNS harvest + the citation count and stamps `last_harvest:` in the project's `status.yaml` (the freshest moment, and verifiable: no stamp = incomplete close-out);
  (2) **the scheduled sweep** (the machine-local DAILY routine `pandacorp-memory-review` — canonical definition committed at `plugin/docs/routines.md`; the installed copy is a deployment of it) — drains the factory inbox + every portfolio project whose `.pandacorp/run/lessons.md` has notes, fires early when a threshold trips (**≥20 pending notes or ≥7 days since the last drain**), catches any `phase: release` project with a missing/stale `last_harvest` (the close-out backstop), then runs `review`;
  (3) **on demand** — surfaced by Mission Control's **memory-health reminder** (FRD-17: pending-notes count + last-run staleness + the command).
