# /pandacorp:design

Design phase. Runs IN the project (requires `docs/product/prd.md` and `docs/frds/` — if they are missing, run `/pandacorp:spec` first).

This phase works at **two layers (DR-049)**: it establishes the project's **global design system** (the PDD — `docs/design/` + root `DESIGN.md`, ONE per project, never per-feature), and when it designs a specific UI feature it writes that feature's design **inside the FRD module** (`docs/frds/frd-NN-<slug>/fdd.md` + `mocks/`). The product-level **direction** mockups below are the exploration that picks the system; they are a different thing from a feature's `mocks/`.

## Steps

0. **Anchor check — SCAN the repo first, then confirm; only ask if nothing is found (DR-054).** Do NOT ask the owner blind whether they have a design. BEFORE any exploration:
   - **(a) Scan the repo** for an existing visual / design system: a navigable prototype (`prototype/*.html`, any `*.html` mockups), `docs/design/` (`DESIGN.md`, `design-tokens.json`, `references.md`, `mockups/`), a token/CSS system in the stylesheets (CSS custom properties, a `@theme`/tokens file), screenshots, and the idea card / `.pandacorp/idea-origin.md`.
   - **(b) If something is found → present it and let the owner confirm (in Spanish), don't assume:** *"Encontré esto como posible referencia visual: `<list the files found>`. ¿Es esta la referencia definitiva de diseño? ¿Falta alguna, o no son estas?"* The owner confirms / corrects / adds. On confirmation (or correction) → **ADOPT-VISUAL path** on the confirmed artifact(s).
   - **(c) If nothing is found → ask whether it lives elsewhere:** *"No encontré ningún diseño en el repo. ¿Tienes un prototipo o sistema de diseño en otro sitio (p.ej. Figma, otra carpeta)? ¿Dónde?"* If they point somewhere → ADOPT-VISUAL on it (read it from there; for a Figma, extract the tokens/screens the owner shares). If they say there is none → **EXPLORE path**.
   - **ADOPT-VISUAL path (fidelity, not novelty).** The confirmed artifact is the **binding visual contract**. The `designer` **extracts it faithfully** — its real palette, typography, spacing, radii, shadows, gradients, density and component shapes — into `docs/design/design-tokens.json` and root `DESIGN.md` (this IS the bespoke identity; do **not** invent an alternative, do **not** "relax" or "modernize" it, do **not** demote it to "functional reference only"). Record `docs/design/design-decisions.md` = "frozen FROM the approved visual at `<path>`". **The whole-app prototype's home is `docs/design/prototype/` (DR-056)** — the PDD layer; if the confirmed artifact sits at an ad-hoc loose path (e.g. a top-level `prototype/`), move it there as part of the extraction, so there is one defined place for it. **Complete extraction — capture EVERY layer (DR-056), not just the palette:** an approved prototype is layered — base components AND any app-wide skin/theme overrides AND each embedded sub-view's own spec are all part of the contract. Extract all of them; partial extraction silently drops the look (this actually happened in Mission Control: an app-wide RPG skin plus a pixel-art sub-view spec were missed, and the build diverged). If the approved visual embeds sub-views (e.g. iframes), each embedded file is part of the contract — extract its own spec too, and stage any assets it needs (fonts, images, sprites) so the build can serve them. **SKIP Step 4 entirely** (no 3-direction exploration). The visual gate (Step 6) becomes a **fidelity check** ("does the extracted system match your prototype, pixel-faithfully?"), not a direction choice. A "reference for functionality only / invent a fresh look" instruction is **forbidden** here — the owner already chose the look.
   - **EXPLORE path** (nothing exists, here or elsewhere): continue with Steps 1–7 (research → directions → choose → freeze). **Generate with Claude Design when available (DR-058); else hand-author the HTML directions (Steps 4–6).** **How Claude Design really connects (verified 2026-06-27, correcting earlier assumptions):** the **generative engine is the web canvas** (`claude.ai/design`), driven by the **owner with a prompt** — the agent **cannot pilot the generator remotely**. What the agent has is the **`DesignSync` tool**: **file sync only** (`list_projects`/`get_file`/`list_files`/`create_project`/`finalize_plan`/`write_files`/`delete_files`). Auth is a **one-time `/design-login`** in an interactive terminal (per machine/user, persists; NOT runnable from a headless/SDK session). `/design-sync` is **push-only** (bundles existing React code → claude.ai/design), NOT a generator and NOT a pull; **`/design` may not exist** in the owner's build. **Never assert a slash command exists/doesn't from a filesystem search** (built-in skills are bundled at runtime, not greppable) — **defer to the owner's autocomplete**. Plan B if the agent session can't authorise: owner generates on the canvas + "Send to Claude Code Web", or runs `/design-sync` from their terminal. **Procedure — a TWO-STAGE cadence; NEVER stop at the landing (DR-101).** (1) no design-system project → `create_project`; (2) **upload ALL context** the canvas needs as markdown (master brief + `microcopy.md` + `voice-and-tone.md` + `references.md` + the per-surface playbooks `references/surface-playbooks.md`); (3) **STAGE 1 — the design SYSTEM:** hand the owner **ONE exhaustive generation prompt** carrying the **anti-omission completeness checklist** (every component + every state + image placeholders + a mandatory motion layer + the right breakpoints by `target_platforms` + the hierarchy/proportion + real-content requirements) — see `factory/standards/design.md` §1c; **PULL-and-REVIEW** (`list_files`/`get_file`) against that checklist + the per-screen rubric (§12), issue corrective prompts, then the owner approves the **system**; (4) **STAGE 2 — ALL the screens:** **enumerate the full screen set from the FRDs yourself** (every FRD with `ui:true` → its screen(s), listed by the skill) and drive generation **one screen at a time**, each prompt carrying that surface's playbook (`references/surface-playbooks.md`), each screen **pulled-and-reviewed against the rubric (§12) before the owner sees it** — the owner must never have to ask "where are the other pages?"; (5) integrate (tokens→`design-tokens.json`+`DESIGN.md`, `_ds_manifest.json`→`components.md`, shard per FRD). **Prompt discipline (DR-101):** every canvas prompt is **explicit and unambiguous** — name the element, state the action AND what stays (*"Delete the ENTIRE element including its text and link; replace with nothing"*, not a vague *"remove the note"*). **Hierarchy = importance (§9):** the primary action/conversion gets prominent treatment (the shared `cta-band`), never an undersized one-line link; no context-free hero opening. **Real content (§10):** derive lists/cards/stack/archive from the owner's REAL sources, balanced lengths; placeholders only where data is genuinely pending, visibly marked. **Tokens:** here Claude Design PROPOSES the palette/system from the brief's qualitative direction (mood, dark-default + first-class light, one rationed accent, OKLCH, AA both themes) and the agent **verifies AA + freezes** it (on ADOPT-VISUAL the agent still EXTRACTS exact tokens). Keep the uploaded `BRIEF.md` in sync when the direction changes. Beta + needs the owner's claude.ai login; **offer it, don't force it.**

1. **Visual research** (`designer` agent): references from 3-5 well-designed apps in the domain → `docs/design/references.md`.
2. **Design system (bespoke per domain)**: the `designer` defines `docs/design/design-tokens.json` (palette, typography, spacing, radii) and `DESIGN.md` at the root (following `${CLAUDE_PLUGIN_ROOT}/templates/docs/design-system-template.md`; tokens + allowed components + prohibitions). shadcn/ui is the **accessible component base, not the look**: the palette/typography/mood are tailored to THIS app's domain and audience (tweakcn.com only as a token-format reference). Each app has its own identity; never reuse Mission Control's RPG style (that's the factory's internal tool, not a product template).
3. **Voice and microcopy** (`copywriter` agent, in parallel with the design system): defines `docs/design/voice-and-tone.md` and writes the real microcopy for the key screens of the FRDs (labels, buttons, empty/loading/error states, onboarding) with i18n keys. The `designer` consumes these texts in the mockups instead of inventing them — so the text stops being improvised filler and keeps a consistent voice.
4. **3 design directions in parallel** (*explore path only — SKIPPED when an approved visual was adopted in Step 0*) (3 `designer` agents, one per direction, genuinely distinct): `docs/design/mockups/direction-{1,2,3}.html` — self-contained, navigable, mobile-first, covering the key screens of the FRDs with the `copywriter`'s real microcopy (never lorem ipsum). These are **product-level** exploration to pick the design system — they stay in `docs/design/`, NOT inside any FRD module.
5. **Automatic verification before the gate**: screenshots at 375px/1280px (Playwright) → `docs/design/mockups/screenshots/`; accessibility (axe-core) → `docs/design/a11y-report.md`. Serious violations are fixed BEFORE presenting.
6. **VISUAL GATE (the owner)**: present the 3 directions (open the HTML or show them the screenshots) and wait for their choice or feedback. **Iterate in place (DR-032)**: first read `.pandacorp/comms/iteration.md` (phase `design`); for each round of changes refine the mockups and **append** an entry (what was tried, what was rejected and why, what's still open). Choosing a direction does **not** force advancing: the owner can keep polishing it for as many rounds as they want. Re-running `design` resumes from the journal — it doesn't regenerate from scratch or repeat what was discarded.
7. **Freeze the contract**: chosen direction → final `design-tokens.json`, `docs/design/design-decisions.md` with the rationale, remove ambiguity from DESIGN.md. The `copywriter` leaves the final strings with their i18n keys so the implementation doesn't rewrite them. **Produce `docs/design/components.md` (following `${CLAUDE_PLUGIN_ROOT}/templates/docs/components-inventory-template.md`) — the living component inventory (DR-057), REQUIRED for any project with UI.** It lists each shared primitive/module the build will reuse: one row per component with **name · purpose · path · key props/variants**. Seed it from the frozen design system (the ADOPT-VISUAL extraction or the EXPLORE direction). **With Claude Design (DR-058), bridge `_ds_manifest.json` → `docs/design/components.md`:** convert each manifest entry into an inventory row (manifest component name → name, its description/role → purpose, its synced repo path → path, its variants/props → key props/variants); the manifest is the source, `components.md` is the build-facing inventory the foundation work order implements and every later agent reuses (`check before you create`).
8a. **Capture the persistent APP SHELL as a foundation concern — BEFORE sharding (DR-075).** A whole-app prototype is a SPA with a persistent shell (topbar/primary nav + active-state, header/footer, the layout frame every screen mounts into). The per-FRD shard in Step 8 deliberately gives "only that FRD's screens, never the whole app" — so the **global shell belongs to no FRD and the shard drops it** (the exact Mission Control failure: the app shipped with no nav menu, and the consistency-only visual gate blessed it green). The shell is **not** a new FRD (it would be an orphan in the `REQ-NN-MMM → WO-NN-MMM` spine and the engine has no "build this FRD between foundation and surfaces" primitive); it is a **foundation concern**, captured here so the build constructs it FIRST and the Shell-Presence Gate can enforce it. Do BOTH:
   - **(a) Inventory it as the foundation `AppShell`/`Nav` (DR-057).** Identify the prototype's persistent chrome and add it to `docs/design/components.md` as a shared foundation component (`AppShell` + `Nav`; `artifacts` include `app/layout.tsx` + the nav component). `blueprint`'s Build Plan then schedules it in the **foundation wave** (built before any surface fans out, DR-057); every surface route mounts into `app/layout.tsx`. If the prototype has **no** persistent chrome (a single-screen tool / landing / API / scraper) → there is no shell; skip both and leave the seed empty.
   - **(b) Record the gate's nav contract from the PROTOTYPE, for `blueprint` to materialize into `e2e/shell.ts` (DR-075).** Read the prototype's persistent-nav block (the `topbar()`/header markup in `docs/design/prototype/`) and record, in the `components.md` `AppShell` row (and/or `docs/design/design-decisions.md`): the shell landmark the build must expose (e.g. `[data-app-shell]`), the **top-level** nav destinations as `{label, path}` pairs — **from the prototype, NOT from `routes.ts` SURFACES**, which is a superset of detail/modal/sub routes — any routes that legitimately render WITHOUT the shell, and a mobile drawer/hamburger if the design uses one. **The contract is prototype-anchored** (it drifts only when the prototype changes, like a blessed baseline) — never derived from the app's own routes, or the gate degrades into the same consistency-not-fidelity trap it exists to kill. (The `e2e/` gate machinery doesn't exist yet at design time — `blueprint` installs it and seeds `e2e/shell.ts` from this recorded contract; on the brownfield `adopt` path the install + seed happen together.)
8. **Per-feature design — SHARD the prototype into per-FRD mocks (DR-049 + DR-056)**: with the global system frozen, the design of a specific UI feature lives **inside its FRD module**, not in `docs/design/`. **For every FRD that has UI this is REQUIRED, not optional** (DR-054). The path decides how the per-FRD `mocks/` are produced:
   - **ADOPT-VISUAL path (a whole-app prototype was approved) → SHARD it (DR-056).** The whole-app prototype (in `docs/design/prototype/`) is **scoped DOWN to one FRD at a time** — this is the #1 visual-fidelity lever for AI implementers: ground each work order with just its own screen, not the whole app. For EACH UI FRD: extract that FRD's relevant screen(s) out of the whole-app prototype into `docs/frds/frd-NN-<slug>/mocks/` as a **self-contained, scoped HTML slice — only that FRD's screens, never the whole app** (inline its slice of components/skin/tokens so it stands alone) plus a screenshot; write `docs/frds/frd-NN-<slug>/fdd.md` referencing that mock; and set the FRD frontmatter `ui: true` + `visual_source` (pointing at the prototype path it was sharded from). The shard must carry **every layer** the screen relies on (base components + app-wide skin/theme + any embedded sub-view spec, per the complete-extraction rule in Step 0) — a shard missing the skin silently drops the look.
   - **EXPLORE path (no prototype) → generate per-FRD mocks from the frozen design system.** For each UI FRD the `designer` generates `docs/frds/frd-NN-<slug>/mocks/` (its screens on the frozen tokens) + `fdd.md`, derived from and visually consistent with the frozen contract.
   - Either way: a UI FRD with no `fdd.md` + `mocks/` is incomplete. Each `fdd.md` follows `${CLAUDE_PLUGIN_ROOT}/templates/docs/fdd-template.md`. The design system itself is **never duplicated per feature**; the per-FRD `mocks/` are screens, on the frozen tokens. Source-of-truth: `FRD > FDD > design-tokens > blueprint > work order`.
8b. **Functional reconciliation prototype ↔ FRD — BIDIRECTIONAL, before the prototype is canonical (DR-064)** *(ADOPT-VISUAL path)*: blessing an approved prototype reconciles **behaviour**, not only the look (the Mission Control hole: behaviour the owner built into the prototype was never in the FRDs, and FRD behaviour the prototype never showed grounded a false visual gate). Walk **both directions and resolve every item explicitly** (~7 of each in MC):
   - **Group A — prototype HAS it, FRDs LACK it → update the FRD.** For every interaction/state/screen/affordance/flow the prototype shows that no FRD documents, write it into the owning FRD (acceptance criteria + `fdd.md`) so the build produces it. Exception: a demo-only control (DR-061) is **marked**, not promoted to a requirement.
   - **Group B — FRD WANTS it, prototype LACKS it → build it into the prototype, or flag the hole.** For every behaviour an FRD specifies that the prototype doesn't show, either add the screen to the prototype (so the visual/fidelity gate has a real target) or **flag it as an explicit gap** — never silently ground a fidelity gate on a screen that doesn't exist.
   Present the itemized A/B list to the owner and resolve it **before** freezing the prototype as canonical; a visual-only re-anchor that leaves functional divergence is **not done**. See `factory/standards/design.md` §8.
9. **Advance gate (DR-032 + DR-054 deliverable check)**: as long as the owner hasn't given their "ok, advance", leave `advance_pending: true` in `.pandacorp/status.yaml` and do **not** write the transition. **Before writing the transition, verify the contract artifacts actually exist** (existence is checked, not self-asserted — a missing artifact is a *not-done*, not advanceable even on an owner "ok"): root `DESIGN.md`, `docs/design/design-tokens.json`, `docs/design/design-decisions.md`, **a non-empty `docs/design/components.md`** (the component inventory, DR-057 — REQUIRED for any project with UI; without it the build's reuse mechanism is inert and the reviewer false-positives), and for **every** FRD with UI a non-empty `fdd.md` + `mocks/`. (This is the check whose absence let Mission Control reach build with code referencing a "frozen" `design-tokens.json` that did not exist.) **On the ADOPT-VISUAL path, also confirm the functional reconciliation (Step 8b, DR-064) is resolved** — no un-itemized Group A/B divergence remains between the prototype and the FRDs. Only when the artifacts are present AND the owner approves: `.pandacorp/status.yaml` → `phase: architecture` + `advance_pending: false`, and close the `design` thread in `.pandacorp/comms/iteration.md`. Next step: `/pandacorp:architecture`.

## Rules
- The later implementation can ONLY use the frozen tokens — this contract is the mechanism that compensates for the owner's weakness in design.
- **An owner-approved visual outranks "invent something fresh" (DR-054).** When the owner has designated a prototype/design as the visual target, fidelity to it wins over novelty: the design phase reproduces it in the tokens/`DESIGN.md`, does NOT generate alternative directions, and the build must look like it. The "bespoke from scratch / 3 directions" path below applies **only when there is no approved visual**.
- Empty, loading and error states are designed here, not improvised while coding.
- **Bespoke per domain, never a house style (explore path).** Absent an approved visual, each app's look & feel is designed from scratch for its sector and audience; shadcn/ui is only the accessible component base. Mission Control's RPG/"guild" style is the factory's internal tool and is never applied to product apps.
- **Claude Design is the preferred design engine when available (DR-058) — but the agent can't pilot it; the OWNER generates on the canvas.** EXPLORE path: prefer Claude Design over hand-coded HTML. The agent's role is **file sync (the `DesignSync` tool) + an exhaustive brief + pull-and-review**, NOT remote generation — the generator is the owner-driven `claude.ai/design` canvas; `/design-sync` is push-only React code; auth is a one-time `/design-login`; never assert command availability from the filesystem (defer to the owner's autocomplete). The agent uploads ALL context, hands the owner **ONE exhaustive prompt** built from the **anti-omission checklist (§1c)**, then **PULLS and REVIEWS** the output against the checklist before the gate, and integrates (`_ds_manifest.json`→`docs/design/components.md`, DR-057). ADOPT-VISUAL: MAY push the extracted system **up** for central management + self-check. Beta + needs the owner's login; **offer it, don't force it.** Full procedure + the completeness checklist: `factory/standards/design.md` §1c.
- **Generate the FULL screen set, in two stages (DR-101).** Claude Design generation is a **two-stage cadence and never stops at the landing**: Stage 1 generates the design **system** (owner approves it), then Stage 2 generates **ALL** the UI-FRD screens — **enumerated by the skill from the FRDs** (`ui:true` → its screen(s)) — **one screen at a time with pull-and-review** against the per-screen rubric (`factory/standards/design.md` §12) and that surface's playbook (`references/surface-playbooks.md`). The owner must never have to ask "where are the other pages?". **Prompts are explicit** (name the element + the action + what stays). **Weight = importance (§9):** the primary action/conversion gets the prominent shared `cta-band`, never an undersized link; no context-free hero. **Real content (§10):** derived from the owner's real sources, never invented filler. Per-surface best practices (case study / Now / blog index + post / contact / projects-archive / stack): `references/surface-playbooks.md`.
- **Seed the component inventory (DR-057).** The frozen design system MUST leave a living `docs/design/components.md` listing each shared primitive/module (name · purpose · path · key props/variants) — the foundation the build's first work order implements and every later agent reuses (`check before you create`).
- **One cohesive application (DR-062).** The app must feel like ONE application, not screens each designed in isolation. Standardize the **framing patterns** across every surface: ONE page-title block (and the page H1 **equals its nav label** — extra context goes in a subtitle, never a different title), ONE section-header style, ONE tab style at each level, one chip/panel/command/empty-state language. A surface that invents its own variant of an existing pattern is a **cohesion defect** (same class as a duplicate component). These framing patterns are first-class entries in `docs/design/components.md`, not per-screen ad-hoc markup; deviations (an intentionally immersive view) are justified, not silent. The `reviewer` checks cross-surface consistency. Canonical: `factory/standards/design.md` §7.
- **Mark demo-only controls in prototypes (DR-061).** Any affordance a mockup adds purely to *preview states* (a mode/effort picker, play/pause, reset, state togglers) that will NOT exist in the real (read-only or skill/CI-driven) app MUST be wrapped in a **dashed-border block with a small uppercase `DEMO` tag + a one-line note of how the real thing actually works** — so the owner never reads a demo affordance as real behaviour. Any real value the app *will* surface (e.g. the effort a build ran with) goes as **read-only data in a real UI surface**, not only inside the demo block. Canonical pattern in `factory/standards/design.md` §6; it applies to every prototype the factory generates.
- **Feasibility before freezing (order fix).** The `designer` emits `docs/design/technical-assumptions.md` (costly/risky interactions the design assumes) for the `architect` to read in `/pandacorp:architecture`; a high-risk assumption triggers a feasibility spike before the design contract is frozen, so the owner never approves a design the architecture can't support cheaply.
