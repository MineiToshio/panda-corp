# /pandacorp:design

Design phase. Runs IN the project (requires `docs/product/prd.md` and `docs/frds/` — if they are missing, run `/pandacorp:spec` first).

This phase works at **two layers (DR-049)**: it establishes the project's **global design system** (the PDD — `docs/design/` + root `DESIGN.md`, ONE per project, never per-feature), and when it designs a specific UI feature it writes that feature's design **inside the FRD module** (`docs/frds/frd-NN-<slug>/fdd.md` + `mocks/`). The product-level **direction** mockups below are the exploration that picks the system; they are a different thing from a feature's `mocks/`.

## Steps

0. **Anchor check — SCAN the repo first, then confirm; only ask if nothing is found (DR-054).** Do NOT ask the owner blind whether they have a design. BEFORE any exploration:
   - **(a) Scan the repo** for an existing visual / design system: a navigable prototype (`prototype/*.html`, any `*.html` mockups), `docs/design/` (`DESIGN.md`, `design-tokens.json`, `references.md`, `mockups/`), a token/CSS system in the stylesheets (CSS custom properties, a `@theme`/tokens file), screenshots, and the idea card / `.pandacorp/idea-origin.md`.
   - **(b) If something is found → present it and let the owner confirm (in Spanish), don't assume:** *"Encontré esto como posible referencia visual: `<list the files found>`. ¿Es esta la referencia definitiva de diseño? ¿Falta alguna, o no son estas?"* The owner confirms / corrects / adds. On confirmation (or correction) → **ADOPT-VISUAL path** on the confirmed artifact(s).
   - **(c) If nothing is found → ask whether it lives elsewhere:** *"No encontré ningún diseño en el repo. ¿Tienes un prototipo o sistema de diseño en otro sitio (p.ej. Figma, otra carpeta)? ¿Dónde?"* If they point somewhere → ADOPT-VISUAL on it (read it from there; for a Figma, extract the tokens/screens the owner shares). If they say there is none → **EXPLORE path**.
   - **ADOPT-VISUAL path (fidelity, not novelty).** The confirmed artifact is the **binding visual contract**. The `designer` **extracts it faithfully** — its real palette, typography, spacing, radii, shadows, gradients, density and component shapes — into `docs/design/design-tokens.json` and root `DESIGN.md` (this IS the bespoke identity; do **not** invent an alternative, do **not** "relax" or "modernize" it, do **not** demote it to "functional reference only"). Record `docs/design/design-decisions.md` = "frozen FROM the approved visual at `<path>`". **The whole-app prototype's home is `docs/design/prototype/` (DR-056)** — the PDD layer; if the confirmed artifact sits at an ad-hoc loose path (e.g. a top-level `prototype/`), move it there as part of the extraction, so there is one defined place for it. **Complete extraction — capture EVERY layer (DR-056), not just the palette:** an approved prototype is layered — base components AND any app-wide skin/theme overrides AND each embedded sub-view's own spec are all part of the contract. Extract all of them; partial extraction silently drops the look (this actually happened in Mission Control: an app-wide RPG skin plus a pixel-art sub-view spec were missed, and the build diverged). If the approved visual embeds sub-views (e.g. iframes), each embedded file is part of the contract — extract its own spec too, and stage any assets it needs (fonts, images, sprites) so the build can serve them. **SKIP Step 4 entirely** (no 3-direction exploration). The visual gate (Step 6) becomes a **fidelity check** ("does the extracted system match your prototype, pixel-faithfully?"), not a direction choice. A "reference for functionality only / invent a fresh look" instruction is **forbidden** here — the owner already chose the look.
   - **EXPLORE path** (nothing exists, here or elsewhere): continue with Steps 1–7 (research → 3 directions → choose → freeze). **Prefer Claude Design (DR-058):** when proposing a design from scratch, use Anthropic's specialized design tool — the `/design-sync` skill + the `DesignSync` tool (claude.ai/design) — to generate a *coherent system + reusable component library*, then sync it into the repo; its `_ds_manifest.json` becomes the component inventory (`docs/design/components.md`, DR-057). Hand-authoring HTML directions (Steps 4–6) is the fallback when Claude Design isn't available. Claude Design needs the owner's claude.ai login + design-system access (the tool prompts once); it is a beta dependency — offer it, don't force it.

1. **Visual research** (`designer` agent): references from 3-5 well-designed apps in the domain → `docs/design/references.md`.
2. **Design system (bespoke per domain)**: the `designer` defines `docs/design/design-tokens.json` (palette, typography, spacing, radii) and `DESIGN.md` at the root (tokens + allowed components + prohibitions). shadcn/ui is the **accessible component base, not the look**: the palette/typography/mood are tailored to THIS app's domain and audience (tweakcn.com only as a token-format reference). Each app has its own identity; never reuse Mission Control's RPG style (that's the factory's internal tool, not a product template).
3. **Voice and microcopy** (`copywriter` agent, in parallel with the design system): defines `docs/design/voice-and-tone.md` and writes the real microcopy for the key screens of the FRDs (labels, buttons, empty/loading/error states, onboarding) with i18n keys. The `designer` consumes these texts in the mockups instead of inventing them — so the text stops being improvised filler and keeps a consistent voice.
4. **3 design directions in parallel** (*explore path only — SKIPPED when an approved visual was adopted in Step 0*) (3 `designer` agents, one per direction, genuinely distinct): `docs/design/mockups/direction-{1,2,3}.html` — self-contained, navigable, mobile-first, covering the key screens of the FRDs with the `copywriter`'s real microcopy (never lorem ipsum). These are **product-level** exploration to pick the design system — they stay in `docs/design/`, NOT inside any FRD module.
5. **Automatic verification before the gate**: screenshots at 375px/1280px (Playwright) → `docs/design/mockups/screenshots/`; accessibility (axe-core) → `docs/design/a11y-report.md`. Serious violations are fixed BEFORE presenting.
6. **VISUAL GATE (the owner)**: present the 3 directions (open the HTML or show them the screenshots) and wait for their choice or feedback. **Iterate in place (DR-032)**: first read `.pandacorp/comms/iteration.md` (phase `design`); for each round of changes refine the mockups and **append** an entry (what was tried, what was rejected and why, what's still open). Choosing a direction does **not** force advancing: the owner can keep polishing it for as many rounds as they want. Re-running `design` resumes from the journal — it doesn't regenerate from scratch or repeat what was discarded.
7. **Freeze the contract**: chosen direction → final `design-tokens.json`, `docs/design/design-decisions.md` with the rationale, remove ambiguity from DESIGN.md. The `copywriter` leaves the final strings with their i18n keys so the implementation doesn't rewrite them. **Produce `docs/design/components.md` — the living component inventory (DR-057), REQUIRED for any project with UI.** It lists each shared primitive/module the build will reuse: one row per component with **name · purpose · path · key props/variants**. Seed it from the frozen design system (the ADOPT-VISUAL extraction or the EXPLORE direction). **With Claude Design (DR-058), bridge `_ds_manifest.json` → `docs/design/components.md`:** convert each manifest entry into an inventory row (manifest component name → name, its description/role → purpose, its synced repo path → path, its variants/props → key props/variants); the manifest is the source, `components.md` is the build-facing inventory the foundation work order implements and every later agent reuses (`check before you create`).
8. **Per-feature design — SHARD the prototype into per-FRD mocks (DR-049 + DR-056)**: with the global system frozen, the design of a specific UI feature lives **inside its FRD module**, not in `docs/design/`. **For every FRD that has UI this is REQUIRED, not optional** (DR-054). The path decides how the per-FRD `mocks/` are produced:
   - **ADOPT-VISUAL path (a whole-app prototype was approved) → SHARD it (DR-056).** The whole-app prototype (in `docs/design/prototype/`) is **scoped DOWN to one FRD at a time** — this is the #1 visual-fidelity lever for AI implementers: ground each work order with just its own screen, not the whole app. For EACH UI FRD: extract that FRD's relevant screen(s) out of the whole-app prototype into `docs/frds/frd-NN-<slug>/mocks/` as a **self-contained, scoped HTML slice — only that FRD's screens, never the whole app** (inline its slice of components/skin/tokens so it stands alone) plus a screenshot; write `docs/frds/frd-NN-<slug>/fdd.md` referencing that mock; and set the FRD frontmatter `ui: true` + `visual_source` (pointing at the prototype path it was sharded from). The shard must carry **every layer** the screen relies on (base components + app-wide skin/theme + any embedded sub-view spec, per the complete-extraction rule in Step 0) — a shard missing the skin silently drops the look.
   - **EXPLORE path (no prototype) → generate per-FRD mocks from the frozen design system.** For each UI FRD the `designer` generates `docs/frds/frd-NN-<slug>/mocks/` (its screens on the frozen tokens) + `fdd.md`, derived from and visually consistent with the frozen contract.
   - Either way: a UI FRD with no `fdd.md` + `mocks/` is incomplete. The design system itself is **never duplicated per feature**; the per-FRD `mocks/` are screens, on the frozen tokens. Source-of-truth: `FRD > FDD > design-tokens > blueprint > work order`.
9. **Advance gate (DR-032 + DR-054 deliverable check)**: as long as the owner hasn't given their "ok, advance", leave `advance_pending: true` in `.pandacorp/status.yaml` and do **not** write the transition. **Before writing the transition, verify the contract artifacts actually exist** (existence is checked, not self-asserted — a missing artifact is a *not-done*, not advanceable even on an owner "ok"): root `DESIGN.md`, `docs/design/design-tokens.json`, `docs/design/design-decisions.md`, **a non-empty `docs/design/components.md`** (the component inventory, DR-057 — REQUIRED for any project with UI; without it the build's reuse mechanism is inert and the reviewer false-positives), and for **every** FRD with UI a non-empty `fdd.md` + `mocks/`. (This is the check whose absence let Mission Control reach build with code referencing a "frozen" `design-tokens.json` that did not exist.) Only when the artifacts are present AND the owner approves: `.pandacorp/status.yaml` → `phase: architecture` + `advance_pending: false`, and close the `design` thread in `.pandacorp/comms/iteration.md`. Next step: `/pandacorp:blueprint`.

## Rules
- The later implementation can ONLY use the frozen tokens — this contract is the mechanism that compensates for the owner's weakness in design.
- **An owner-approved visual outranks "invent something fresh" (DR-054).** When the owner has designated a prototype/design as the visual target, fidelity to it wins over novelty: the design phase reproduces it in the tokens/`DESIGN.md`, does NOT generate alternative directions, and the build must look like it. The "bespoke from scratch / 3 directions" path below applies **only when there is no approved visual**.
- Empty, loading and error states are designed here, not improvised while coding.
- **Bespoke per domain, never a house style (explore path).** Absent an approved visual, each app's look & feel is designed from scratch for its sector and audience; shadcn/ui is only the accessible component base. Mission Control's RPG/"guild" style is the factory's internal tool and is never applied to product apps.
- **Claude Design is the preferred design engine (DR-058).** For the EXPLORE path, generate the system + reusable components with **Claude Design** (the `/design-sync` skill + the `DesignSync` tool, claude.ai/design) — Anthropic's specialized design tool — rather than hand-coded HTML, then sync it into the repo. For ADOPT-VISUAL you MAY push the extracted system **up** to a Claude Design project (design-sync) for central management + self-checking. Either path **seeds `docs/design/components.md`** — the component inventory (DR-057) — so the build reuses components instead of reinventing them (Claude Design's `_ds_manifest.json` IS that inventory). It needs the owner's claude.ai login; offer it, don't force it.
- **Seed the component inventory (DR-057).** The frozen design system MUST leave a living `docs/design/components.md` listing each shared primitive/module (name · purpose · path · key props/variants) — the foundation the build's first work order implements and every later agent reuses (`check before you create`).
- **Feasibility before freezing (order fix).** The `designer` emits `docs/design/technical-assumptions.md` (costly/risky interactions the design assumes) for the `architect` to read in `/pandacorp:blueprint`; a high-risk assumption triggers a feasibility spike before the design contract is frozen, so the owner never approves a design the architecture can't support cheaply.
