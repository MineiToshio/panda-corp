# /pandacorp:upgrade

Brings this project's Pandacorp overlay up to the factory's current version. The `/pandacorp:*` skills always run against the **latest** plugin, so a project born on an older overlay can drift; this closes the gap. Runs IN the project. It touches only the overlay/integration layer — never product code or `docs/`.

## How it decides

- **Project version**: `overlay_version` in `.pandacorp/status.yaml`. A pre-6.0.0 project has no `.pandacorp/` and no `overlay_version` → treat it as the oldest (it needs the overlay-relocation migration). A pre-DR-049 project (overlay ≥ 6.0.0 but with the old **by-type** `docs/` layout — flat `docs/frds/frd-NN.md`, global `docs/blueprint.md`, global `docs/work-orders/`, `docs/prd.md` at the root of `docs/`) additionally needs the **feature-centric docs migration** (step 2b).
- **Target version**: `${CLAUDE_PLUGIN_ROOT}/templates/OVERLAY_VERSION`.
- **In sync** (`overlay_version` == target) → say so and do nothing (idempotent).
- **Behind** → upgrade. **Compatible** bumps (same MAJOR) apply **silently** (DR-048) and you only see the commit; a **MAJOR** jump (breaking — a moved/renamed field or path) is **announced first** with the diff, never applied blind.

## Steps

1. **Detect the gap.** Read the project's `overlay_version` (absent = pre-6.0.0 layout) and the plugin's `OVERLAY_VERSION`. If in sync, stop ("already on overlay X").

2. **Overlay-relocation migration (pre-6.0.0 → 6.0.0+).** If the project still has the OLD layout, move the integration layer into `.pandacorp/` (use `git mv` where tracked to preserve history):
   - `docs/status.yaml` → `.pandacorp/status.yaml`
   - `docs/iteration.md` → `.pandacorp/comms/iteration.md`; `docs/summary.md` → `.pandacorp/comms/summary.md`; `docs/progress.md` → `.pandacorp/comms/progress.md`; `docs/activity.md` → `.pandacorp/comms/activity.md`
   - `docs/decisions.md` → `.pandacorp/inbox/decisions.md`; `docs/bugs/` → `.pandacorp/inbox/bugs/`
   - `docs/idea-origin.md` → `.pandacorp/idea-origin.md`
   - **STAY in `docs/` (product)**: `decision-log.md`, `prd.md`, `frds/`, `blueprint.md`, `adr/`, `work-orders/`, `design/`, `reviews/`, `product-research.md` (the feature-centric migration in step 2b then reshapes these).
   - Update the project's `.gitignore`: add `.pandacorp/comms/`, `.pandacorp/inbox/`, `.pandacorp/run/`; drop the old `docs/{summary,progress,decisions,iteration,activity}.md` ignores.

2b. **Feature-centric docs migration (pre-DR-049 by-type → feature-centric).** If the project still has the old by-type `docs/` shape, reshape it into the feature-centric layout (DR-049 — see `factory/standards/structure.md`). Use `git mv` where tracked; preserve the stable IDs (`REQ-NN-MMM`, `AC-NN-MMM.K`, `WO-NN-MMM`) — numbers never change, only their location:
   - **Product layer**: `docs/prd.md` → `docs/product/prd.md`; `docs/product-research.md` → `docs/product/research.md`. Add a living **feature-landscape** table to `prd.md` if absent (one row per FRD module).
   - **FRD modules**: each flat `docs/frds/frd-NN-<slug>.md` → its own module folder `docs/frds/frd-NN-<slug>/frd.md`. Per-feature design (if the project had a separate per-feature FDD) → `frd-NN-<slug>/fdd.md`; feature mocks → `frd-NN-<slug>/mocks/`. The global design system (`docs/design/`) stays GLOBAL — never split it per feature.
   - **Blueprint split**: the global `docs/blueprint.md` is split — the **platform** parts (stack, data model, deploy, cross-cutting, service boundaries) → `docs/product/architecture.md`; each **per-feature** implementation section → that feature's `docs/frds/frd-NN-<slug>/blueprint.md` (referencing `product/architecture.md`). When the per-feature split isn't cleanly separable, keep the bulk in `architecture.md` and add thin per-feature `blueprint.md` files that point to it; don't fabricate detail.
   - **Work orders**: distribute the global `docs/work-orders/wo-NN-MMM-*.md` into their FRD's folder by their `NN` prefix → `docs/frds/frd-NN-<slug>/work-orders/`, each with a per-FRD `work-orders/README.md` (intra-feature order + parallelism). Cross-feature order continues to come from each WO's `Dependencies`.
   - **Unchanged**: `docs/design/` (the global PDD), `docs/adr/`, `docs/analytics/events.md`, `docs/decision-log.md`, `docs/reviews/` — these stay where they are.
   - Fix internal cross-links broken by the moves (PRD↔FRD, WO→FRD/blueprint references) so they still resolve.

3. **Regenerate the managed layer (non-destructive).**
   - **`.pandacorp/guide.md` and `.pandacorp/README.md`**: regenerate ENTIRELY from the current template (fully managed — no merge), substituting the template vars from `.pandacorp/status.yaml` + factory paths.
   - **Machinery** (`.claude/workflows/pandacorp-build.js` and any factory script the overlay ships): regenerate from the template.
   - **`CLAUDE.md`**: ensure it imports `@AGENTS.md`, `@docs/rules/README.md` and `@.pandacorp/guide.md`. If the project still has the old fat `CLAUDE.md`, replace the managed sections with the thin import header but **keep everything the owner wrote below the managed line**. Never touch the owner's notes.
   - **`AGENTS.md`**: reconcile the managed block (priority order, the `docs/rules/` pointer, language & interaction, write-gate, stack pointer) with the template; keep project-specific additions. A project on an older overlay that still inlines the long durable-conventions block: replace that block with the `docs/rules/` pointer (the rules now live as files in `docs/rules/`, synced just below) — don't leave two copies.
   - **`docs/rules/` (engineering rules re-sync)**: this is how rule changes reach every project. Read the project's stack from `docs/product/architecture.md`. For each file in `${CLAUDE_PLUGIN_ROOT}/templates/rules/`: if its `applies_when`/`also_applies_when` is `always` or matches a technology the project uses, **copy it in verbatim, overwriting the project's copy** (the plugin is the source of truth for these files); if it no longer applies (a technology was dropped), remove the stale project copy. Leave untouched any file the owner added that has no counterpart in the plugin (project-specific rules). Then **regenerate `docs/rules/README.md`** (canonical spec in the `scaffold` skill). For a pre-rules project (no `docs/rules/`), create it now with the applicable set (first-time adoption). Bumping `OVERLAY_VERSION` whenever a rule file changes is what triggers this on the next skill run (DR-051).
   - **Gate-config conformance check (DR-059)** — the meta-fix that catches silent enforcement drift. The gate config propagates VERBATIM, exactly like `docs/rules/`. Read the project's stack from `docs/product/architecture.md` and resolve `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/`. **Diff** the project's `verify.sh` and `canary.sh` (both in `.pandacorp/`), `biome.json` and `knip.json` (project root) against the stack templates; on **any drift** (or a missing file) **regenerate it from the template, overwriting the project's copy** (the plugin is the source of truth) and note the drift in the report — a pre-DR-079 project gets `.pandacorp/canary.sh` (the gate canary `verify.sh --canary` runs) installed for the first time here — a project's enforcement can never silently fall behind the standard (this conformance check is what would have caught the audit: smoke/visual gates, `knip`, the `react`/`next`/`test` domains and `complexity: error` all silently OFF, with nothing checking the wiring). **Loud back-port detector (DR-094): BEFORE overwriting each diverging file, LOUDLY REPORT the diff with the DR-076 instruction** — the divergence may be an in-place fix the overwrite is about to revert, so back-port it to `plugin/templates/stack-<chosen>/` rather than fix in place. The overwrite still proceeds unconditionally (DR-059's guarantee), but a real in-place fix is now impossible to lose silently (exactly how MC's `knip@6` + biome `!**/*.py` were once reverted). A pre-DR-059 project simply gets these three files installed for the first time. Leave untouched any project-specific lint/knip config the owner added in a **separate** file (only the canonical ones are conformance-checked).
     - **The e2e gate machinery is conformance-checked too (DR-055/056/074/075).** Extend the byte-diff/overwrite-on-drift set to the **six VERBATIM e2e files**: `e2e/{smoke.spec.ts,visual.spec.ts,responsive.spec.ts,_responsive-helper.ts,shell.spec.ts}` and `playwright.config.ts` (project root). Diff each against the stack template and regenerate on drift, exactly like `verify.sh`. A pre-DR-074 project gets `responsive.spec.ts`/`_responsive-helper.ts`, a pre-DR-075 project gets `shell.spec.ts` (and any missing of the others) installed for the first time. **NEVER byte-diff `e2e/routes.ts`, `e2e/_target.ts` nor `e2e/shell.ts`** — they are PER-PROJECT (the project's own surface list / platform read / app-shell nav contract); **create them from the template only if missing, never overwrite** (overwriting `routes.ts` would clobber the project's blessed surface manifest, and overwriting `shell.ts` its nav contract = data loss). **Back-fill `e2e/shell.ts` EMPTY (DR-075) — never a filled contract:** a pre-DR-075 project gets the template's empty-`NAV_DESTINATIONS` seed (a vacuous-pass gate), exactly the conservative analogue of the `target_platforms: desktop` back-fill below — filling it with the prototype's nav on an app that has no shell would turn the new gate RED and **red-lock all commits**. The real contract is filled deliberately, in the SAME change that builds the shell (the re-anchor), never by the upgrade.
   - **Toolchain conformance check (DR-059, extends the gate-config check)** — a byte-perfect `verify.sh` still goes RED at cold start if the tools it calls aren't installed: a missing `knip` or `playwright` fails the gate even with a canonical config. Conformance therefore covers the **toolchain**, not just the config files — this is what makes the fail-closed gate actually *runnable*. After the config diff, read the project's `package.json` and reconcile it against what the canonical `verify.sh` requires (resolve from the same `stack-<chosen>/` template); on any gap, install/fix it and note it in the report:
     - **Required devDependencies** — `knip`, `madge`, `@playwright/test`, `vitest`, `@biomejs/biome`, `@axe-core/playwright` (the responsive gate's tap-target check, DR-074). Any missing from `devDependencies` → install (`@latest` for a current project; framework-major-compatible for brownfield, DR-052).
     - **Required `package.json` scripts** the canonical `verify.sh` invokes — `test:smoke` (`playwright test e2e/smoke.spec.ts`, DR-055), `test:visual` (`playwright test e2e/visual.spec.ts`, DR-056), `test:responsive` (`playwright test e2e/responsive.spec.ts`, DR-074) and `test:shell` (`playwright test e2e/shell.spec.ts`, DR-075). Add any that are absent.
     - **Strict `tsconfig`** — ensure `compilerOptions.strict: true` **and** `noUncheckedIndexedAccess: true`; set whichever is missing or `false`.
     Gated by the same `OVERLAY_VERSION` bump (DR-051) and reported alongside the config drift.
   - **`status.yaml`**: set `overlay_version` to the target. For `created_with` (immutable — the version the project was BORN on): keep it if present; for a project migrated from a pre-6.0.0 layout, set it to `pre-6.0.0` (it was not born on the current overlay).
   - **Back-fill `target_platforms` for an existing project to `desktop` (DR-074)** — if the field is absent, add `target_platforms: desktop`, NOT `responsive`. This is the conservative, currently-passing default: an existing project may be non-responsive, so back-filling `responsive` would turn the new mobile responsive gate RED immediately and **block all commits** (the red-lock). `desktop` keeps the gate green (the responsive checks become a vacuous pass); promoting to `responsive` is a deliberate, SEPARATE change once the project is verified responsive at mobile width. A NEW project, by contrast, is born `responsive` (the status.yaml template default) — only the retrofit is conservative.

4. **Bump + record.** Set `overlay_version` to the target in `.pandacorp/status.yaml`. Add a `docs/decision-log.md` entry: "Pandacorp overlay upgraded `<old>` → `<new>`"; if step 2b ran, note the feature-centric docs migration (DR-049) and that it was a pure relocation (IDs preserved, no behavior changed); if the gate-config conformance check (DR-059) regenerated `verify.sh`/`biome.json`/`knip.json`, note which drifted and that enforcement was re-aligned to the standard; if the toolchain check installed missing devDependencies/scripts or fixed `tsconfig`, note that too (the gate is now runnable cold).

5. **Commit on its own**: `chore: upgrade pandacorp overlay <old> → <new>` (moved files + machinery + managed layer + any DR-049 docs relocation from step 2b). Do NOT mix product *behavior* changes into this commit — the docs migration is a structural move, not a content rewrite.

6. **Report** (in Spanish): what moved, what was regenerated, and the new version. If a MAJOR jump required a manual decision, surface it explicitly.

## Rules
- **Compatible (same MAJOR) = silent** (DR-048): apply and commit without interrupting; the owner sees it via the commit. **MAJOR = announce** the breaking parts + diff before applying.
- **Never clobber the owner's content**: managed files (`guide.md`, `README.md`, machinery) regenerate whole; `CLAUDE.md`/`AGENTS.md` merge by managed-section, the owner's notes stay.
- Touch only the overlay/integration layer — never product **code**, and never rewrite the **content** of product docs. The one exception is the DR-049 **feature-centric docs migration** (step 2b): a one-time structural *relocation* of `docs/` files (moves + link fixes, IDs preserved), never a content rewrite. A MAJOR overlay jump that carries this migration is announced first like any breaking change.
- **Gate config is conformance-checked, not merged (DR-059)**: `verify.sh`/`biome.json`/`knip.json` **and the six VERBATIM e2e files** (`e2e/{smoke,visual,responsive,shell}.spec.ts`, `e2e/_responsive-helper.ts`, root `playwright.config.ts`) are canonical — the plugin is the source of truth and any project drift is overwritten from the template, exactly like `docs/rules/`. **`e2e/routes.ts`, `e2e/_target.ts` and `e2e/shell.ts` are PER-PROJECT and NEVER byte-diffed** (created only if missing; `shell.ts` back-filled EMPTY, DR-075) — byte-diffing the surface manifest / nav contract would clobber the project's data. The owner's project-specific lint/knip tweaks belong in a separate file, which is left untouched.
- **Canonical gate files are edited at the SOURCE, never in-place (DR-076)**: because conformance overwrites them, a fix made directly in a project's `biome.json`/`knip.json`/`.pandacorp/verify.sh`/the six VERBATIM e2e files/`playwright.config.ts` is **silently reverted on the next `/upgrade`** AND leaves the template stale for every other project (exactly how MC's `knip@6` fix was lost while the template kept the v6-rejected `_pandacorp` key, until the re-anchor caught it). So a genuine fix MUST be made in `plugin/templates/stack-<chosen>/` and propagated via `/upgrade` (back-port in the SAME change); a genuinely project-specific exception goes in a SEPARATE never-overwritten file, never by editing the canonical one. The `remind-backport-gate-config.sh` hook nudges this at edit time. **Conformance therefore assumes the template is never behind a project** — when a project legitimately diverges (a fix landed in-place), the answer is to back-port to the template, not to weaken the overwrite (which stays unconditional — DR-059's binary guarantee is preserved, NOT a "diff-and-flag" that would re-admit silent drift).
- **Conformance covers the toolchain, not just the config (DR-059)**: a byte-perfect `verify.sh` is useless if `knip`/`playwright`/`@axe-core/playwright` aren't installed — the check also reconciles the required devDependencies, the `test:smoke`/`test:visual`/`test:responsive` scripts and a strict `tsconfig`, so the fail-closed gate is actually runnable at cold start.
- **Prove the gates still BITE after re-syncing them (DR-079)**: once the gate-config conformance has regenerated `verify.sh` + `canary.sh` + the e2e files, run `.pandacorp/verify.sh --canary` — `canary.sh` GENERATES a deliberately-broken input per gate (no broken files kept in the repo), pushes each through its gate, and REDs if any gate that should reject its broken input stays green (a rotted gate: a renamed selector, a disabled rule, a swallowed exit code). v1 canaries the cheap, deterministic, config-rot-prone gates — biome (`--error-on-warnings`), the test-placement structure guard, and the DR-100 readiness gate; the heavier gates (tsc/vitest/knip/Playwright) are the documented next additions, simply not canaried yet (never falsely RED). Still a clean pass on a project that has no `canary.sh` yet (older overlay), so it never blocks an upgrade; note the result in the report. Conformance proves the gate FILES match the template (DR-059/076); the canary proves they still FUNCTIONALLY reject.
- Idempotent: re-running when already in sync is a no-op.
