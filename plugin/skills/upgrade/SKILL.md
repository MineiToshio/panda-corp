# /pandacorp:upgrade

Brings this project's Pandacorp overlay up to the factory's current version. The `/pandacorp:*` skills always run against the **latest** plugin, so a project born on an older overlay can drift; this closes the gap. Runs IN the project. It touches only the overlay/integration layer — never product code or `docs/`.

> **Active-build guard (HARD, runs before anything else).** Read `running`, `supervisor_heartbeat` and
> `run_started_at` from `.pandacorp/status.yaml`. If `running: true` AND the heartbeat is **less than 10
> minutes old**, **ABORT — do not upgrade**: this skill regenerates `.claude/engines/pandacorp-build.js`,
> overwrites the conformance-managed gate files (`verify.sh`, `biome.json`, e2e specs) and **commits on its
> own** — doing that under a live build races the engine's single-git-writer chain (DR-086) and can change
> gate semantics between a WO's self-test and its FRD gate. Tell the owner (in Spanish) that the upgrade is
> deferred until the build stops, and let the CALLING skill proceed on the current overlay (capture-only
> skills like `change`/`bug` work fine on an older overlay). A stale heartbeat (≥ 10 min) means the
> supervisor died — treat as not running (same TTL rule as `/pandacorp:implement`'s concurrent-run guard).

## How it decides

- **Project version**: `overlay_version` in `.pandacorp/status.yaml`. A pre-6.0.0 project has no `.pandacorp/` and no `overlay_version` → treat it as the oldest (it needs the overlay-relocation migration). A pre-DR-049 project (overlay ≥ 6.0.0 but with the old **by-type** `docs/` layout — flat `docs/frds/frd-NN.md`, global `docs/blueprint.md`, global `docs/work-orders/`, `docs/prd.md` at the root of `docs/`) additionally needs the **feature-centric docs migration** (step 2b).
- **Target version**: `${CLAUDE_PLUGIN_ROOT}/templates/OVERLAY_VERSION`.
- **In sync** (`overlay_version` == target) → say so and do nothing (idempotent).
- **Behind** → upgrade. **Compatible** bumps (same MAJOR) apply **silently** (DR-048) and you only see the commit; a **MAJOR** jump (breaking — a moved/renamed field or path) is **announced first** with the diff, never applied blind.

## Steps

1. **Detect the gap.** Read the project's `overlay_version` (absent = pre-6.0.0 layout) and the plugin's `OVERLAY_VERSION`. If in sync, stop ("already on overlay X").

2. **Overlay-relocation migration (pre-6.0.0 → 6.0.0+).** If the project still has the OLD layout, move the integration layer into `.pandacorp/` (use `git mv` where tracked to preserve history):
   - `docs/status.yaml` → `.pandacorp/status.yaml`
   - `docs/iteration.md` → `.pandacorp/comms/iteration.md`; `docs/summary.md` → `.pandacorp/comms/summary.md`; `docs/progress.md` → `.pandacorp/comms/progress.md`; `docs/activity.md` → `.pandacorp/comms/activity.md`
   - `docs/decisions.md` → `.pandacorp/inbox/decisions.md`; `docs/bugs/` → `.pandacorp/inbox/changes/` (each old bug file becomes a `type: bug` item in the unified queue — DR-069; an old `.pandacorp/inbox/bugs/` from a 6.x overlay migrates the same way)
   - `docs/idea-origin.md` → `.pandacorp/idea-origin.md`
   - **STAY in `docs/` (product)**: `decision-log.md`, `prd.md`, `frds/`, `blueprint.md`, `adr/`, `work-orders/`, `design/`, `reviews/`, `product-research.md` (the feature-centric migration in step 2b then reshapes these).
   - Update the project's `.gitignore`: add `.pandacorp/comms/`, `.pandacorp/inbox/`, `.pandacorp/run/`; drop the old `docs/{summary,progress,decisions,iteration,activity}.md` ignores.

2b. **Feature-centric docs migration (pre-DR-049 by-type → feature-centric).** If the project still has the old by-type `docs/` shape, reshape it into the feature-centric layout (DR-049 — see `factory/standards/structure.md`). Use `git mv` where tracked; preserve the stable IDs (`REQ-NN-MMM`, `AC-NN-MMM.K`, `WO-NN-MMM`) — numbers never change, only their location:
   - **Product layer**: `docs/prd.md` → `docs/product/prd.md`; `docs/product-research.md` → `docs/product/research.md`. Add a living **feature-landscape** table to `prd.md` if absent (one row per FRD module).
   - **FRD modules**: each flat `docs/frds/frd-NN-<slug>.md` → its own module folder `docs/frds/frd-NN-<slug>/frd.md`. Per-feature design (if the project had a separate per-feature FDD) → `frd-NN-<slug>/fdd.md`; feature mocks → `frd-NN-<slug>/mocks/`. The global design system (`docs/design/`) stays GLOBAL — never split it per feature.
   - **Blueprint split**: the global `docs/blueprint.md` is split — the **platform** parts (stack, data model, deploy, cross-cutting, service boundaries) → `docs/product/architecture.md`; each **per-feature** implementation section → that feature's `docs/frds/frd-NN-<slug>/blueprint.md` (referencing `product/architecture.md`). When the per-feature split isn't cleanly separable, keep the bulk in `architecture.md` and add thin per-feature `blueprint.md` files that point to it; don't fabricate detail.
   - **Work orders**: distribute the global `docs/work-orders/wo-NN-MMM-*.md` into their FRD's folder by their `NN` prefix → `docs/frds/frd-NN-<slug>/work-orders/`, each with a per-FRD `work-orders/README.md` (intra-feature order + parallelism). Cross-feature order continues to come from each WO's `Dependencies`.
   - **Unchanged**: `docs/design/` (the global PDD), `docs/adr/`, `docs/analytics/events.md`, `docs/decision-log.md`, `docs/reviews/` — these stay where they are.
   - Fix internal cross-links broken by the moves (PRD↔FRD, WO→FRD/blueprint references) so they still resolve.

3. **Regenerate the managed layer (non-destructive).**
   - **`.pandacorp/guide.md` and `.pandacorp/README.md`**: regenerate ENTIRELY from the current template (fully managed — no merge), substituting the template vars from `.pandacorp/status.yaml` + factory paths.
   - **Machinery** (`.claude/engines/pandacorp-build.js` and any factory script the overlay ships): regenerate from the template. **Migrate the engine's home (proposal 31 T0):** the engine moved from `.claude/workflows/` (auto-exposed in the `/` menu) to the unscanned `.claude/engines/`. After regenerating into `.claude/engines/pandacorp-build.js`, **if a leftover `pandacorp-build.js` still sits under `.claude/workflows/`, DELETE that stale copy** (and remove the now-empty `.claude/workflows/` dir) — otherwise the old menu entry keeps polluting the owner's `/` menu.
   - **`CLAUDE.md`**: ensure it imports `@AGENTS.md`, `@docs/rules/README.md` and `@.pandacorp/guide.md`. If the project still has the old fat `CLAUDE.md`, replace the managed sections with the thin import header but **keep everything the owner wrote below the managed line**. Never touch the owner's notes.
   - **`AGENTS.md`**: reconcile the managed block (priority order, the `docs/rules/` pointer, language & interaction, write-gate, stack pointer) with the template; keep project-specific additions. A project on an older overlay that still inlines the long durable-conventions block: replace that block with the `docs/rules/` pointer (the rules now live as files in `docs/rules/`, synced just below) — don't leave two copies.
   - **`docs/rules/` (engineering rules re-sync)**: this is how rule changes reach every project. Read the project's stack from `docs/product/architecture.md`. For each file in `${CLAUDE_PLUGIN_ROOT}/templates/rules/`: if its `applies_when`/`also_applies_when` is `always` or matches a technology the project uses, **copy it in verbatim, overwriting the project's copy** (the plugin is the source of truth for these files); if it no longer applies (a technology was dropped), remove the stale project copy. Leave untouched any file the owner added that has no counterpart in the plugin (project-specific rules). Then **regenerate `docs/rules/README.md`** (canonical spec in the `scaffold` skill). For a pre-rules project (no `docs/rules/`), create it now with the applicable set (first-time adoption). Bumping `OVERLAY_VERSION` whenever a rule file changes is what triggers this on the next skill run (DR-051).
   - **Gate-config conformance check (DR-059)** — catches silent enforcement drift; the gate config propagates VERBATIM, exactly like `docs/rules/`. Read the project's stack from `docs/product/architecture.md` and resolve `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/`. **Diff** the project's `verify.sh` and `canary.sh` (both in `.pandacorp/`), `biome.json` and `knip.json` (project root) against the stack templates; on **any drift** (or a missing file) **regenerate it from the template, overwriting the project's copy** (the plugin is the source of truth) and note the drift in the report. A pre-DR-079 project gets `.pandacorp/canary.sh` (the gate canary `verify.sh --canary` runs) installed here, and a pre-DR-059 project gets these three files, for the first time. A project's enforcement can never silently fall behind the standard. **Loud back-port detector (DR-094): BEFORE overwriting each diverging file, LOUDLY REPORT the diff with the DR-076 instruction** — the divergence may be an in-place fix the overwrite is about to revert, so back-port it to `plugin/templates/stack-<chosen>/` rather than fix in place. The overwrite still proceeds unconditionally (DR-059's guarantee), but a real in-place fix is now impossible to lose silently. Leave untouched any project-specific lint/knip config the owner added in a **separate** file (only the canonical ones are conformance-checked).
     - **"Project config NEWER than template" detector (DR-076 amendment, BL-0003) — BLOCKS a silent downgrade.** DR-076's "the template is never behind a project" is an ASSUMPTION, not an invariant: a project can legitimately race ahead when a pinned tool ships a config-format change before the template catches up, and a stale-template overwrite then **red-locks the baseline** under `--error-on-warnings` (BL-0003 / LESSON-0004). So BEFORE overwriting, run `${CLAUDE_PLUGIN_ROOT}/scripts/detect-gate-config-newer.sh <project_dir> <template_dir>` — it compares the **tool/schema version** in the project's config against the version pinned for the installed tool (biome's `$schema` URL semver vs `@biomejs/biome` in `package.json`) and classifies each canonical file `project-newer | template-current | unknown` (exit 1 if ANY is `project-newer`). On `project-newer` you MUST **back-port the project's version to `plugin/templates/stack-<chosen>/` FIRST**, so the subsequent overwrite is a no-op — **NEVER silently downgrade** a config that is correct for the installed tool. `template-current` (project at/behind the template) → the normal overwrite is right. `unknown` (e.g. knip's major-only schema tag) → fall back to the DR-094 loud diff report + human judgment. This is the version-compare guard the DR-094 "loud report" alone lacked (it overwrote regardless, downgrading a genuinely-ahead config).
     - **The e2e gate machinery is conformance-checked too (DR-055/056/074/075 + header-scan).**
       - **Verbatim-overwrite set — the seven VERBATIM e2e files**: `e2e/{smoke.spec.ts,visual.spec.ts,responsive.spec.ts,_responsive-helper.ts,shell.spec.ts,headers.spec.ts}` and `playwright.config.ts` (project root). Diff each against the stack template and regenerate on drift, exactly like `verify.sh`. First-time installs: a pre-DR-074 project gets `responsive.spec.ts`/`_responsive-helper.ts`, a pre-DR-075 project gets `shell.spec.ts`, a pre-header-scan project gets `headers.spec.ts` (advisory until `deploy_target: external`, so it can't red-lock a retrofit), plus any of the others that are missing.
       - **Per-project never-diff set — `e2e/routes.ts`, `e2e/_target.ts`, `e2e/shell.ts`**: **NEVER byte-diff** these — they are PER-PROJECT (the project's own surface list / platform read / app-shell nav contract); **create them from the template only if missing, never overwrite** (overwriting `routes.ts` would clobber the project's blessed surface manifest, and overwriting `shell.ts` its nav contract = data loss).
       - **Back-fill exception 1 — `e2e/shell.ts` back-filled EMPTY (DR-075), never a filled contract:** a pre-DR-075 project gets the template's empty-`NAV_DESTINATIONS` seed (a vacuous-pass gate), the conservative analogue of the `target_platforms: desktop` back-fill below — filling it with the prototype's nav on an app that has no shell would turn the new gate RED and **red-lock all commits**. The real contract is filled deliberately, in the SAME change that builds the shell (the re-anchor), never by the upgrade.
       - **Back-fill exception 2 — `e2e/_target.ts` additive append:** the file only READS `status.yaml` (it carries no per-project data), so if the project's copy lacks the `readDeployTarget`/`DEPLOY_TARGET` export (pre-header-scan), APPEND that block from the template — additive, nothing project-specific to lose; never rewrite the rest of the file.
   - **Toolchain conformance check (DR-059, extends the gate-config check)** — a byte-perfect `verify.sh` still goes RED at cold start if the tools it calls aren't installed: a missing `knip` or `playwright` fails the gate even with a canonical config. Conformance therefore covers the **toolchain**, not just the config files — this is what makes the fail-closed gate actually *runnable*. After the config diff, read the project's `package.json` and reconcile it against what the canonical `verify.sh` requires (resolve from the same `stack-<chosen>/` template); on any gap, install/fix it and note it in the report:
     - **Required devDependencies** — `knip`, `madge`, `@playwright/test`, `vitest`, `@biomejs/biome`, `@axe-core/playwright` (the responsive gate's tap-target check, DR-074). Any missing from `devDependencies` → install (`@latest` for a current project; framework-major-compatible for brownfield, DR-052).
     - **Required `package.json` scripts** the canonical `verify.sh` invokes — `test:smoke` (`playwright test e2e/smoke.spec.ts`, DR-055), `test:visual` (`playwright test e2e/visual.spec.ts`, DR-056), `test:responsive` (`playwright test e2e/responsive.spec.ts`, DR-074) and `test:shell` (`playwright test e2e/shell.spec.ts`, DR-075). Add any that are absent.
     - **Strict `tsconfig`** — ensure `compilerOptions.strict: true` **and** `noUncheckedIndexedAccess: true`; set whichever is missing or `false`.
     Gated by the same `OVERLAY_VERSION` bump (DR-051) and reported alongside the config drift.
   - **`status.yaml`**: set `overlay_version` to the target. For `created_with` (immutable — the version the project was BORN on): keep it if present; for a project migrated from a pre-6.0.0 layout, set it to `pre-6.0.0` (it was not born on the current overlay). For `created_via` (immutable provenance, DR-077 amendment/BL-0009): keep it if present; **NEVER back-fill it to `scaffold`** on an existing project — a pre-existing project's doc spine was never proven complete under the fail-closed rule, so promoting it would surprise-red-lock the gate on the next `verify.sh`. Leave it absent (doc-lint.sh treats a missing/unrecognized value as advisory, same as `adopt`) unless the project record already says how it was born.
   - **Back-fill `target_platforms` for an existing project to `desktop` (DR-074)** — if the field is absent, add `target_platforms: desktop`, NOT `responsive`. This is the conservative, currently-passing default: an existing project may be non-responsive, so back-filling `responsive` would turn the new mobile responsive gate RED immediately and **block all commits** (the red-lock). `desktop` keeps the gate green (the responsive checks become a vacuous pass); promoting to `responsive` is a deliberate, SEPARATE change once the project is verified responsive at mobile width. A NEW project, by contrast, is born `responsive` (the status.yaml template default) — only the retrofit is conservative.

3b. **Post-conformance PROJECT gate run (DR-076 amendment, BL-0003) — fail loud, never leave a broken baseline.** After the conformance overwrite has re-synced the gate config (step 3) AND the `--canary` proved the gates still bite (see the DR-079 rule below), run the FULL project gate ON THE PROJECT: `.pandacorp/verify.sh` (a full baseline run, NOT `--since` — the config the overwrite just touched is untracked-by-diff at the FRD level, exactly the blind spot that hid the personal-page-v2 red-lock; the per-FRD `--since` gate lints only changed files, so a red-locking `biome.json` slips through until pass-2). If `verify.sh` is not runnable standalone, at minimum run `biome check . --error-on-warnings` + `knip` + `tsc --noEmit`. **If the freshly-synced config RED-LOCKS the gate, STOP — this is BLOCKING for the upgrade:** do NOT stamp `overlay_version`, do NOT commit. Report the red loudly and either (a) back-port the project's newer config to the template (the "project newer than template" case the detector in step 3 should already have caught — the conformance overwrite downgraded a correct config), or (b) fix the template if the template itself is genuinely broken; then re-run conformance + this gate until green. The DR-079 canary proves the gates FUNCTIONALLY reject a broken fixture; it does NOT prove the PROJECT still passes after the overwrite — this run is the missing check that would have caught the silent downgrade before the next build discovered a dead gate (`builtFrds:[]`, `blockedReasons:{baseline:error}`). Skip only when the project has no `verify.sh` at all (a pre-DR-059 project that just had it installed for the first time still runs it here — the install is the point).

4. **Bump + record.** Set `overlay_version` to the target in `.pandacorp/status.yaml`. Add a `docs/decision-log.md` entry: "Pandacorp overlay upgraded `<old>` → `<new>`"; if step 2b ran, note the feature-centric docs migration (DR-049) and that it was a pure relocation (IDs preserved, no behavior changed); if the gate-config conformance check (DR-059) regenerated `verify.sh`/`biome.json`/`knip.json`, note which drifted and that enforcement was re-aligned to the standard; if the toolchain check installed missing devDependencies/scripts or fixed `tsconfig`, note that too (the gate is now runnable cold).

5. **Commit on its own**: `chore: upgrade pandacorp overlay <old> → <new>` (moved files + machinery + managed layer + any DR-049 docs relocation from step 2b). Do NOT mix product *behavior* changes into this commit — the docs migration is a structural move, not a content rewrite.

6. **Report** (in Spanish): what moved, what was regenerated, and the new version. If a MAJOR jump required a manual decision, surface it explicitly.

## Rules
- **Compatible (same MAJOR) = silent** (DR-048): apply and commit without interrupting; the owner sees it via the commit. **MAJOR = announce** the breaking parts + diff before applying.
- **Never clobber the owner's content**: managed files (`guide.md`, `README.md`, machinery) regenerate whole; `CLAUDE.md`/`AGENTS.md` merge by managed-section, the owner's notes stay.
- Touch only the overlay/integration layer — never product **code**, and never rewrite the **content** of product docs. The one exception is the DR-049 **feature-centric docs migration** (step 2b): a one-time structural *relocation* of `docs/` files (moves + link fixes, IDs preserved), never a content rewrite. A MAJOR overlay jump that carries this migration is announced first like any breaking change.
- **Gate config is conformance-checked, not merged (DR-059)** — canonical files (`verify.sh`/`biome.json`/`knip.json` + the seven VERBATIM e2e files) are overwritten from the template on any drift; `e2e/routes.ts`/`_target.ts`/`shell.ts` are PER-PROJECT and NEVER byte-diffed. See step 3 for the full set, the DR-076 downgrade detector, and the back-fill exceptions.
- **Canonical gate files are edited at the SOURCE, never in-place (DR-076)**: because conformance overwrites them, a fix made directly in a project's `biome.json`/`knip.json`/`.pandacorp/verify.sh`/the six VERBATIM e2e files/`playwright.config.ts` is **silently reverted on the next `/upgrade`** AND leaves the template stale for every other project. So a genuine fix MUST be made in `plugin/templates/stack-<chosen>/` and propagated via `/upgrade` (back-port in the SAME change); a genuinely project-specific exception goes in a SEPARATE never-overwritten file. The `remind-backport-gate-config.sh` hook nudges this at edit time. Conformance therefore assumes the template is never behind a project, and the overwrite stays **unconditional** (DR-059's binary guarantee, NOT a "diff-and-flag" that would re-admit silent drift). The DR-076 amendment (BL-0003) handles the case where that assumption breaks — a project legitimately ahead of the template — via the `detect-gate-config-newer.sh` back-port-FIRST guard; see step 3.
- **Conformance covers the toolchain, not just the config (DR-059)** — a byte-perfect `verify.sh` is useless if its tools aren't installed, so the check also reconciles the required devDependencies, scripts and a strict `tsconfig`, making the fail-closed gate runnable at cold start. See step 3 for the exact set.
- **Prove the gates still BITE after re-syncing them (DR-079)**: once the gate-config conformance has regenerated `verify.sh` + `canary.sh` + the e2e files, run `.pandacorp/verify.sh --canary` — `canary.sh` GENERATES a deliberately-broken input per gate (no broken files kept in the repo), pushes each through its gate, and REDs if any gate that should reject its broken input stays green (a rotted gate: a renamed spec, a disabled rule, a swallowed exit code). It canaries EVERY fail-closed gate: biome (`--error-on-warnings`), the test-placement structure guard, the DR-100 readiness gate, tsc (strict types, via an isolated temp tsconfig), madge (circular deps), vitest (a failing test), knip (an orphan file), and Playwright spec DISCOVERY (proves the config still finds smoke/visual/responsive/shell so the e2e layer can't pass vacuously — a fast `--list`, no browser run). NOT canaried, by design: a full Playwright browser run (discovery + the fail-closed missing-harness checks + conformance cover its rot modes) and `doc-lint.sh` (advisory, never reds). **A canary failure after the re-sync is BLOCKING for the upgrade**: a rotted gate means the project's enforcement is fiction — fix the gate (or back-port the genuine fix, DR-076) and re-run before stamping `overlay_version`; never stamp an upgrade "done" over a red canary. (A project with no `canary.sh` can no longer occur post-conformance — the same pass installs it; if the install itself failed, that IS the blocking finding.) Conformance proves the gate FILES match the template (DR-059/076); the canary proves they still FUNCTIONALLY reject.
- **Prove the PROJECT still passes after re-syncing (DR-076 amendment, BL-0003) — separate from the canary.** The canary proves the gates bite on a broken fixture; it does NOT prove the freshly-overwritten config lets the REAL project through. So after conformance + canary, run the FULL project gate on the project (`.pandacorp/verify.sh`, a full baseline — never `--since`). **A red baseline here is BLOCKING for the upgrade:** never stamp `overlay_version` or commit over a config the overwrite just red-locked — back-port the project's newer config (or fix the template) and re-run until green. See step 3b for the full-baseline rationale and the fallback command.
- Idempotent: re-running when already in sync is a no-op.
