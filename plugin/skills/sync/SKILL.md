# /pandacorp:sync

**The reverse flow: "I changed the code by hand — now catch the docs up, completely."** The whole pipeline runs `doc → code` (`spec → design → architecture → implement`, and day-to-day `change`/`iterate`). `sync` is the inverse: you edited an app directly — to go fast, to see it change live — and the code outran the docs. `sync` propagates that reality **upstream** to the owning canonical docs **and** the area's decision log. It is `iterate` run backwards, and it is **exhaustive** — the owner's standard is *lots of well-made documentation over too little; nothing slips through*.

**The inverse flow inverts the oracle (read this — it's the whole design).** Forward, the **doc is the oracle**: code is validated against the FRD. Here the **code becomes the oracle** and the doc conforms to it (constitution §24). So `sync` **cannot verify a change is correct — only describe what the code DOES.** Hence: (1) an **intent gate** — only you know whether what's in the code was *deliberate* (document it) or *a half-finished experiment / accident* (don't bless it); (2) **provenance** — every doc it writes is marked *reconciled-from-code* (like `adopt`'s as-built), never dressed up as authored design intent.

> **Preflight (DR-045) — modo proyecto.** This skill writes to the project's docs, so first confirm the marker `.pandacorp/status.yaml` exists. If it's missing **but** you're at the factory root with hand-edits under `plugin/` or `factory/`, this is *modo fábrica* — out of v1 scope: route the owner (in Spanish) to `/pandacorp:learn` (durable know-how, which already versions the plugin) and stop. If it's missing entirely, this folder isn't a factory project — `/pandacorp:adopt` brings one in, `/pandacorp:spec` creates a new one. Then, if `overlay_version` is behind the plugin's `OVERLAY_VERSION`, run `/pandacorp:upgrade` first (silent for compatible bumps, DR-048).

`$ARGUMENTS` (or the conversation): optionally what you changed by hand (e.g. `/pandacorp:sync "puse el botón de descartar en rojo y rehíce el header del card-detail"`); empty (`/pandacorp:sync`) lets it find the changes itself — including, with no context at all, a full audit.

## Two modes — pick by how much context you have

- **Warm** — the changes are already in this conversation. Start from them, **but still verify against the real code** (don't trust the chat's memory of what changed; read the diff).
- **Cold** — no context (a fresh conversation, or hand-edits made long ago and never documented). Run a **FULL AUDIT**: enumerate every surface of the app from the code — routes, components, the data model, APIs, business logic — and compare each to its owning doc; the "change set" is the set of **code↔doc divergences (gaps)**. For a large app, **fan out** (a reader per area: features/UI / data+architecture / build state) and consolidate. The bar is **find every gap — nothing passes unnoticed.**

## Steps

1. **Detect context, pick the mode** (warm vs cold, above). You're in *modo proyecto* — Mission Control or a product app.
2. **Build the change set.** *Warm:* the uncommitted working tree (`git status`/`git diff`) + commits since the last one that touched `docs/decision-log.md` (or since `last_green_sha`) + what's in context. *Cold:* the audit's gap list. Either way **read the real code**, not just file names — you need *what behaviour / look / architecture / scope changed*, not which lines moved.
3. **Classify each divergence — DIRECTION FIRST, then the cascade (the key step).** Decide which side is right *before* deciding what to write, because only one direction gets documented:
   - **Code has MORE than the doc** (behaviour the doc doesn't describe) → the code outran the doc → **document it** (the full cascade below).
   - **You changed the behaviour ON PURPOSE**, the old doc no longer applies → the intent moved → **update the doc** (you confirm this at the intent gate, step 5).
   - **The doc has MORE than the code** (the doc specifies something the code doesn't do) → **do NOT rewrite the doc down to the broken/absent code** — that would bless a bug or erase wanted work. **Route it, don't document it:** a **bug** (code contradicts a doc that's right) → file via `/pandacorp:change`/`bug` to fix the CODE, the doc untouched; **documented-but-unbuilt** (a wanted feature never implemented "por diferentes motivos") → mark it **pending** (the PRD feature-landscape / the work order's `implementation_status`), the spec untouched.

   Then, for what IS to be documented (the code outran the doc, or a deliberate change), expand its **full CASCADE** — don't stop at the top doc:
   - **behaviour / feature** → the owning **FRD** (`docs/frds/frd-NN-<slug>/frd.md`) **and almost always its work order(s)** (`work-orders/wo-*.md` — a behaviour change not reflected in the WO leaves the build's own record false) **and its `fdd.md`** if the UI moved. Behaviour with no FRD → a **new FRD module** (frd + blueprint + work-orders, + fdd/mocks if UI; numbering continues).
   - **architecture / data model / stack / a tech choice** → the feature **`blueprint.md`** **+** an **ADR** (`docs/adr/`) **+** `docs/product/architecture.md` **+** the affected **work orders**.
   - **UI / interface / interaction** → the feature **`fdd.md`** **+**, if it touched the shared system, **`DESIGN.md`** / `docs/design/design-tokens.json` / `docs/design/components.md`.
   - **product scope / goal / metric** → the **PRD** (`docs/product/prd.md`, the feature-landscape).
   The source-of-truth hierarchy (FRD > fdd > tokens > blueprint > work order) decides who *wins* a conflict; the **cascade** decides what you *touch* — update the owning doc **and** every derived doc the change made obsolete.
4. **Sweep BOTH trees — `docs/` and `.pandacorp/`.** Beyond `docs/`, refresh the `.pandacorp/` text layer the change affects: `comms/summary.md` (the Spanish project narrative), a line in `comms/progress.md`, `comms/iteration.md` if a manual phase was in flight, and the `status.yaml` counters. Neither tree is left stale.
5. **Present the sync plan → INTENT GATE (the owner judges intent, not correctness).** Show, in Spanish, a tight "this changed → I'll do X" list (for a cold audit with many gaps, group them by doc). The owner sorts each divergence into ONE action: **document** (real behaviour the code outran the doc on) · **update the doc** (a deliberate behaviour change — the old spec no longer applies) · **it's a bug** (the doc is right, the code is wrong → file via `/pandacorp:change`/`bug`, the doc stays) · **pending** (documented but never built → mark not-built, the spec stays) · **experiment / accident** (don't bless — leave it, or flag to undo). **This gate is the independent oracle the skill lacks** — and the only place a bug can be told apart from an on-purpose change, which the code alone can't reveal. Confirm before writing.
6. **Write the whole cascade — marked reconciled-from-code.** For each blessed change, write **every** doc in its cascade (step 3) across both trees (step 4). Header reconstructions *"Reconciled from code by `/pandacorp:sync` on {{DATE}} — describes existing behaviour, not a forward plan."* Re-use the doc templates (DR-077) for shape. Product docs in **English** (committed); the `.pandacorp/comms/` layer in **Spanish** (gitignored) — the language rule (DR-009). Don't invent rationale; a stub with a TODO is more honest than a confident fiction.
7. **External design source — Claude Design (DR-058).** A design system can have an external mirror: if the project's design came FROM / was pushed TO a Claude Design project via `/design-sync`, `docs/design/` carries a `_ds_manifest.json` (or a recorded projectId). If **this** sync changed the shared system (tokens, a shared component, the inventory) **and** that provenance exists, the Claude Design project is now **stale** → handle it like the Manual: **detect → flag → OFFER, never auto-push.** Tell the owner (Spanish) it diverged and offer to re-sync via `/design-sync` (incremental, one component at a time). **Don't push automatically** — it's an outward-facing write to an external beta service that needs the owner's claude.ai login (a human gate); the repo's `docs/design/` stays the source of truth (DR-054). No manifest ⇒ nothing to sync; skip silently.
8. **Decision log, lint, commit, report.** Record the two-layer entry in the area's `docs/decision-log.md` (date · **What** · **Why** · **Impact** = which docs you touched; most recent on top) — **the diff has the *what*, not the *why*: ASK the owner the why.** Refresh `status.yaml` carefully (reset `pending_changes`, bump `updated_at`; read, don't invent). Run `.pandacorp/doc-lint.sh -v` as a safety net. Capture any durable lesson (DR-047) / owner fact (DR-053) that surfaced. Commit in English, conventional, touching the canonical docs **and** the decision-log entry together. Report in Spanish: everything propagated, where, and any gap left open.

## Rules

- **It documents, it does NOT verify (the inverted oracle).** `sync` describes what the code DOES; it can't tell you whether that's *right*. Correctness is judged by you at the intent gate (step 5). Never present observed behaviour as authored, blessed design — mark it reconciled-from-code, exactly as `adopt` marks as-built.
- **It documents what the code does in EXCESS — it never trims what the doc asks in excess.** The code is the oracle of what the app DOES; the doc stays the oracle of what it SHOULD do. A divergence where the **doc is right** is not closed by editing the doc down: a **bug** (code contradicts a correct doc) is routed to `/pandacorp:change`/`bug` to fix the code, a **documented-but-unbuilt** feature is marked pending (build it forward) — the FRD/PRD spec is **never degraded to describe broken or absent code**. Only a divergence you confirm as a *deliberate* behaviour change rewrites the spec. (`sync` documents code-ahead-of-doc; `change`/`bug` fixes code-behind-doc — complementary, and `sync` detects-and-routes the latter.)
- **Exhaustive within the diff, bounded to it (DR-012).** Propagate each change through its **whole cascade** — a behaviour change that updates the FRD but leaves the work order or the FDD stale is a half-done sync; *nothing slips through*. The owner's standard is **plenty of well-made documentation over too little** — but "plenty" means *the complete cascade*, not padding: don't rewrite docs the change never touched, and don't inflate small features into noise.
- **Two layers, always (the `CLAUDE.md` rule).** Every blessed change updates the **owning canonical doc** *and* records the **area's decision-log entry**, in the same commit, before the turn closes. Never one without the other.
- **The "Why" isn't in the code — ask for it.** A diff is a record of *what* changed, never *why*. Pull the reason from the owner; don't fabricate a rationale to fill the field.
- **Both trees, each in its language (DR-009).** Sweep `docs/` **and** `.pandacorp/`: product docs in English (committed), the `.pandacorp/comms/` narrative in Spanish (gitignored). Don't leave one tree current and the other stale.
- **External mirrors: detect + offer, never auto-push.** Claude Design (or any outward-facing design mirror) is refreshed only through `/design-sync` behind the owner's gate — `sync` flags the divergence, it doesn't publish.
- **Modo proyecto only (v1).** Hand-edits to the plugin/factory go to `/pandacorp:learn` (it owns plugin versioning + `templates/rules` + `OVERLAY_VERSION`). Don't duplicate that here.
- **`status.yaml` is read, not hand-authored (DR-050).** Refresh its counters/timestamps; never write product/phase state into it by hand (skills/CI own it).
- **Conventional commits in English, direct to main is fine, never force-push.**
