# /pandacorp:blueprint

Architecture phase. Runs IN the project (requires PRD, FRDs and frozen design). Produces **two artifacts**: the technical blueprint and the work orders ready to build.

## Steps

1. **Stack proposal + approval** (`architect` agent): starts from the recommended stack (`factory/standards/stack.md`, **latest stable versions**), evaluates whether there are better technologies for THIS project and proposes them with trade-offs. **Presents the proposal to the owner and waits for approval** (light gate, DR-002) before locking it in. The choice is recorded as an ADR. Durable conventions (`factory/standards/`) are not up for debate.
2. **Architecture — two layers, never fused (DR-049)** (`architect` agent):
   - **Platform architecture** (`docs/product/architecture.md`, stable, ONE per project): approved stack, high-level architecture, complete data model, deploy strategy, cross-cutting concerns and service boundaries — the things that are stable across features.
   - **Per-FRD blueprint** (`docs/frds/frd-NN-<slug>/blueprint.md`, one per FRD in scope): how THIS feature is implemented on top of the platform — its components `CMP-NN-<slug>` and interfaces `IF-NN-<slug>`, API/action contracts and external integrations specific to the feature (limits and costs), each traced back to the FRD's `REQ-NN-MMM`. Each blueprint restates the **source-of-truth hierarchy** in its header (`FRD > FDD > design-tokens > blueprint > work order`) and references `docs/product/architecture.md` instead of re-describing the platform. Large feature → promote `blueprint.md` to a `blueprints/` folder split by coherent concern (NOT "pillars"). A single global `docs/blueprint.md` is an anti-pattern — don't produce one.
   - ADRs in `docs/adr/` for each non-obvious platform-level decision. **For monetary/mixed (DR-042)**: compute the **per-active-user variable cost** (Polar fee ~4%+$0.40, Vercel Pro share, Neon/R2/Resend/PostHog usage tiers) and hand it to the PRD's unit-economics block so the break-even is real, not a guess.
3. **Event plan** (`analytics` agent): translates the PRD's success metrics and value hypothesis into `docs/analytics/events.md` (which event measures each metric, where it fires, what properties — no PII, DR-025). This is what makes v1 ship instrumented rather than "flying blind". It is instrumented in `implement` and verified in `release`.
4. **Deploy design** (`devops` agent, only if the project actually deploys): deployment section/ADR — where to, environments (dev → staging → prod), reproducible CI/CD pipeline, secrets management (SOPS+age store, `factory/standards/external-services.md`) and rollback plan, starting from `factory/standards/infra.md`. **If v1 includes payments (PRD, DR-035) and the host is Vercel, WARN the owner** (warning, not a blocker): charging requires Vercel Pro (Hobby is non-commercial, full account ban) → it triggers a push (DR-038) and the owner decides. For a trivial project with no deploy of its own, the standard is enough and this step is skipped.
5. **Cross-validation + feasibility + FRD reconciliation**: each FRD must map to components (`CMP-NN-<slug>`) of its blueprint. **Read `docs/design/technical-assumptions.md`** (the designer's feasibility flags): for each costly/risky assumption (real-time, offline, server-side media, costly/uncertain third-party, scraping with ToS/rate-limit risk) confirm the architecture can support it cheaply; if a critical technical unknown remains, run a **timeboxed feasibility spike to prove it BEFORE committing the data model and work orders** (the order fix — don't discover infeasibility mid-build). **If the cross-validation or a spike changes an FRD's criteria, the `product-manager` updates the affected FRD(s) (+ADR) before work orders are generated** — work orders must never carry stale criteria (anti spec-drift). If something doesn't add up, go back to the `architect` before continuing.
6. **Install the stack**: follow the guide `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/STACK.md` and the standards (`factory/standards/`): strict typing, linter+formatter, test framework, folder structure. Verify that lint + typecheck + test run clean.
7. **Create `.pandacorp/verify.sh`** in the project (used by the Stop hook): runs lint + typecheck + tests fast and exits ≠0 if anything fails. **Hardened (DR-019)**: fail-closed (ambiguous output = failure), clean environment, does not expose the test/fixture names to the generating agent, and **actionable messages** (which rule failed and why). The stack guide provides the base content.
8. **CI**: GitHub Actions with lint + typecheck + tests on PR (template in the stack guide). Toward main: **mutation testing** (Stryker/mutmut, DR-016) + **OWASP agentic audit** if applicable (DR-017) + **changelog/ADR** generation (DR-018). Branch protection on main if the GitHub repo exists. The deploy pipeline was left designed by the `devops` (step 4).
9. **Work orders** (same step, second artifact): generates **per-FRD** work orders at `docs/frds/frd-NN-<slug>/work-orders/wo-NN-MMM-<slug>.md` from each FRD + its blueprint (see the detail in the `work-orders` skill): implementable tasks, **small and testable in isolation**, each targeting exactly ONE deploy unit, citing its `REQ-NN-MMM`/`AC-NN-MMM.K`/`CMP-NN-*`/`IF-NN-*` with the AC copied in, ordered by `Dependencies` (intra- and cross-feature), with a definition of done. Per-FRD list in `docs/frds/frd-NN-<slug>/work-orders/README.md` with the intra-feature order and what can be parallelized; cross-feature order comes from each WO's `Dependencies` (the build walks every FRD's `work-orders/`).
10. **Iterate / advance gate (DR-032)**: if a blueprint/work orders already existed, this is an **iteration** — read `.pandacorp/comms/iteration.md` (phase `architecture`) and refine what's there (don't regenerate or repeat what was discarded), appending one entry per round. Commit the blueprint and present it to the owner: chosen stack and why, estimated cost, risks, and how many work orders. **Don't start the build on your own:** leave `advance_pending: true` in `.pandacorp/status.yaml` and **wait for their "ok, let's build"** — re-running `blueprint` in the meantime = keep polishing the plan. When they approve: `.pandacorp/status.yaml` → `phase: architecture` (ready to build) + `advance_pending: false`, and close the `architecture` thread in `.pandacorp/comms/iteration.md`. Next step: `/pandacorp:implement` (starts the build with a dynamic workflow that orchestrates the subagents, followed in Mission Control).

## Rules
- The TECHNICAL research happens HERE (stack, libraries, integrations, limits): leave little to resolve during the build. The dev agents only research specific gaps on demand.
- The MINIMUM that satisfies the FRDs: no microservices, no speculative infrastructure features, ~$0 cost at launch.
- If an external integration requires payment (paid API, plan), DR-005 applies: escalate to the owner with the amount.
