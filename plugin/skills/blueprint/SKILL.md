# /pandacorp:blueprint

Architecture phase. Runs IN the project (requires PRD, FRDs and frozen design). Produces **two artifacts**: the technical blueprint and the work orders ready to build.

> **Preflight (DR-045) — is this a Pandacorp project?** This skill mutates the project, so first confirm the Pandacorp marker: `.pandacorp/status.yaml` exists. If it's missing, STOP and tell the owner (in Spanish) that this folder isn't a factory project yet — `/pandacorp:adopt` brings an existing project in, `/pandacorp:spec` creates a new one. Then, if `overlay_version` in `.pandacorp/status.yaml` is behind the plugin's `OVERLAY_VERSION`, run `/pandacorp:upgrade` first (silent for compatible bumps, DR-048) so this skill runs against the current structure. Don't proceed or invent docs over a missing structure.

## Steps

1. **Stack proposal + approval** (`architect` agent): starts from the recommended stack (`factory/standards/stack.md`, **latest stable versions**), evaluates whether there are better technologies for THIS project and proposes them with trade-offs. **Presents the proposal to the owner and waits for approval** (light gate, DR-002) before locking it in. The choice is recorded as an ADR. Durable conventions (`factory/standards/`) are not up for debate.
2. **Architecture — two layers, never fused (DR-049)** (`architect` agent):
   - **Platform architecture** (`docs/product/architecture.md`, stable, ONE per project): approved stack, high-level architecture, complete data model, deploy strategy, cross-cutting concerns and service boundaries — the things that are stable across features.
   - **Per-FRD blueprint** (`docs/frds/frd-NN-<slug>/blueprint.md`, one per FRD in scope): how THIS feature is implemented on top of the platform — its components `CMP-NN-<slug>` and interfaces `IF-NN-<slug>`, API/action contracts and external integrations specific to the feature (limits and costs), each traced back to the FRD's `REQ-NN-MMM`. Each blueprint follows `${CLAUDE_PLUGIN_ROOT}/templates/docs/blueprint-template.md`: it restates the **source-of-truth hierarchy** in its header, references `docs/product/architecture.md` instead of re-describing the platform, carries the **state frontmatter** (`status` + `implementation_status`), and MUST include a **Build Plan** — the DAG of that FRD's work orders (which is first, which depend on which, what runs in parallel, the integration order, and any cross-FRD dependencies). **The build engine READS this Build Plan instead of re-inferring dependencies at runtime (DR-050)** — this is what makes the build fast and the integration correct. The Build Plan also **records each WO's `artifacts` (the globs of files/dirs it writes) and must be verifiably DISJOINT within a wave (DR-060)**: design wave-parallel WOs so their `artifacts` don't overlap — the engine enforces this from the declared artifacts (it serializes any overlap into a later wave), but the plan should be disjoint by design. The API contract is **per-WO at `docs/api/<wo-id>.md`** (in that backend WO's artifacts), never a single shared file.
     - **Foundation wave first (DR-057).** Before the feature work orders fan out, the Build Plan MUST identify the **shared design-system foundation** — the core primitives that multiple features reuse (`Button`, `Banner`/`Alert`, `Card`, `Chip`, `Modal`, `Input`, `Table`, …) — and schedule them as the **FIRST work order(s)**, a *foundation wave* that completes BEFORE any feature work order starts. That wave produces the primitives in `components/core`/`components/modules` **and seeds the living component inventory `docs/design/components.md`** (name · purpose · path · key props/variants). Feature work orders then declare a **`Dependency` on the foundation WO** and build against real, existing primitives — never reinvent them. This is what stops parallel feature agents (who never talk to each other) from each building their own near-duplicate banner/card/modal; without it you get the exact MC defect that motivated DR-057 (two warn banners, two layouts, no shared `Banner`). Large feature → promote `blueprint.md` to a `blueprints/` folder split by coherent concern. A single global `docs/blueprint.md` is an anti-pattern — don't produce one.
   - ADRs in `docs/adr/` for each non-obvious platform-level decision. **For monetary/mixed (DR-042)**: compute the **per-active-user variable cost** (Polar fee ~4%+$0.40, Vercel Pro share, Neon/R2/Resend/PostHog usage tiers) and hand it to the PRD's unit-economics block so the break-even is real, not a guess.
3. **Event plan** (`analytics` agent): translates the PRD's success metrics and value hypothesis into `docs/analytics/events.md` (which event measures each metric, where it fires, what properties — no PII, DR-025). This is what makes v1 ship instrumented rather than "flying blind". It is instrumented in `implement` and verified in `release`.
4. **Deploy design** (`devops` agent, only if the project actually deploys): deployment section/ADR — where to, environments (dev → staging → prod), reproducible CI/CD pipeline, secrets management (SOPS+age store, `factory/standards/external-services.md`) and rollback plan, starting from `factory/standards/infra.md`. **If v1 includes payments (PRD, DR-035) and the host is Vercel, WARN the owner** (warning, not a blocker): charging requires Vercel Pro (Hobby is non-commercial, full account ban) → it triggers a push (DR-038) and the owner decides. For a trivial project with no deploy of its own, the standard is enough and this step is skipped.
5. **Cross-validation + feasibility + FRD reconciliation**: each FRD must map to components (`CMP-NN-<slug>`) of its blueprint. **Read `docs/design/technical-assumptions.md`** (the designer's feasibility flags): for each costly/risky assumption (real-time, offline, server-side media, costly/uncertain third-party, scraping with ToS/rate-limit risk) confirm the architecture can support it cheaply; if a critical technical unknown remains, run a **timeboxed feasibility spike to prove it BEFORE committing the data model and work orders** (the order fix — don't discover infeasibility mid-build). **If the cross-validation or a spike changes an FRD's criteria, the `product-manager` updates the affected FRD(s) (+ADR) before work orders are generated** — work orders must never carry stale criteria (anti spec-drift). If something doesn't add up, go back to the `architect` before continuing.
6. **Install the stack**: follow the guide `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/STACK.md` and the standards (`factory/standards/`): strict typing, linter+formatter, test framework, folder structure. Verify that lint + typecheck + test run clean. **Install the canonical gate config VERBATIM (DR-059)** — the same way `docs/rules/` are copied: copy `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/biome.json` and `knip.json` to the project **root**, and `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/verify.sh` to `.pandacorp/verify.sh`, **byte-for-byte, never hand-rolled or partially re-tuned**. These carry the MAXIMUM fail-closed config (the `react`/`next`/`test` domains, the `a11y` group, the hard-enforcement rules promoted to **error**, `noExcessiveCognitiveComplexity: "error"`, and the dead-code/circular/smoke/visual gates) — the meta-fix for the audit where the prose standards never got wired into a real project. The STACK.md guide's **hard-enforcement** section and **Library conventions** are the *rationale* for that config; project-specific additions go in separate files, never by editing the canonical ones (so `/pandacorp:upgrade`'s conformance check stays clean). **Wire the dev ports** from the project's `.pandacorp/status.yaml` `dev_port_base` (reserved at scaffold from the factory ledger; port convention in `factory/standards/infra.md`, DR-022): write `.claude/launch.json` with the app on `base+0` (`runtimeExecutable`/`runtimeArgs` = the stack's dev command, e.g. `pnpm` `["run","dev"]`, plus the matching `--port`/`-p base` flag and `"port": base`) so the dev/preview server starts straight from Claude Code, and set the app/DB/service ports in `.env` per the block offsets (Postgres `+2`, Redis `+4`). If `dev_port_base` is `0` (e.g. a project scaffolded before this convention), allocate a block now from `factory/ports.yaml` exactly as `scaffold` does, then materialize.

6b. **Inject the tech-gated engineering rules.** Now that the stack is locked, copy from `${CLAUDE_PLUGIN_ROOT}/templates/rules/` every rule file whose `applies_when` (or `also_applies_when`) matches a technology this project uses — e.g. `react.md` (React), `nextjs.md` (Next.js App Router), `styling-and-ui.md` (Tailwind), `web-performance.md`/`web-security.md` (web UI / public web), `prisma.md` (Prisma), `i18n-next-intl.md` (next-intl), `analytics-and-errors.md` (PostHog/Sentry) — into the project's `docs/rules/` (the `always` files are already there from scaffold). A technology the project does NOT use → its file is NOT copied. Then **regenerate `docs/rules/README.md`** (canonical spec in the `scaffold` skill: directive header + one `@import` per file present + table). Copy verbatim — never re-condense.
7. **`.pandacorp/verify.sh`** (used by the Stop hook): this is the canonical `verify.sh` installed VERBATIM in step 6 (DR-059) — do NOT author a fresh one. It is MAXIMUM fail-closed (DR-019): `set -euo pipefail` + `inherit_errexit`, every Biome warn promoted to a hard gate (`--error-on-warnings`), `tsc`/`knip`/`madge`/`vitest`, and the smoke (DR-055) + visual (DR-056) shims where a **missing** harness is RED, never a skip — with actionable messages (which rule failed and why). Confirm it runs; never hand-edit it (drift fails the conformance check).
8. **CI**: GitHub Actions with lint + typecheck + tests on PR (template in the stack guide). Toward main: **mutation testing** (Stryker/mutmut, DR-016) + **OWASP agentic audit** if applicable (DR-017) + **changelog/ADR** generation (DR-018). Branch protection on main if the GitHub repo exists. The deploy pipeline was left designed by the `devops` (step 4).
   - **Visual spec for UI features (DR-054).** For a UI FRD (`ui: true`), the Build Plan must point implementers at the FRD's `fdd.md` + the specific `mocks/` screen(s) as the **binding visual spec to reproduce** (layout, structure, components, density — not an approximation; "uses tokens" is not enough), and **enumerate the assets the screens need** (fonts/images/sprites/icons) so they get staged (e.g. into `public/`) and actually render. The fidelity is checked by the preview smoke gate (DR-055).
9. **Work orders** (same step, second artifact): generates **coarse per-FRD** work orders (see the `work-orders` skill) following `${CLAUDE_PLUGIN_ROOT}/templates/docs/work-order-template.md`. Each is **one cohesive view / page / capability** (NOT an atomic component), carries the **state frontmatter** (`implementation_status: PLANNED`, and `artifacts: [globs]` — the files/dirs it writes; the engine serializes WOs whose artifacts overlap, DR-060) and a `## Status Note` hand-off, cites its `REQ-NN-MMM`/`AC-NN-MMM.K`/`CMP-NN-*`/`IF-NN-*` with the AC copied in, and lists its `Dependencies` consistent with the blueprint's **Build Plan**. A UI work order also carries its `## Visual reference` (the `mocks/` it reproduces + the fidelity AC, DR-054). Per-FRD list in `docs/frds/frd-NN-<slug>/work-orders/README.md`.
10. **Iterate / advance gate (DR-032)**: if a blueprint/work orders already existed, this is an **iteration** — read `.pandacorp/comms/iteration.md` (phase `architecture`) and refine what's there (don't regenerate or repeat what was discarded), appending one entry per round. Commit the blueprint and present it to the owner: chosen stack and why, estimated cost, risks, and how many work orders. **Don't start the build on your own:** leave `advance_pending: true` in `.pandacorp/status.yaml` and **wait for their "ok, let's build"** — re-running `blueprint` in the meantime = keep polishing the plan. When they approve: `.pandacorp/status.yaml` → `phase: architecture` (ready to build) + `advance_pending: false`, and close the `architecture` thread in `.pandacorp/comms/iteration.md`. Next step: `/pandacorp:implement` (starts the build with a dynamic workflow that orchestrates the subagents, followed in Mission Control).

## Rules
- The TECHNICAL research happens HERE (stack, libraries, integrations, limits): leave little to resolve during the build. The dev agents only research specific gaps on demand.
- The MINIMUM that satisfies the FRDs: no microservices, no speculative infrastructure features, ~$0 cost at launch.
- If an external integration requires payment (paid API, plan), DR-005 applies: escalate to the owner with the amount.
