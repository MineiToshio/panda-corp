# /pandacorp:blueprint

Fase de arquitectura. Se ejecuta EN el proyecto (requiere PRD, FRDs y diseño congelado). Produce **dos artefactos**: el blueprint técnico y los work orders listos para construir.

## Pasos

1. **Propuesta de stack + aprobación** (agente `architect`): parte del stack recomendado (`fabrica/estandares/stack.md`, **últimas versiones estables**), evalúa si hay tecnologías mejores para ESTE proyecto y las propone con trade-offs. **Presenta la propuesta al dueño y espera su aprobación** (gate ligero, DR-002) antes de fijarla. La elección queda como ADR. Las convenciones duraderas (`fabrica/estandares/`) no se discuten.
2. **Blueprint** (agente `architect`): `docs/blueprint.md` con el stack aprobado, arquitectura, modelo de datos completo, contratos de API, integraciones externas (límites y costos), estrategia de testing y deploy. ADRs en `docs/adr/` por cada decisión no obvia.
3. **Plan de eventos** (agente `analytics`): traduce las métricas de éxito y la hipótesis de valor del PRD a `docs/analitica/eventos.md` (qué evento mide cada métrica, dónde se dispara, qué properties — sin PII, DR-025). Es lo que hará que la v1 salga instrumentada y no "a ciegas". Se instrumenta en `implement` y se verifica en `release`.
4. **Diseño de deploy** (agente `devops`, solo si el proyecto despliega de verdad): sección/ADR de despliegue — a dónde, entornos (dev → staging → prod), pipeline CI/CD reproducible, gestión de secretos (store SOPS+age, `fabrica/estandares/servicios-externos.md`) y plan de rollback, partiendo de `fabrica/estandares/infra.md`. **Si la v1 incluye pagos (PRD, DR-035) y el host es Vercel, AVISA al dueño** (warning, no bloqueo): el cobro requiere Vercel Pro (Hobby es no comercial, ban de cuenta completa) → dispara push (DR-038) y el dueño decide. Para un proyecto trivial sin deploy propio, el estándar basta y se omite este paso.
5. **Validación cruzada**: cada FRD debe mapear a componentes del blueprint. Si algo no cierra, vuelve al `architect` antes de continuar.
6. **Instala el stack**: sigue la guía `${CLAUDE_PLUGIN_ROOT}/templates/stack-<elegido>/STACK.md` y los estándares (`fabrica/estandares/`): tipado strict, linter+formatter, framework de tests, estructura de carpetas. Verifica que lint + typecheck + test corren en limpio.
7. **Crea `.pandacorp/verify.sh`** en el proyecto (lo usa el hook de Stop): corre lint + typecheck + tests rápido y sale ≠0 si algo falla. **Endurecido (DR-019)**: fail-closed (salida ambigua = fallo), entorno limpio, no expone al agente generador los nombres de tests/fixtures, y **mensajes accionables** (qué regla falló y por qué). La guía del stack trae el contenido base.
8. **CI**: GitHub Actions con lint + typecheck + tests en PR (plantilla en la guía del stack). Hacia main: **mutation testing** (Stryker/mutmut, DR-016) + **auditoría OWASP agentic** si aplica (DR-017) + generación de **changelog/ADRs** (DR-018). Branch protection en main si el repo GitHub existe. El pipeline de deploy lo dejó diseñado el `devops` (paso 4).
9. **Work orders** (mismo paso, segundo artefacto): genera `docs/work-orders/wo-NN-<nombre>.md` desde los FRDs + blueprint (ver el detalle del skill `work-orders`): tareas implementables, **pequeñas y testeables en aislamiento**, ordenadas por dependencias, con criterios copiados y definición de terminado. Lista maestra en `docs/work-orders/README.md` con orden y qué se paraleliza.
10. **Iterar / gate de avance (DR-032)**: si ya había blueprint/work orders, esto es una **iteración** — lee `docs/iteracion.md` (fase `arquitectura`) y refina lo que hay (no regeneres ni repitas lo descartado), apendando una entrada por ronda. Commitea el blueprint y presenta al dueño: stack elegido y por qué, costo estimado, riesgos, y cuántos work orders. **No arranques la construcción solo:** deja `avance_pendiente: true` en `docs/estado.yaml` y **espera su "ok, a construir"** — re-correr `blueprint` mientras tanto = seguir puliendo el plan. Cuando apruebe: `docs/estado.yaml` → `fase: arquitectura` (lista para construir) + `avance_pendiente: false`, y cierra el hilo `arquitectura` en `docs/iteracion.md`. Siguiente paso: `/pandacorp:implement` (arranca la construcción con un workflow dinámico que orquesta a los subagentes, se sigue en Mission Control).

## Reglas
- La investigación TÉCNICA se hace AQUÍ (stack, librerías, integraciones, límites): deja poco para resolver en construcción. Los dev agents solo investigan huecos puntuales a demanda.
- Lo MÍNIMO que cumple los FRDs: sin microservicios, sin features de infraestructura especulativas, costo ~$0 al lanzar.
- Si una integración externa requiere pagar (API de pago, plan), aplica DR-005: escalar al dueño con el monto.
