# /pandacorp:architecture

Architecture phase. Runs IN the project (requires PRD, FRDs and frozen design). Produces **two artifacts**: the technical blueprint and the work orders ready to build.

> **Preflight (DR-045) — is this a Pandacorp project?** This skill mutates the project, so first confirm the Pandacorp marker: `.pandacorp/status.yaml` exists. If it's missing, STOP and tell the owner (in Spanish) that this folder isn't a factory project yet — `/pandacorp:adopt` brings an existing project in, `/pandacorp:spec` creates a new one. Then, if `overlay_version` in `.pandacorp/status.yaml` is behind the plugin's `OVERLAY_VERSION`, run `/pandacorp:upgrade` first (silent for compatible bumps, DR-048) so this skill runs against the current structure. Don't proceed or invent docs over a missing structure.

## Steps

1. **Stack proposal + approval** (`architect` agent): starts from the recommended stack (`factory/standards/stack.md`, **latest stable versions**), evaluates whether there are better technologies for THIS project and proposes them with trade-offs. **Presents the proposal to the owner and waits for approval** (light gate, DR-002) before locking it in. The choice is recorded as an ADR. Durable conventions (`factory/standards/`) are not up for debate.
2. **Architecture — two layers, never fused (DR-049)** (`architect` agent):
   - **Platform architecture** (`docs/product/architecture.md`, stable, ONE per project, following `${CLAUDE_PLUGIN_ROOT}/templates/docs/architecture-template.md`): approved stack, high-level architecture, complete data model, deploy strategy, cross-cutting concerns and service boundaries — the things that are stable across features.
   - **Per-FRD blueprint** (`docs/frds/frd-NN-<slug>/blueprint.md`, one per FRD in scope): how THIS feature is implemented on top of the platform — its components `CMP-NN-<slug>` and interfaces `IF-NN-<slug>`, API/action contracts and external integrations specific to the feature (limits and costs), each traced back to the FRD's `REQ-NN-MMM`. Each blueprint follows `${CLAUDE_PLUGIN_ROOT}/templates/docs/blueprint-template.md`: it restates the **source-of-truth hierarchy** in its header, references `docs/product/architecture.md` instead of re-describing the platform, carries the **state frontmatter** (`status` + `implementation_status`), and MUST include a **Build Plan** — the DAG of that FRD's work orders (which is first, which depend on which, what runs in parallel, the integration order, and any cross-FRD dependencies). **The build engine READS this Build Plan instead of re-inferring dependencies at runtime (DR-050)** — this is what makes the build fast and the integration correct. The Build Plan also **records each WO's `artifacts` (the globs of files/dirs it writes) and must be verifiably DISJOINT within a wave (DR-060)**: design wave-parallel WOs so their `artifacts` don't overlap — the engine enforces this from the declared artifacts (it serializes any overlap into a later wave), but the plan should be disjoint by design. The API contract is **per-WO at `docs/api/<wo-id>.md`** (in that backend WO's artifacts), never a single shared file.
     - **Foundation wave first (DR-057).** Before the feature work orders fan out, the Build Plan MUST identify the **shared design-system foundation** — the core primitives that multiple features reuse (`Button`, `Banner`/`Alert`, `Card`, `Chip`, `Modal`, `Input`, `Table`, …) — and schedule them as the **FIRST work order(s)**, a *foundation wave* that completes BEFORE any feature work order starts. **The persistent APP SHELL is a foundation work order too (DR-075):** when `design` Step 8a captured an `AppShell`/`Nav` in `components.md` (a whole-app prototype has a persistent topbar/nav/footer/layout frame), schedule it in the foundation wave — its `artifacts` include `app/layout.tsx` + the nav component — so it is built FIRST and every surface route mounts into it. This is the structural fix for the MC failure (the per-FRD shard drops the global shell because it belongs to no FRD): making it a foundation WO — **not** a new FRD — means the existing foundation-completeness gate enforces "no surface until the shell is green" for free, with no engine change and no `REQ-NN-MMM` spine collision. That wave produces the primitives in `components/core`/`components/modules` **and seeds the living component inventory `docs/design/components.md`** (name · purpose · path · key props/variants). Feature work orders then declare a **`Dependency` on the foundation WO** and build against real, existing primitives — never reinvent them. This is what stops parallel feature agents (who never talk to each other) from each building their own near-duplicate banner/card/modal; without it you get the exact MC defect that motivated DR-057 (two warn banners, two layouts, no shared `Banner`). **Foundation-WO NUMBERING (do NOT invent a `00` FRD):** the foundation/shell work orders belong to a REAL owning FRD — almost always the platform-shell feature **FRD-01** (site shell / i18n / theming) — so they are numbered as that FRD's own correlatives (`WO-01-001`, `WO-01-002`, `WO-01-003`, …) and carry `parent: FRD-01` + `foundation: true`. The id's first segment is ALWAYS the owning FRD number (`WO-NN-MMM`); the "this is a Wave-0 foundation WO" fact is expressed by the **`foundation: true` frontmatter + the Build Plan/`dependsOn`**, NEVER by a `WO-00-*` prefix (there is no FRD-00 — a `00` id is a defect the readiness gate rejects). Large feature → promote `blueprint.md` to a `blueprints/` folder split by coherent concern. A single global `docs/blueprint.md` is an anti-pattern — don't produce one.
   - ADRs in `docs/adr/` (each following `${CLAUDE_PLUGIN_ROOT}/templates/docs/adr-template.md`) for each non-obvious platform-level decision. **For monetary/mixed (DR-042)**: compute the **per-active-user variable cost** (Polar fee ~4%+$0.40, Vercel Pro share, Neon/R2/Resend/PostHog usage tiers) and hand it to the PRD's unit-economics block so the break-even is real, not a guess.
3. **Event plan** (`analytics` agent): translates the PRD's success metrics and value hypothesis into `docs/analytics/events.md` (following `${CLAUDE_PLUGIN_ROOT}/templates/docs/events-template.md`; which event measures each metric, where it fires, what properties — no PII, DR-025). This is what makes v1 ship instrumented rather than "flying blind". It is instrumented in `implement` and verified by the build's hardening step (DR-085 — the engine's telemetry-verification stage; `release` only confirms it live).
4. **Deploy design** (`devops` agent, only if the project actually deploys): deployment section/ADR — where to, environments (dev → staging → prod), reproducible CI/CD pipeline, secrets management (SOPS+age store, `factory/standards/external-services.md`) and rollback plan, starting from `factory/standards/infra.md`. **If v1 includes payments (PRD, DR-035) and the host is Vercel, WARN the owner** (warning, not a blocker): charging requires Vercel Pro (Hobby is non-commercial, full account ban) → it triggers a push (DR-038) and the owner decides. For a trivial project with no deploy of its own, the standard is enough and this step is skipped.
5. **Cross-validation + feasibility + FRD reconciliation**: each FRD must map to components (`CMP-NN-<slug>`) of its blueprint. **Read `docs/design/technical-assumptions.md`** (the designer's feasibility flags): for each costly/risky assumption (real-time, offline, server-side media, costly/uncertain third-party, scraping with ToS/rate-limit risk) confirm the architecture can support it cheaply; if a critical technical unknown remains, run a **timeboxed feasibility spike to prove it BEFORE committing the data model and work orders** (the order fix — don't discover infeasibility mid-build). **If the cross-validation or a spike changes an FRD's criteria, the `product-manager` updates the affected FRD(s) (+ADR) before work orders are generated** — work orders must never carry stale criteria (anti spec-drift). If something doesn't add up, go back to the `architect` before continuing.
6. **Install the stack**: follow the guide `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/STACK.md` and the standards (`factory/standards/`): strict typing, linter+formatter, test framework, folder structure. Verify that lint + typecheck + test run clean. **CANONICAL-HARNESS TRIGGER GATE (BL-0016, DR-105) — run this BEFORE the VERBATIM copy below.** The VERBATIM install assumes the chosen stack's canonical gate harness EXISTS under `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/` (it does for stack A). Stacks B (Hono) / C (FastAPI) / D (scraper) are **PROVISIONAL** — their STACK.md carries a `PROVISIONAL STACK` banner and ships **prose only**: no canonical `verify.sh`, `canary.sh`, lint/type config or e2e set. **You MUST NOT silently hand-roll a `verify.sh` from the prose contract** — that is the exact DR-059 pre-audit failure mode (each project inventing its own gates, `/upgrade` unable to diff them). So, before copying anything: check whether `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/verify.sh` exists. **The absent `verify.sh` is the load-bearing signal — there is no canonical harness to copy VERBATIM — corroborated by the `PROVISIONAL STACK` banner in that stack's STACK.md.** (Note stack D is a *pointer stub* folded into stack C — its STACK.md redirects to `stack-c-fastapi`, so a scraper/collector project resolves to and installs **stack C**; run this gate against stack C.) **If `verify.sh` is MISSING for the resolved chosen stack (the provisional stacks B and C today), this project is the FIRST real project on that stack, which is precisely the BL-0016 trigger (per DR-105: "the architecture skill must treat choosing B/C/D as triggering that harness work, not a silent hand-rolled verify.sh").** Build that ONE stack's canonical harness FIRST, as a prerequisite step of THIS architecture run, mirroring stack A's shape exactly: author `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/{verify.sh,canary.sh}` (`set -euo pipefail` + `inherit_errexit`; fail-closed lint/type/test/dead-code gates at max strictness — the `--error-on-warnings`-equivalent; the stack's structural grep gates from its STACK.md "gate contract"; the DR-100 residual-ambiguity scan; actionable messages) and the canonical lint/type config (**B**: `biome.json` = stack-a's minus the `react`/`next` domains + `knip.json`; **C/D**: ruff + mypy strict `pyproject.toml` tool sections), plus `canary.sh` with the deliberately-broken fixtures that prove every fail-closed gate still goes RED (DR-079), and wire the **browser-gate opt-out as a DECLARED vacuous pass** for the headless stack (DR-059: an opt-out is a decision, not an omission). Then **run `verify.sh --canary` to PROVE the gates bite** before the project depends on them. If building the full harness inline is too large to finish cleanly in this run, **STOP and escalate to the owner** that the chosen provisional stack needs its canonical harness built (file/advance BL-0016) as a prerequisite — **never proceed with an ad-hoc hand-rolled gate**. (Building the harness for the OTHER provisional stacks you did not choose is out of scope — build ON FIRST USE only, DR-105.) Once the harness exists (either it already shipped, or you just built + canary-proved it in this step, and — for a stack you just built — you have removed that STACK.md's `PROVISIONAL STACK` banner and updated DR-105's nota), continue with the VERBATIM install below exactly as for stack A. **Install the canonical gate config VERBATIM (DR-059)** — the same way `docs/rules/` are copied: copy `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/biome.json` and `knip.json` to the project **root**, and `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/{verify.sh,canary.sh}` to `.pandacorp/` (`verify.sh` plus the DR-079 gate canary `canary.sh` that `verify.sh --canary` runs to prove each fail-closed gate still goes RED on a deliberately-broken input), **byte-for-byte, never hand-rolled or partially re-tuned**. These carry the MAXIMUM fail-closed config (the `react`/`next`/`test` domains, the `a11y` group, the hard-enforcement rules promoted to **error**, `noExcessiveCognitiveComplexity: "error"`, and the dead-code/circular/smoke/visual gates) — the meta-fix for the audit where the prose standards never got wired into a real project. The STACK.md guide's **hard-enforcement** section and **Library conventions** are the *rationale* for that config; project-specific additions go in separate files, never by editing the canonical ones (so `/pandacorp:upgrade`'s conformance check stays clean). **Also install the e2e gate machinery VERBATIM (DR-055/056/074/075)** — copy `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/e2e/{smoke.spec.ts,visual.spec.ts,responsive.spec.ts,_responsive-helper.ts,shell.spec.ts,headers.spec.ts}` into the project's `e2e/` and `${CLAUDE_PLUGIN_ROOT}/templates/stack-<chosen>/e2e/playwright.config.ts` to the project **root**, byte-for-byte (this is the conformance-checked set). **Seed (do NOT overwrite) the per-project files**: create `e2e/routes.ts`, `e2e/_target.ts` and `e2e/shell.ts` from the templates **only if missing** — these carry the project's own surface list / platform read / app-shell nav contract and are NEVER byte-diffed by `/pandacorp:upgrade`. **For `e2e/shell.ts` (DR-075), materialize the nav contract recorded by `design` Step 8a:** if a whole-app prototype with a persistent shell exists, populate `SHELL_SELECTOR` + `NAV_DESTINATIONS` (`{label, path}` per TOP-LEVEL nav link, read from the prototype's nav block in `docs/design/prototype/` / the `components.md` `AppShell` row — NOT from `routes.ts`) + `SHELL_EXEMPT` + `NAV_TOGGLE`; if the app has no persistent shell, leave `NAV_DESTINATIONS` empty (a vacuous-pass gate). Ensure the `package.json` scripts `test:smoke` (`playwright test e2e/smoke.spec.ts`), `test:visual` (`playwright test e2e/visual.spec.ts`), `test:responsive` (`playwright test e2e/responsive.spec.ts`) and `test:shell` (`playwright test e2e/shell.spec.ts`) exist, and add `@axe-core/playwright` to `devDependencies` (the responsive gate's tap-target check). **Wire the dev ports** from the project's `.pandacorp/status.yaml` `dev_port_base` (reserved at scaffold from the factory ledger; port convention in `factory/standards/infra.md`, DR-022): write `.claude/launch.json` with the app on `base+0` (`runtimeExecutable`/`runtimeArgs` = the stack's dev command, e.g. `pnpm` `["run","dev"]`, plus the matching `--port`/`-p base` flag and `"port": base`) so the dev/preview server starts straight from Claude Code, and set the app/DB/service ports in `.env` per the block offsets (Postgres `+2`, Redis `+4`). If `dev_port_base` is `0` (e.g. a project scaffolded before this convention), allocate a block now from `factory/ports.yaml` exactly as `scaffold` does, then materialize. **Pin dependency versions AT THE INSTALL POINT, never only in prose (DR-102).** Install the stack's libraries at the exact versions the `architect` chose — **applying the new-release cooldown (`web-security.md` SEC-4): before adding, check the version's publish date (`pnpm view <pkg> time --json`) and never install one published < ~7 days ago (take the previous stable instead); set `.npmrc` `minimum-release-age` where the package manager supports it** — run `pnpm add <pkg>@<version>` for each, OR (when a dep is installed later by a foundation work order to keep `knip` clean) write the exact `pnpm add <pkg>@<version>` command INTO that work order's install step. A version that lives only as text in `architecture.md` lets a build agent resolve a different (often newer-major) version → avoidable iterations. **Generate `.env.example` at the project root (DR-102):** the inventory of every env var the app reads (name + a one-line comment, empty value), so an agent never reverse-engineers var names from prose; commit `.env.example`, keep real `.env`/`.env.local` gitignored. **Generate/refresh `AGENTS.md` to point agents at the framework's VERSION-MATCHED docs (DR-102):** when the framework ships bundled, version-pinned agent docs (e.g. Next.js `create-next-app` emits `AGENTS.md`/`CLAUDE.md` redirecting agents to the INSTALLED version's docs instead of their training data — measurably fewer errors on its `/evals`), ensure that pointer exists, merged below the managed Pandacorp header. This is the direct fix for the dominant 100%-AI failure mode: the model coding against an outdated framework API it "remembers" (this run's `middleware.ts`-vs-`proxy.ts` / missing `setRequestLocale` blockers were exactly that).

6b. **Inject the tech-gated engineering rules.** Now that the stack is locked, copy from `${CLAUDE_PLUGIN_ROOT}/templates/rules/` every rule file whose `applies_when` (or `also_applies_when`) matches a technology this project uses — e.g. `react.md` (React), `nextjs.md` (Next.js App Router), `styling-and-ui.md` (Tailwind), `web-performance.md`/`web-security.md` (web UI / public web), `prisma.md` (Prisma), `i18n-next-intl.md` (next-intl), `analytics-and-errors.md` (PostHog/Sentry) — into the project's `docs/rules/` (the `always` files are already there from scaffold). A technology the project does NOT use → its file is NOT copied. Then **regenerate `docs/rules/README.md`** (canonical spec in the `scaffold` skill: directive header + one `@import` per file present + table). Copy verbatim — never re-condense.
7. **`.pandacorp/verify.sh`** (used by the Stop hook): this is the canonical `verify.sh` installed VERBATIM in step 6 (DR-059) — do NOT author a fresh one. It is MAXIMUM fail-closed (DR-019): `set -euo pipefail` + `inherit_errexit`, every Biome warn promoted to a hard gate (`--error-on-warnings`), `tsc`/`knip`/`madge`/`vitest`, and the smoke (DR-055) + visual (DR-056) + responsive (DR-074) + shell (DR-075) shims where a **missing** harness is RED, never a skip — with actionable messages (which rule failed and why). Confirm it runs; never hand-edit it (drift fails the conformance check).
8. **CI**: GitHub Actions with lint + typecheck + tests on PR (template in the stack guide). Toward main: **mutation testing** (Stryker/mutmut, DR-016) + **OWASP agentic audit** if applicable (DR-017) + **changelog/ADR** generation (DR-018). Branch protection on main if the GitHub repo exists. The deploy pipeline was left designed by the `devops` (step 4).
   - **Visual spec for UI features (DR-054).** For a UI FRD (`ui: true`), the Build Plan must point implementers at the FRD's `fdd.md` + the specific `mocks/` screen(s) as the **binding visual spec to reproduce** (layout, structure, components, density — not an approximation; "uses tokens" is not enough), and **enumerate the assets the screens need** (fonts/images/sprites/icons) so they get staged (e.g. into `public/`) and actually render. The fidelity is checked by the preview smoke gate (DR-055).
9. **Work orders** (same step, second artifact): generates **coarse per-FRD** work orders (see the `work-orders` skill) following `${CLAUDE_PLUGIN_ROOT}/templates/docs/work-order-template.md`, born **`status: DRAFT`** (they flip to `ACTIVE` only in step 9d, after BOTH gates pass — nothing is born `ACTIVE`). Each is **one cohesive view / page / capability** (NOT an atomic component), carries the **state frontmatter** (`implementation_status: PLANNED`; `artifacts: [globs]` — the files/dirs it writes, the engine serializes WOs whose artifacts overlap, DR-060; and **`difficulty: low | medium | high`** — rate it by scope/novelty/integration surface: `high` builds on opus a-priori, DR-073 — without this field the escalation silently never fires) and a `## Status Note` hand-off, cites its `REQ-NN-MMM`/`AC-NN-MMM.K`/`CMP-NN-*`/`IF-NN-*` with the AC copied in, and lists its `Dependencies` consistent with the blueprint's **Build Plan**. A UI work order also carries its `## Visual reference` (the `mocks/` it reproduces + the fidelity AC, DR-054). Per-FRD list in `docs/frds/frd-NN-<slug>/work-orders/README.md`. **Then run the blueprint READINESS GATE (DR-100) — by a FRESH agent, never the author.** Spawn a dedicated reviewer (the same independence DR-102's step 9b demands; constitution rule 4 — an author grading its own plan is self-certification, audit-20 P1-2) that asserts fail-closed: every `REQ` maps to a component, **every `AC` is covered by exactly one WO**, the data model has no `TBD`, `dependsOn` is acyclic + complete, wave artifacts are disjoint by design, the foundation is complete, **no `[NEEDS CLARIFICATION]` survives**, and **each backend WO's API contract has an owner** — a WO in the plan whose install/build step will materialize `docs/api/<wo-id>.md` BEFORE its consumer runs (the contract file itself is authored at build time by that WO, DR-060 — the gate checks the *plan assigns it*, not that the file already exists). A blueprint with holes produces ambiguous work orders; don't advance until the gate is clean.

9b. **REPO-GROUNDING VALIDATION — validate the plan against the INSTALLED scaffold before anything goes `ACTIVE` (DR-102).** The readiness gate (Step 9) proves internal *cohesion*; this proves *external fidelity to the real repo* — the single most effective error-reducer for AI-built apps in the literature (it front-loads detection of "context blindness": specs/WOs that are internally coherent yet reference framework APIs, file paths or versions the model only *remembers*, which then surface as mid-build rework). Run a dedicated reviewer (a fresh agent, not the author) that, for EVERY concrete claim in `architecture.md` + the per-FRD blueprints + the work orders, checks it against the scaffold actually installed in Step 6: (a) every cited **file/dir path** (artifacts, `proxy.ts` vs `middleware.ts`, route files, config) exists or is creatable as written under the installed framework's conventions; (b) every cited **framework API / convention / config key** is current for the INSTALLED major (cross-check the framework's version-matched docs the `AGENTS.md` points at — not training memory); (c) every **dependency version** named in `architecture.md` is actually installed or carried as an exact `pnpm add <pkg>@<version>` in a work order (no version that exists only in prose); (d) the **rendering/SSG and other hard guarantees** are achievable as specified (e.g. for next-intl, that `setRequestLocale` is specified so `[locale]` is static — verify against a real `pnpm build`, not assumption). Any mismatch is a BLOCKER fixed (in the docs, the WO, or the scaffold) BEFORE `ACTIVE`. Keep the altitude discipline (DR-100): this validates that the plan is *grounded and current*, NOT that it is *voluminous* — tighten contracts and fix mismatches, don't inflate specs (more/better prose alone does not reduce defects; grounded contracts + executable checks do).
9b2. **The DRAFT→ACTIVE flip — owned HERE, with the evidence stamp (audit-20 P1-3/P1-1).** When BOTH gates pass (9's readiness + 9b's repo-grounding), the flip finally has an owner: set each per-FRD `blueprint.md`'s frontmatter `status: ACTIVE` **and stamp the evidence in the same frontmatter** — `readiness_gate: passed YYYY-MM-DD` and `grounding_gate: passed YYYY-MM-DD` — then flip that FRD's work orders `status: DRAFT → ACTIVE`. The stamp is what makes the gates CHECKABLE downstream: `/pandacorp:implement`'s preflight asserts these fields exist on every ACTIVE blueprint (a blueprint that reached ACTIVE without them never passed the gates — the preflight REFUSES to build it), and `verify.sh`'s DR-100 marker scan covers the rest. If either gate found blockers, docs STAY `DRAFT` until fixed — never flip to silence a gate. (The PRD/FRDs' own ACTIVE flip belongs to `spec`, at the owner's "ok, advance".)

9c. **Architecture digest — the owner-facing Spanish summary (powers Mission Control's "Arquitectura" tab).** Write `.pandacorp/comms/arquitectura-resumen.md` following `${CLAUDE_PLUGIN_ROOT}/templates/docs/architecture-digest-template.md` — a HIGH-LEVEL, scannable **Spanish** digest of the platform architecture: the stack table (capa · elección · por qué from `docs/product/architecture.md`), the **data model** (the DB schema/entities when the project has one; for a content-as-code / no-DB project a one-line "Sin base de datos — contenido como código" + the content entities), communication & services (how the app talks to the outside), and ONE card per FRD (its blueprint one-liner + its work orders summarized). Mission Control renders it natively in the **Arquitectura tab** (after Spec), shown only when this file exists — so the board surfaces the project's architecture from the architecture phase. The implementation-plan DAG, the ADRs and the env vars are read LIVE by the tab from the real artifacts (the work-order frontmatter, `docs/adr/*`, `.env.example`) — the digest does NOT duplicate them. It is a SUMMARY for the owner, **not a copy** of the docs (those stay English under `docs/`); the digest is gitignored Spanish (the comms layer, per the language rule). **Keep the template's markers EXACTLY** (the `## 🧱 Stack & tecnologías` table, the conditional `## 🗄️ Modelo de datos`, `## 🔌 Comunicación & servicios`, and the per-FRD `### FRD-NN · Título` + `**Blueprint:**` / `**Work orders:**` blocks) so the tab's parser renders it rich. **Regenerate it on every `architecture` iteration** so it never drifts from `architecture.md`/the blueprints.

10. **Iterate / advance gate (DR-032)**: if a blueprint/work orders already existed, this is an **iteration** — read `.pandacorp/comms/iteration.md` (phase `architecture`) and refine what's there (don't regenerate or repeat what was discarded), appending one entry per round. Commit the blueprint and present it to the owner: chosen stack and why, estimated cost, risks, and how many work orders. **Don't start the build on your own:** leave `advance_pending: true` in `.pandacorp/status.yaml` and **wait for their "ok, let's build"** — re-running `architecture` in the meantime = keep polishing the plan. When they approve: `.pandacorp/status.yaml` → `phase: architecture` (ready to build) + `advance_pending: false`, and close the `architecture` thread in `.pandacorp/comms/iteration.md`. Next step: `/pandacorp:implement` (starts the build with a dynamic workflow that orchestrates the subagents, followed in Mission Control).

## Rules
- The TECHNICAL research happens HERE (stack, libraries, integrations, limits): leave little to resolve during the build. The dev agents only research specific gaps on demand.
- The MINIMUM that satisfies the FRDs: no microservices, no speculative infrastructure features, ~$0 cost at launch.
- If an external integration requires payment (paid API, plan), DR-005 applies: escalate to the owner with the amount.
