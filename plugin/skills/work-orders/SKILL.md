# /pandacorp:work-orders

Decomposition into **coarse** work orders. Runs IN the project (requires the FRDs with their per-FRD blueprints and the platform `docs/product/architecture.md`).

## Steps

1. Read all the v1 FRDs (`docs/frds/frd-NN-<slug>/frd.md`), each FRD's `blueprint.md` (including its **Build Plan**) and the platform `docs/product/architecture.md`.
2. **Generate per-FRD work orders** at `docs/frds/frd-NN-<slug>/work-orders/wo-NN-MMM-<slug>.md`, each following the template `${CLAUDE_PLUGIN_ROOT}/templates/docs/work-order-template.md`:
   - **The frontmatter is the source of truth** (DR-050): `implementation_status: PLANNED` (the build engine reads this to know what to build), `status: ACTIVE`, `parent: FRD-NN`, **`artifacts: [globs]`** (every file/dir this WO writes — the engine keeps parallel WOs disjoint by serializing any whose artifacts overlap into different waves, DR-060; a backend WO lists its own contract `docs/api/<wo-id>.md` here too), **`difficulty: low | medium | high`** (DR-073 — the model uses it to pick opus a-priori; **default `medium` when absent**, backward-compatible. Rubric: **low** = one small component / ≤2 EARS / pure presentational reuse; **medium** = a cohesive view / a few components / 1 integration seam; **high** = a feature-rich surface / multiple integration seams / cross-navigation or stateful interactions / ≥5 EARS / novel logic — a `high` WO is built on opus from the first attempt), `source_requirements` with the `REQ-NN-MMM` it delivers, and **`dependsOn: [WO-NN-MMM]`** (DR-087) — the upstream work orders this one depends on, the **machine-readable mirror** of the README "Depends on" column / blueprint Build Plan DAG (list the real upstream WOs, cross-FRD allowed; `[]` when none). Mission Control's dependency graph reads `dependsOn` verbatim — keep it consistent with the README table (no fabricated chains). **Capture the architectural couplings, not just build order** (DR-087a, see `build-orchestration.md` "What counts as a dependency"): a UI-surface WO depends on its own libs + the **design-system foundation** WOs it uses + the **data-reader** WOs it consumes + any **feature whose data/engine it renders** (e.g. live view ← build modes, achievements ← event readers). Under-capturing deps yields a sparse, misleading DAG; only declare evidence-based couplings (what the WO actually imports/reads), never speculative ones.
   - The `REQ-NN-MMM`/`AC-NN-MMM.K` it implements with the **acceptance criteria copied inline** (the implementer should not have to go look for them) and the `CMP-NN-*`/`IF-NN-*` it builds against.
   - In/Out scope, `Dependencies` (prior WOs by `WO-NN-MMM`, intra- or cross-FRD), and a **`## Status Note` (hand-off)** section the implementer fills on close so the next agent reads it instead of re-reading all the code.
   - **Reuse before create (DR-057).** A **foundation work order** (the first wave of the Build Plan, DR-057) **produces/updates the living component inventory `docs/design/components.md`** alongside the shared primitives it builds (`Button`, `Banner`, `Card`, `Modal`, …). Every **UI feature** work order then lists, in its scope, which **existing** components from `docs/design/components.md` it **reuses** (so the implementer adapts, not re-creates them) and declares a `Dependency` on the foundation WO. The implementer's contract (`implementer`/`frontend-dev` agents): read the inventory, reuse→adapt→create, and append any genuinely-new shared component to `docs/design/components.md`. A WO that re-creates an inventory primitive (a second banner/card/modal) is building a defect.
   - **UI work orders carry a FILLED `## Visual reference` (DR-054 + DR-056).** For a work order with UI, the `## Visual reference` is **materialized, never a placeholder** — the build engine passes the implementer only a one-line summary, so **the WO body IS the agent's design context**; an empty `## Visual reference` means the agent builds blind. Fill it with three things: (1) the exact `docs/frds/frd-NN-<slug>/mocks/<file>` screen(s) it must **reproduce** (the binding visual spec — the scoped per-FRD shard, on the frozen tokens); (2) the relevant **token slice** — the specific tokens that screen actually uses (colors/spacing/type/radii/elevation it touches), not the whole token file; (3) a pointer to the FRD's `fdd.md`. Add a **fidelity acceptance criterion** — *"the built screen visually matches `mocks/<file>` (layout, structure, components, density) on the frozen tokens; verified by the preview smoke gate, DR-055"* — alongside the functional `AC-NN-MMM.K`. Not only the functional ACs: a screen that "uses tokens" but doesn't look like the mock is not done. Non-UI work orders mark this section `N/A`.
3. **Granularity — COARSE, with a named ceiling (DR-050 + DR-100).** One work order = one cohesive view / page / capability (e.g. "order detail view + payments panel + action menu"), NOT one atomic component. **Prefer fewer, larger work orders — when in doubt, merge.** The build engine reviews and tests **per FRD**, so each WO must carry enough context to build the whole slice end-to-end. The lower bound stays coarse; the **upper bound is concrete**: split a WO that is **too big to review in one sitting**, that would push the implementer **past the model's reliable context zone** (quality degrades well before the window fills), or that **mixes more than one concern** (don't bundle refactor + feature + analytics). Merge anything smaller than a usable slice. Typical: a handful of coarse work orders per FRD (think PandaTrack: ~one per view), not dozens of tiny ones.
   - **One module per work order — declared, ENGINE-ENFORCED disjoint artifacts (DR-060).** Two work orders that build in the same wave must NOT write the same file/module: parallel implementers would collide (a real failure mode). Each WO **declares `artifacts: [globs]`** in its frontmatter; the engine computes the sets and **serializes any WOs whose artifacts overlap into different waves**. Still design for disjointness: if two slices need the same file, **merge them into one work order** (preferred) or make one depend on the other so they serialize by plan. A work order's `artifacts` must be **disjoint** from its wave-siblings'. The per-WO API contract (`docs/api/<wo-id>.md`) replaces the old shared `docs/api.md` that N parallel agents raced on, and is itself listed in that WO's artifacts. This alone pushes granularity coarser, the right way.
4. **The build order lives in the blueprint's Build Plan** (written by `/pandacorp:blueprint`: the DAG of deps + parallelism + integration order). Each work order's `Dependencies` must be consistent with that plan; the per-FRD list with what parallelizes lives in `docs/frds/frd-NN-<slug>/work-orders/README.md` (following `${CLAUDE_PLUGIN_ROOT}/templates/docs/work-orders-readme-template.md`). Cross-feature order comes from the `Dependencies` + the blueprints' cross-FRD notes.
5. **Update** `.pandacorp/status.yaml` → `phase: implementation` and commit. Summary to the owner: how many coarse work orders, the build order, rough estimate. Next step: `/pandacorp:implement`.

## Rules
- **State lives in the frontmatter** (`implementation_status`: `PLANNED → IN_PROGRESS → IN_REVIEW → VERIFIED`, + `BLOCKED`), never in body prose or in git (DR-050). The engine builds only `PLANNED`/`IN_PROGRESS` and never rebuilds `VERIFIED`.
- The first work order of the data-owning FRD is the base: data schema + seeds + environment smoke test.
- Each acceptance criterion (`AC-NN-MMM.K`) of each FRD is covered by **exactly one** work order (no gaps or duplicates) — this is a **checked** condition of the blueprint **readiness gate** (DR-100), not a hope: the union of the WOs' `source_requirements` must equal the FRD's AC set.
- **No `[NEEDS CLARIFICATION]` in a work order (DR-100).** A WO is generated from a settled blueprint; if a material ambiguity remains, it belongs back in the FRD/blueprint (resolve it there), not buried in a build task. The readiness gate REDs on a residual marker.
