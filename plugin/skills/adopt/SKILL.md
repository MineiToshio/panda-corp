# /pandacorp:adopt

Brings a project built **outside** the factory under Pandacorp management. Run it **inside** the existing project folder. It is the brownfield mirror of `scaffold` (greenfield): instead of creating an empty project, it wraps what already exists, so that from then on the `/pandacorp:*` skills, Mission Control and the write-gate (DR-044) apply to it. Touching an existing project = **human gate**: present the adoption plan and get the owner's OK before writing anything (DR-045, DR-038).

## When to use
The owner has a project they built by hand (or cloned) and wants the factory to manage it. If the folder is ALREADY a Pandacorp project (`.pandacorp/status.yaml` present), stop — it's already adopted. If the folder is empty / has no real code, this isn't an adoption → use `/pandacorp:spec`.

## Steps

1. **Idempotence / sanity check.** If `.pandacorp/status.yaml` exists, say it's already a factory project and stop. Confirm there's a real codebase here.

2. **Read the project to understand it (NO writes yet).** Inspect the repo to learn what it IS and what PHASE it's in:
   - Stack & structure: `package.json` / `pyproject.toml` / etc., framework, entry points, `git log` (history, maturity), `git remote` (does a GitHub repo already exist?).
   - Maturity signals → **inferred phase** (the valid phases are `product` | `design` | `architecture` | `implementation` | `release` — there is no `operation`, DR-085): a launched/live project (a live deploy, CI to prod, a production URL, or an in-house tool already in daily use) → `release` (set `deploy_target: external` for a hosted deploy, `internal` for an in-house tool used as-is); substantial code + tests but not launched → `implementation`; only boilerplate/scaffolding → `architecture`. State the inference AND its evidence.
   - What the app DOES (features, routes, data model) — the raw material for the as-built docs.
   - **Existing design (DR-054)**: scan for a visual source of truth — a `prototype/` or any `*.html` mockups, `docs/design/` artifacts (`DESIGN.md`, `design-tokens.json`, references), a token/CSS system in the stylesheets, screenshots, or the live styled UI itself. **Inventory what you find** (don't ask blind); it is presented for the owner to confirm in step 3.

3. **Present the adoption plan + its consequences (HUMAN GATE).** Adoption is a commitment, so spell it out and get an explicit OK before writing (in Spanish): (a) the files to be CREATED vs MERGED (never overwritten) and the retroactive idea card + portfolio row; (b) the inferred phase + evidence; (c) ask the `return_type` (personal / monetary / mixed / opportunity, DR-039); (d) **confirm the existing visual you found (DR-054)** — from the design inventory in step 2, **present what you found and let the owner confirm** (in Spanish): *"Encontré esto como posible referencia visual: `<list the files>`. ¿Es esta la referencia definitiva? ¿Falta alguna, o no son estas?"* If the repo had **nothing**, ask whether it lives elsewhere (*"¿tienes un prototipo o sistema de diseño en otro sitio, p.ej. Figma?"*). The answer decides whether design is **adopted now** as the frozen visual contract (step 6) or deferred; and (e) **what it means going forward** — the project becomes a Pandacorp project, so from now on changes flow through the `/pandacorp:*` skills (write-gate, DR-044), while staying fully workable without the plugin (anyone cloning just the repo follows `AGENTS.md` by hand). DR-038: also push-notify. On "no", stop and change nothing.

4. **Inject the overlay WITHOUT clobbering.** Copy the template overlay (`${CLAUDE_PLUGIN_ROOT}/templates/shared/`) but never overwrite existing files:
   - **New files** (don't exist): copy as-is, process the `.tpl` vars (`{{PROJECT_NAME}}`, `{{FACTORY_PATH}}`, `{{IDEA_FILE}}`, `{{DATE}}`, `{{OVERLAY_VERSION}}` from `${CLAUDE_PLUGIN_ROOT}/templates/OVERLAY_VERSION`), drop the `.tpl` suffix.
   - **`.pandacorp/guide.md`**: always write it (the managed Origin + write-gate + doc-map). It's regenerable — no merge needed.
   - **CLAUDE.md**: if the project HAS one, do NOT replace it — APPEND the import lines (`@AGENTS.md`, `@docs/rules/README.md`, `@.pandacorp/guide.md`) and leave its own content untouched; if it has none, use the thin template. **AGENTS.md**: if absent, add the factory-conventions one (with the `docs/rules/` pointer); if present, merge the write-gate bullet + the `docs/rules/` pointer without disturbing its rules.
   - **.gitignore present**: MERGE the integration-layer ignores (`.pandacorp/comms/`, `.pandacorp/inbox/`, `.pandacorp/run/`), without duplicating.
   - **.claude/settings.json present**: merge the `permissions.deny` entries.
   - Always bring `.claude/workflows/pandacorp-build.js` (the build engine `implement` launches). The marker that makes the factory recognize the project is `.pandacorp/status.yaml` (written in step 5).

5. **Create the structure + state** (feature-centric, DR-049). Create the `docs/` skeleton where missing: `product/`, `design/`, `frds/` (each existing feature → its own `frds/frd-NN-<slug>/` module folder, created in step 6), `adr/`, `reviews/`, and `.pandacorp/{inbox/bugs,run}`. Write `.pandacorp/status.yaml` with the inferred `phase`, the real `version` (from tags / `package.json`, default `v1`), `overlay_version` + `created_with` = the current `OVERLAY_VERSION`, `running: false`, counters at 0, `repo` from `git remote`, **`deploy_target: internal | external` when the inferred phase is `release`** (external for a hosted deploy / production URL, internal for an in-house tool used as-is — DR-085), and **`target_platforms: desktop` (DR-074)** — brownfield has no spec phase to decide it, and the conservative default keeps the new responsive gate green (an adopted app may not be responsive; back-filling `responsive` would red-lock all commits). The owner promotes it to `responsive`/`mobile` later, deliberately, once the project is verified at mobile width. Seed `docs/decision-log.md` and `.pandacorp/comms/iteration.md` if absent. **Inject the engineering rules**: from `${CLAUDE_PLUGIN_ROOT}/templates/rules/` copy into `docs/rules/` every file that is `always` or matches the project's inferred stack (step 1), and generate `docs/rules/README.md` (canonical spec in the `scaffold` skill). For a brownfield project these are as-built guardrails going forward; flag — don't auto-fix — existing code that violates them.

6. **Reconstruct minimal AS-BUILT docs** (so downstream skills have an anchor; feature-centric, DR-049). This is reverse-documentation, clearly marked as reconstructed:
   - `product-manager` writes `docs/product/prd.md` describing what the app does TODAY (problem, users, current scope) + a living **feature-landscape** table of the reconstructed FRDs. For each existing feature, create its module folder `docs/frds/frd-NN-<slug>/` with a `frd.md` (REQ-NN-MMM + EARS AC-NN-MMM.K anchored to ACTUAL behavior; FRD names its parent PRD in frontmatter). Header each one: *"As-built — reconstructed by /pandacorp:adopt on {{DATE}}; describes existing behavior, not a forward plan."*
   - `architect` writes the platform `docs/product/architecture.md` capturing the REAL stack/data model/deploy + `docs/adr/adr-000-adopted-stack.md` (as-built). Where a feature has non-trivial implementation worth recording, add a thin `docs/frds/frd-NN-<slug>/blueprint.md` referencing `product/architecture.md`; don't pre-stub empty ones.
   - **Design — adopt the existing visual, don't leave it orphaned: ORCHESTRATE the full chain (DR-054 + DR-056).** If step 3 surfaced an approved visual (a `prototype/` of mockups, the live styled UI, a tokens/CSS system, a Figma), the `designer` runs the **ADOPT-VISUAL path end-to-end** — detect → confirm → extract → place → **shard** → stage — so EVERY UI FRD ends up with its own scoped mock, not just a global prototype:
     1. **Extract the design system faithfully** into `docs/design/design-tokens.json` + root `DESIGN.md`, recording `docs/design/design-decisions.md` = "frozen FROM the approved visual at `<path>` (adopted {{DATE}})". **Capture every layer (complete extraction, DR-056):** base components + any app-wide skin/theme overrides + each embedded sub-view's own spec — partial extraction silently drops the look (it happened in Mission Control: an app-wide skin and a sub-view spec were missed). **Seed `docs/design/components.md` — the component inventory (DR-057), REQUIRED when the project has UI:** one row per shared primitive/module the adopted system exposes (name · purpose · path · key props/variants), taken from the components found in the adopted/extracted system (and, where the existing UI already implements them, their real source paths). If the adopted system was pushed up to a Claude Design project (DR-058), bridge its `_ds_manifest.json` → `components.md` (each manifest entry → one inventory row). This is the build-facing inventory later skills reuse (`check before you create`).
     2. **Place the whole-app prototype in `docs/design/prototype/` (DR-056)** — its defined home (the PDD layer); move it there if it sits at a loose ad-hoc path.
     3. **Capture the persistent APP SHELL as a foundation concern BEFORE sharding (DR-075) — this is the brownfield path where MC actually lost its shell.** The shard ("never the whole app") drops the global chrome because it belongs to no FRD; `adopt` documents (it does not build code), so branch on the existing code:
        - Identify the prototype's persistent shell (topbar/primary nav + active-state, header/footer, layout frame) and add `AppShell`+`Nav` to `docs/design/components.md` as a foundation component. Record the nav contract for the Shell-Presence Gate (the shell landmark, the `{label, path}` TOP-LEVEL destinations **from the prototype, not from the app's routes**, the routes that render without the shell, a mobile toggle if any) in the `AppShell` row / `design-decisions.md`.
        - **The existing code HAS a real shell** (a `layout.tsx`/header with a working nav) → record its real source path in `components.md`; the seeded `e2e/shell.ts` can be **filled** from the prototype's nav (the gate verifies the existing shell, which should pass).
        - **The existing code LACKS the shell the prototype shows (the MC case)** → record `AppShell`/`Nav` as an **owed** foundation component AND flag it in the Group-B reconciliation below (DR-064: "prototype HAS it, the app LACKS it") so the re-anchor builds it; the seeded `e2e/shell.ts` stays **empty** (a vacuous-pass gate — no red-lock on a shipped app with no shell) and is filled in the SAME change that builds the shell. (`e2e/shell.ts` is a per-project seed installed/back-filled by the upgrade conformance check, DR-075; `adopt` records the contract here so it's not silently dropped.)
        - **No persistent chrome** (single-screen tool / landing / API) → no shell; leave the seed empty.
     4. **SHARD it into each UI FRD (DR-056):** for every UI FRD reconstructed above, extract that FRD's relevant screen(s) into `docs/frds/frd-NN-<slug>/mocks/` as a **self-contained, scoped HTML slice (only that FRD's screens, never the whole app)** + a screenshot, write its `fdd.md` referencing the mock, and set the FRD frontmatter `ui: true` + `visual_source`. Scoping each work order down to one screen is the top visual-fidelity lever for the build.
     5. **Stage its assets** (fonts, images, sprites — e.g. into `public/`) so the app can serve them; if the visual embeds sub-views (iframes), each embedded file is part of the contract.
     6. **Reconcile FUNCTIONALITY bidirectionally before the prototype is canonical (DR-064).** The reconstructed FRDs are as-built; the prototype may show MORE (or different) behaviour. Resolve both directions explicitly: **Group A** — behaviour the prototype HAS but no reconstructed FRD documents → write it into the owning FRD (AC + `fdd.md`); **Group B** — behaviour an FRD specifies that the prototype LACKS → build it into the prototype or flag it as an explicit gap (never ground a fidelity gate on a missing screen) — **a missing persistent app shell (DR-075) is recorded here as an owed Group-B item**. Surface the itemized A/B list to the owner. A visual-only adoption that leaves functional divergence is not done. See `factory/standards/design.md` §8.

     This becomes the **binding visual source of truth** going forward — **never demote an approved prototype to a "functional-only reference"**, and never tell a later `/pandacorp:design` run to invent a fresh look over it. (This is exactly the step whose absence let Mission Control's `prototype/` sit unused while the build diverged.) If the owner says there is **no** approved visual, record that explicitly in `docs/design/design-decisions.md` (a decision, not an omission) and leave the design system to a later `/pandacorp:design` run.
   - Keep it minimal and honest: stubs with TODOs are fine where behavior is unclear; do NOT invent product rationale. Don't reconstruct work orders for already-built features (they implement forward work). Flag the gaps for the owner.

7. **Register in the factory (bidirectional links) — the board card ALWAYS, the portfolio row once building.** The board (idea cards) and the portfolio (portfolio.md) are two different reads; an adopted project must be registered in BOTH planes correctly or the two desync (the symptom: a project shows in the portfolio but not on the board, or lands in the wrong column).
   - **Always create the retroactive idea card** `factory/ideas/<slug>.md`: `status: in-pipeline`, `project:` = the path **`resolveProjectPath` understands** (relative to the factory root — `../<slug>` for a sibling repo, just `<slug>` for a project that lives *inside* the factory like Mission Control — or an absolute path; **do NOT store a bare slug for a sibling**, the board/portfolio both resolve via the factory root, not via a hardcoded `../`), plus `project_type` (inferred from the stack, step 1), `return_type` (from step 3), `created`, and a note "adopted external project on {{DATE}}". Copy it to `.pandacorp/idea-origin.md`. **This card is what puts the project on Mission Control's BOARD** — it freezes as a pointer, and the board derives its column from the project's `status.yaml` phase (Mission Control FRD-02). Create it for **every** adopted project, whatever its phase, so the board never desyncs from the set of real projects.
   - **Add the `factory/portfolio.md` row only once the build has started** — i.e. the inferred phase (step 2) is `implementation` or `release` (DR-085 — there is no `operation`). The portfolio is the index of projects **en obra y lanzados**; a project still in `architecture` (only boilerplate) lives on the board via its card and gets its portfolio row when `/pandacorp:implement` begins the build (or on the next `/pandacorp:sync-portfolio`). Use the **same path string as the card's `project:`** so both planes resolve to the same folder. Row = project, path, repo, origin idea = "adopted", inferred phase, date.

8. **Commit** in the project: `chore: adopt into pandacorp factory` (overlay + docs only — do NOT touch the app's source). If `gh` is authenticated and there is no remote, OFFER to create a private repo (DR-010); don't force it.

9. **Report & next step (in Spanish).** Summarize: inferred phase, what was created vs merged, the reconstructed docs and their gaps, and the natural next skill for the phase (`release` — a launched project → `/pandacorp:iterate` or `/pandacorp:review-launch`; `implementation` → `/pandacorp:implement` to continue, which now also runs the security/quality/metrics hardening as its last step, DR-085). Remind that from now changes flow through the skills (write-gate, DR-044).

## Rules
- **Never overwrite the project's existing files** — create or merge, never clobber. When in doubt, ask.
- **Don't rewrite existing code to English.** The language rule (DR-009) applies going forward; adopt documents the project, it does not refactor the existing codebase.
- The as-built docs describe **existing** behavior and are always marked as reconstructed — never presented as an authored plan.
- Human gate before writing (step 3); the conversation with the owner is in Spanish, the committed docs in English.
- Don't change the production state or deploy anything — adoption is documentation + wiring, not a release.
