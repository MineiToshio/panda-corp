# /pandacorp:spec

Takes the idea indicated in `$ARGUMENTS`, **creates its project** (handoff) and documents the MVP. Runs FROM the factory (panda-corp) because the project folder doesn't exist yet â€” that's why it takes the idea's name. `$ARGUMENTS` may also carry a **clarification mode** flag â€” `--ask` (always ask before generating), `--auto` (infer all, ask only the key gaps), `--infer` (never ask, state assumptions). Default by origin: a `new-idea` (`origin: owner`) â†’ `ask`; a `discover` idea (`origin: discovery`) â†’ `auto` (see Step 4, DR-095).

## Step 0 â€” Handoff (create the project)

1. Read the card `factory/ideas/<idea>.md`. It must exist and NOT be `discarded`. **If the project ALREADY exists** (the card has `project:` with a path): the handoff was already done â€” **skip all of Step 0** (don't re-scaffold) and go straight to Step 1 to **iterate** the product documents.
   - **Return-aware demand pre-check (DR-042), BEFORE scaffolding.** Read the card's `return_type`. For **monetary/mixed**: do a quick demand check first â€” are there live **paying** competitors (the market exists and pays), is the pain frequent/recurrent, and for B2B *who* pays? If the signal is absent or contradicts the idea, **STOP, tell the owner and recommend kill/pivot â€” do NOT scaffold** (killing here leaves no orphan repo). For **opportunity**: confirm the leveraged asset/audience is real and would engage. For **personal**: the owner's own need is the validation â€” skip this check. **Record the demand signal durably (audit-20 F11)**: append what you found (the evidence + verdict, with links) to the idea's CARD under `### Evidencia` â€” the project doesn't exist yet, so the card is the only durable home â€” and, once the project scaffolds, carry it into the PRD's **Demand signal** subsection (the PM's rule 9 reads it from there; a gate decision must never live only in the chat).
2. Run the project scaffold (same steps as the `scaffold` skill): create `<slug>/` as a **sibling of the factory root** (never inside; use `projects_path` from `factory/profile.md` if it is defined, by default the parent directory of the factory), copy the overlay from `${CLAUDE_PLUGIN_ROOT}/templates/shared/`, process the `.tpl` files (incl. `{{OVERLAY_VERSION}}`), create the feature-centric `docs/` skeleton (`product/`, `design/`, `frds/`, `adr/`, `analytics/`, `decision-log.md` â€” folders on demand, no empty FRD stubs; DR-049) and the `.pandacorp/` layer (inbox/changes â€” the unified queue, DR-069 â€” plus run; `comms/iteration.md` comes seeded), copy the card to `.pandacorp/idea-origin.md`, initialize git, create a private repo with `gh` if available (DR-010), and write the bidirectional links (card â†’ `status: in-pipeline` + `project:` with the path â€” same as `scaffold`; row in `factory/portfolio.md`). > From here on you work INSIDE the project folder. The other skills (`design`, `architecture`, `implement`, `release`) run there and do NOT need the name.

## Step 1 â€” Product (in the project)

3. **Deep research** (`researcher` agents in parallel), **scoped by `return_type` (DR-042)**: monetary/mixed â†’ full competitor + market + a price anchor (what live paying competitors charge) + launch-market research; opportunity â†’ the leveraged asset/reach + positioning angle; personal â†’ light (the owner's own need + feasibility), skip market sizing. In all cases: users, table-stakes vs differentiating features, feasibility (APIs/data, costs, terms). Consolidate in `docs/product/research.md` (following `${CLAUDE_PLUGIN_ROOT}/templates/docs/research-template.md`) with sources (â†’ `product/research/` folder when large). This is the PRODUCT research (at FRD level): the more complete it is, the fewer decisions are left to resolve during the build. **Also research the launch market & language** (DR-041): which market/country to launch in first and in which language(s) â€” single vs multi from the MVP â€” weighing market size/competition, customer-acquisition ease, ROI and the owner's `country`/`language` (from `factory/profile.md`). No fixed default: put all variables on the table and **recommend**; the owner approves the choice in the PRD.
4. **Clarification gate â€” ask BEFORE generating (DR-095).** Before writing the PRD/FRDs, the `product-manager` resolves intent so it never invents requirements. First it **INFERS everything it can** from the card, the research, `factory/profile.md` and the factory standards/registry, and writes those as explicit **`[ASSUMPTION: â€¦ â€” source]`** lines (a wrong default is then visible, not silent â€” the spec's primary value is forcing the decisions before the build makes them implicitly). The behaviour is set by the **mode**: `$ARGUMENTS` may carry `--ask` / `--auto` / `--infer`; otherwise the default comes from the card's `origin`:
   - **`ask`** (DEFAULT for `origin: owner` â€” a `new-idea`): ALWAYS present the understanding + assumptions + a question batch and **WAIT** for the owner before generating. The owner answers, says "asume tĂş", or adds context.
   - **`auto`** (DEFAULT for `origin: discovery` â€” a `discover` idea): infer everything possible from the card + research; **ask ONLY the key gaps** the research didn't resolve and that would change the build; if there are none, state the assumptions and proceed.
   - **`infer`**: never ask â€” state all `[ASSUMPTION]`s explicitly and generate directly; the owner corrects afterwards at the iterate gate.
   Record the resolved answers + the standing assumptions in the PRD's **Assumptions & open decisions** section and (Spanish) in `.pandacorp/comms/iteration.md`. 5. **PRD** (`product-manager` agent): `docs/product/prd.md` â€” vision, problem, users, value hypothesis, monetization (including the explicit decision **v1 with payments? yes/no**, DR-035), metrics, minimum v1 scope + backlog, an **Assumptions & open decisions** section (from the clarification gate, step 4), plus a LIVING **feature landscape** table indexing the v1 FRDs (DR-049). 6. **FRDs** (same agent): one self-contained **module folder per v1 feature** at `docs/frds/frd-NN-<slug>/frd.md`, following `${CLAUDE_PLUGIN_ROOT}/templates/docs/frd-template.md` (state frontmatter `status` + `implementation_status: PLANNED`, parent PRD), with stable IDs (`REQ-NN-MMM` requirements, `AC-NN-MMM.K` acceptance criteria) and testable EARS criteria (DR-049/DR-050).    - **The FRD is the agent's real contract â€” make it COMPLETE, not verbose (DR-095).** Each FRD MUST carry the agent-critical fields the template now requires: **data model / entities & fields touched, edge cases & error states (not only the happy path), explicit non-goals, a worked example for the trickiest acceptance criteria, and an interface/contract sketch where it exposes or consumes one**.    - **UI FRDs declare a visual contract (DR-054).** Set `ui: true|false` on every FRD. A **UI FRD** (`ui: true`) MUST set `visual_source` (the approved prototype/screen, or this feature's `mocks/`) and is **incomplete without a plan for its `fdd.md` + `mocks/`** (derived in `design` from the frozen design contract â€” `docs/design/design-tokens.json` + `DESIGN.md`). 7. Self-review: each FRD traceable to the PRD; v1 truly minimal (DR-012); criteria machine-verifiable; the agent-critical fields present (DR-095); and **no contradictions** across FRDs (a value/term/entity means the same everywhere).
