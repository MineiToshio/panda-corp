# /pandacorp:spec

Takes the idea indicated in `$ARGUMENTS`, **creates its project** (handoff) and documents the MVP. Runs FROM the factory (panda-corp) because the project folder doesn't exist yet — that's why it takes the idea's name. `$ARGUMENTS` may also carry a **clarification mode** flag — `--ask` (always ask before generating), `--auto` (infer all, ask only the key gaps), `--infer` (never ask, state assumptions). Default by origin: a `new-idea` (`origin: owner`) → `ask`; a `discover` idea (`origin: discovery`) → `auto` (see Step 4, DR-095).

## Step 0 — Handoff (create the project)

1. Read the card `factory/ideas/<idea>.md`. It must exist and NOT be `discarded`. **If the project ALREADY exists** (the card has `project:` with a path): the handoff was already done — **skip all of Step 0** (don't re-scaffold) and go straight to Step 1 to **iterate** the product documents.
   - **Return-aware demand pre-check (DR-042), BEFORE scaffolding.** Read the card's `return_type`. For **monetary/mixed**: do a quick demand check first — are there live **paying** competitors (the market exists and pays), is the pain frequent/recurrent, and for B2B *who* pays? If the signal is absent or contradicts the idea, **STOP, tell the owner and recommend kill/pivot — do NOT scaffold** (killing here leaves no orphan repo). For **opportunity**: confirm the leveraged asset/audience is real and would engage. For **personal**: the owner's own need is the validation — skip this check. **Record the demand signal durably (audit-20 F11)**: append what you found (the evidence + verdict, with links) to the idea's CARD under `### Evidencia` — the project doesn't exist yet, so the card is the only durable home — and, once the project scaffolds, carry it into the PRD's **Demand signal** subsection (the PM's rule 9 reads it from there; a gate decision must never live only in the chat).
2. **Execute the full project scaffold — delegate to `scaffold`'s Steps 2–8 in full, never a subset.** Run EVERY step of `plugin/skills/scaffold/SKILL.md`'s Steps 2 through 8 exactly as written there (the canonical project-birth recipe) — including **Step 4b** (inject the always-on `docs/rules/` files + regenerate `docs/rules/README.md`) and **Step 4c** (reserve this project's dev-port block in `factory/ports.yaml` and write `dev_port_base` into `.pandacorp/status.yaml`). Skipping 4b or 4c is a silent defect: the project would boot with no loadable engineering rules, or with no port reservation (colliding with a sibling project's dev server later). One-line summary of what this produces: `<slug>/` as a **sibling of the factory root** (never inside; `projects_path` from `factory/profile.md` if defined, else the parent directory of the factory) with the overlay copied + `.tpl`s processed, the feature-centric `docs/` skeleton + `.pandacorp/` layer + always-on rules + reserved port block, git initialized, the card copied to `.pandacorp/idea-origin.md`, a private repo via `gh` if available (DR-010), and the bidirectional links (card → `status: in-pipeline` + `project:` with the path; row in `factory/portfolio.md`). The project is born in `phase: product`, so the board shows it in the "Documentada" column (derived from the phase, not the card status).

> From here on you work INSIDE the project folder. The other skills (`design`, `architecture`, `implement`, `release`) run there and do NOT need the name.

## Step 1 — Product (in the project)

3. **Deep research** (`researcher` agents in parallel), **scoped by `return_type` (DR-042)**: monetary/mixed → full competitor + market + a price anchor (what live paying competitors charge) + launch-market research; opportunity → the leveraged asset/reach + positioning angle; personal → light (the owner's own need + feasibility), skip market sizing. In all cases: users, table-stakes vs differentiating features, feasibility (APIs/data, costs, terms). Consolidate in `docs/product/research.md` (following `${CLAUDE_PLUGIN_ROOT}/templates/docs/research-template.md`) with sources (→ `product/research/` folder when large). This is the PRODUCT research (at FRD level): the more complete it is, the fewer decisions are left to resolve during the build. **Also research the launch market & language** (DR-041): which market/country to launch in first and in which language(s) — single vs multi from the MVP — weighing market size/competition, customer-acquisition ease, ROI and the owner's `country`/`language` (from `factory/profile.md`). No fixed default: put all variables on the table and **recommend**; the owner approves the choice in the PRD.
4. **Clarification gate — ask BEFORE generating (DR-095).** Before writing the PRD/FRDs, the `product-manager` resolves intent so it never invents requirements. First it **INFERS everything it can** from the card, the research, `factory/profile.md` and the factory standards/registry, writing those as explicit **`[ASSUMPTION: … — source]`** lines. Then it surfaces the **material open questions** — only genuinely decision-changing gaps, grouped by product lens (**problema/scope · usuarios/flujos · datos & reglas · edge cases/errores · métricas de éxito · non-goals**), in **Spanish**, concise batches, **never filler** (never ask what the card/research/standards already answer). This gate is the #1 lever on doc quality (DR-095). The behaviour is set by the **mode**: `$ARGUMENTS` may carry `--ask` / `--auto` / `--infer`; otherwise the default comes from the card's `origin`:
   - **`ask`** (DEFAULT for `origin: owner` — a `new-idea`): ALWAYS present the understanding + assumptions + a question batch and **WAIT** for the owner before generating. The owner answers, says "asume tú", or adds context.
   - **`auto`** (DEFAULT for `origin: discovery` — a `discover` idea): infer everything possible from the card + research; **ask ONLY the key gaps** the research didn't resolve and that would change the build; if there are none, state the assumptions and proceed.
   - **`infer`**: never ask — state all `[ASSUMPTION]`s explicitly and generate directly; the owner corrects afterwards at the iterate gate.
   - **`[ASSUMPTION]` vs `[NEEDS CLARIFICATION]` — the load-bearing distinction.** An `[ASSUMPTION: … — source]` is a decision made and visible (a wrong default is then correctable, not silent). A material gap you could NOT resolve is marked inline **`[NEEDS CLARIFICATION: <question>]` (DR-100)** and **BLOCKS**: an FRD/PRD may NOT reach `status: ACTIVE` while one survives (the downstream readiness gate REDs on it). Resolve it or escalate; never let the build start over a buried question.
   Record the resolved answers + the standing assumptions in the PRD's **Assumptions & open decisions** section and (Spanish) in `.pandacorp/comms/iteration.md`.
5. **PRD** (`product-manager` agent): `docs/product/prd.md` — vision, problem, users, value hypothesis, monetization (including the explicit decision **v1 with payments? yes/no**, DR-035), metrics, minimum v1 scope + backlog, an **Assumptions & open decisions** section (from the clarification gate, step 4), plus a LIVING **feature landscape** table indexing the v1 FRDs (DR-049). Also record the **target platform** (`desktop | mobile | responsive`, default `responsive` — who uses it on what device) as a one-line product decision in the PRD AND set `target_platforms` in `.pandacorp/status.yaml` (DR-074). Just the decision + the field — the responsive gate enforces it downstream; do NOT thread it through design/blueprint/WO as separate enforcement.
6. **FRDs** (same agent): one self-contained **module folder per v1 feature** at `docs/frds/frd-NN-<slug>/frd.md`, following `${CLAUDE_PLUGIN_ROOT}/templates/docs/frd-template.md` (state frontmatter `status` + `implementation_status: PLANNED`, parent PRD), with stable IDs (`REQ-NN-MMM` requirements, `AC-NN-MMM.K` acceptance criteria) and testable EARS criteria (DR-049/DR-050). The PRD itself follows `templates/docs/prd-template.md` (a **minimal** v1 scope + an explicit backlog). Don't pre-stub the FRD's `blueprint.md`/`fdd.md`/`work-orders/` — those come in later phases.
   - **The FRD is the agent's real contract — make it COMPLETE, not verbose (DR-095).** Each FRD MUST carry the agent-critical fields the template requires: **data model / entities & fields touched, edge cases & error states (not only the happy path), explicit non-goals, a worked example for the trickiest acceptance criteria, and an interface/contract sketch where it exposes or consumes one**. Keep each FRD **complete on what matters, scoped, example-backed, and free of contradictions** (a term/value means the same everywhere) — padding is not completeness: over-specification overloads the agent's attention budget (lost-in-the-middle).
   - **UI FRDs declare a visual contract (DR-054).** Set `ui: true|false` on every FRD. A **UI FRD** (`ui: true`) MUST set `visual_source` (the approved prototype/screen, or this feature's `mocks/`) and is **incomplete without a plan for its `fdd.md` + `mocks/`** (derived in `design` from the frozen design contract — `docs/design/design-tokens.json` + `DESIGN.md`). Its acceptance criteria (`AC-NN-MMM.K`) MUST include **visual fidelity to the mock** (the built screen reproduces it — layout, structure, components, density — not just "uses tokens"), alongside the functional ones.
7. Self-review: each FRD traceable to the PRD; v1 truly minimal (DR-012); criteria machine-verifiable; the agent-critical fields present (DR-095); and **no contradictions** across FRDs (a value/term/entity means the same everywhere).
7c. **Fresh-set contradiction gate (DR-116) — a FRESH verifier over the whole PRD+FRD set.** The set was just generated with no supersession intent, so it MUST come out internally coherent. Spawn a dedicated verifier (a fresh agent, NOT the `product-manager` that authored it — constitution rule 4, the same independence the readiness gate demands) that reads the PRD + every FRD and asserts fail-closed that **no hard internal contradiction survives**: two acceptance criteria that cannot both hold, a value/term/entity meaning two different things across FRDs, an FRD denying a PRD scope/decision. A hard contradiction is a **BLOCKER** — reconcile it (the FRD/PRD stay `DRAFT`), never advance to silence it, exactly like a surviving `[NEEDS CLARIFICATION]` (DR-100). This is additive to step 7's self-review (which the author runs); the point is the independent read. Canonical: `factory/standards/document-consistency.md`.

7b. **Populate the root README (DR-112/DOC-3).** Now that the PRD/FRDs exist, replace the scaffold placeholder (`README.md`, still carrying the `PANDACORP-README-PLACEHOLDER` marker) with real content following `${CLAUDE_PLUGIN_ROOT}/templates/docs/readme-template.md`: the one-liner, "what it does" (from the PRD's problem/value hypothesis, plain language), and links to `docs/product/prd.md`/`docs/decision-log.md`. Remove the placeholder marker. Leave "Getting started" and "Tech stack" as a short "not yet installed" placeholder — `architecture` fills those once the stack is real. On a `spec` **iteration**, refresh this section like any other product doc; don't duplicate the PRD's prose, just point to it.
8. **Spec digest — the owner-facing Spanish summary (powers Mission Control's "Spec" tab).** Write `.pandacorp/comms/spec-resumen.md` following `${CLAUDE_PLUGIN_ROOT}/templates/docs/spec-digest-template.md` — a HIGH-LEVEL, scannable **Spanish** digest of the PRD, the research and EACH v1 FRD (one card per FRD: a one-line summary + `Overview` / `User stories` / `Reglas de negocio` / `Fuera de alcance` / `Open questions` blocks). Mission Control renders it natively in the **Spec tab** (between Propuesta and Documentos), shown only when this file exists — so the board surfaces the project's spec from the product phase. It is a SUMMARY for the owner, **not a copy** of the docs (those stay English under `docs/`); the digest is gitignored Spanish (the comms layer, per the language rule). **Keep the template's markers EXACTLY** so the tab's parser renders it rich: the `## 📋 PRD` / `## 🔬 Research` / `## 🧩 FRDs` sections, the `### subsection` labels — `Usuarios`/`Métricas de éxito`/`Alcance v1`/`Fuera del v1`/`Decisiones abiertas`/`Referentes` — and the per-FRD `### FRD-NN · Título · TAG` + `**Label:**` blocks. **Regenerate it on every `spec` iteration** so it never drifts from the PRD/FRDs.
9. **Iterate / resume (DR-032)**: if a PRD/FRDs already existed, this is an **iteration** — read `.pandacorp/comms/iteration.md` (phase `product`) and the existing docs, refine with the owner's feedback (do **not** regenerate from scratch or repeat what was already discarded) and **append** an entry per round (what was tried, what was rejected and why, what's still open). Present to the owner: value hypothesis, v1 FRDs, what was left out — and that they can review it all in Mission Control's **Spec tab** (the digest from step 8).
10. **Advance gate (DR-032)**: leave `advance_pending: true` in `.pandacorp/status.yaml` and **wait for the owner's "ok, advance"** — re-running `spec` in the meantime = keep polishing. **DO NOT write `phase: design` before.** Only when they approve: **flip the PRD + every v1 FRD's frontmatter `status: DRAFT → ACTIVE`** (the owner's ok IS the product-layer activation — audit-20 P1-3: the flip must have an owner; a doc with a surviving `[NEEDS CLARIFICATION]` — or an unresolved fresh-set contradiction from step 7c, DR-116 — may NOT flip, DR-100), then `.pandacorp/status.yaml` → `phase: design` + `advance_pending: false`, and close the `product` thread in `.pandacorp/comms/iteration.md`. Next step: open Claude Code in the project folder and run `/pandacorp:design`. (The blueprint/work-order ACTIVE flip belongs to `architecture` step 9b2, after its gates.)

## Rules
- If the research contradicts the idea (nonexistent pain, unbeatable competition), SAY SO and recommend killing/pivoting before writing the PRD.
- **Independent fact-check before the PRD:** the research that feeds the PRD gets a quick **independent** verification (a second `researcher` pass or the `product-manager`) that the load-bearing claims resolve to real sources (URLs exist and support the claim) — research that poisons the PRD cascades downstream; constitution §22 ("don't trust the model's honesty") applies to research, not only to code.
- Product/technical docs in English (committed); the conversation with the owner is in Spanish.
