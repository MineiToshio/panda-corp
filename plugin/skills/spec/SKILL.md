# /pandacorp:spec

Takes the idea indicated in `$ARGUMENTS`, **creates its project** (handoff) and documents the MVP. Runs FROM the factory (panda-corp) because the project folder doesn't exist yet — that's why it takes the idea's name.

## Step 0 — Handoff (create the project)

1. Read the card `factory/ideas/<idea>.md`. It must exist and NOT be `discarded`. **If the project ALREADY exists** (the card has `project:` with a path): the handoff was already done — **skip all of Step 0** (don't re-scaffold) and go straight to Step 1 to **iterate** the product documents.
   - **Return-aware demand pre-check (DR-042), BEFORE scaffolding.** Read the card's `return_type`. For **monetary/mixed**: do a quick demand check first — are there live **paying** competitors (the market exists and pays), is the pain frequent/recurrent, and for B2B *who* pays? If the signal is absent or contradicts the idea, **STOP, tell the owner and recommend kill/pivot — do NOT scaffold** (killing here leaves no orphan repo). For **opportunity**: confirm the leveraged asset/audience is real and would engage. For **personal**: the owner's own need is the validation — skip this check.
2. Run the project scaffold (same steps as the `scaffold` skill): create `<slug>/` as a **sibling of the factory root** (never inside; use `projects_path` from `factory/profile.md` if it is defined, by default the parent directory of the factory), copy the overlay from `${CLAUDE_PLUGIN_ROOT}/templates/shared/`, process the `.tpl` files (incl. `{{OVERLAY_VERSION}}`), create the feature-centric `docs/` skeleton (`product/`, `design/`, `frds/`, `adr/`, `analytics/`, `decision-log.md` — folders on demand, no empty FRD stubs; DR-049) and the `.pandacorp/` layer (inbox/bugs, run; `comms/iteration.md` comes seeded), copy the card to `.pandacorp/idea-origin.md`, initialize git, create a private repo with `gh` if available (DR-010), and write the bidirectional links (card → `status: in-pipeline` + `project:` with the path — same as `scaffold`; row in `factory/portfolio.md`). The project is born in `phase: product`, so the board shows it in the "Documentada" column (derived from the phase, not the card status).

> From here on you work INSIDE the project folder. The other skills (`design`, `blueprint`, `implement`, `release`) run there and do NOT need the name.

## Step 1 — Product (in the project)

3. **Deep research** (`researcher` agents in parallel), **scoped by `return_type` (DR-042)**: monetary/mixed → full competitor + market + a price anchor (what live paying competitors charge) + launch-market research; opportunity → the leveraged asset/reach + positioning angle; personal → light (the owner's own need + feasibility), skip market sizing. In all cases: users, table-stakes vs differentiating features, feasibility (APIs/data, costs, terms). Consolidate in `docs/product/research.md` with sources (→ `product/research/` folder when large). This is the PRODUCT research (at FRD level): the more complete it is, the fewer decisions are left to resolve during the build. **Also research the launch market & language** (DR-041): which market/country to launch in first and in which language(s) — single vs multi from the MVP — weighing market size/competition, customer-acquisition ease, ROI and the owner's `country`/`language` (from `factory/profile.md`). No fixed default: put all variables on the table and **recommend**; the owner approves the choice in the PRD.
4. **PRD** (`product-manager` agent): `docs/product/prd.md` — vision, problem, users, value hypothesis, monetization (including the explicit decision **v1 with payments? yes/no**, DR-035), metrics, minimum v1 scope + backlog, plus a LIVING **feature landscape** table indexing the v1 FRDs (DR-049).
5. **FRDs** (same agent): one self-contained **module folder per v1 feature** at `docs/frds/frd-NN-<slug>/frd.md`, following `${CLAUDE_PLUGIN_ROOT}/templates/docs/frd-template.md` (state frontmatter `status` + `implementation_status: PLANNED`, parent PRD), with stable IDs (`REQ-NN-MMM` requirements, `AC-NN-MMM.K` acceptance criteria) and testable EARS criteria (DR-049/DR-050). The PRD itself follows `templates/docs/prd-template.md` (a **minimal** v1 scope + an explicit backlog). Don't pre-stub the FRD's `blueprint.md`/`fdd.md`/`work-orders/` — those come in later phases.
6. Self-review: each FRD traceable to the PRD; v1 truly minimal (DR-012); criteria machine-verifiable.
7. **Iterate / resume (DR-032)**: if a PRD/FRDs already existed, this is an **iteration** — read `.pandacorp/comms/iteration.md` (phase `product`) and the existing docs, refine with the owner's feedback (do **not** regenerate from scratch or repeat what was already discarded) and **append** an entry per round (what was tried, what was rejected and why, what's still open). Present to the owner: value hypothesis, v1 FRDs, what was left out.
8. **Advance gate (DR-032)**: leave `advance_pending: true` in `.pandacorp/status.yaml` and **wait for the owner's "ok, advance"** — re-running `spec` in the meantime = keep polishing. **DO NOT write `phase: design` before.** Only when they approve: `.pandacorp/status.yaml` → `phase: design` + `advance_pending: false`, and close the `product` thread in `.pandacorp/comms/iteration.md`. Next step: open Claude Code in the project folder and run `/pandacorp:design`.

## Rules
- If the research contradicts the idea (nonexistent pain, unbeatable competition), SAY SO and recommend killing/pivoting before writing the PRD.
- **Independent fact-check before the PRD:** the research that feeds the PRD gets a quick **independent** verification (a second `researcher` pass or the `product-manager`) that the load-bearing claims resolve to real sources (URLs exist and support the claim) — research that poisons the PRD cascades downstream; constitution §22 ("don't trust the model's honesty") applies to research, not only to code.
- Product/technical docs in English (committed); the conversation with the owner is in Spanish.
