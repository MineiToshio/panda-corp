# Security audit — Mission Control

- **Date:** 2026-07-06
- **Scope:** whole project (Next.js 16 / React 19, App Router). DR-085 hardening step (construction's last step, BL-0012).
- **Threat model:** internal tool. `deploy_target: internal`, bound to `127.0.0.1`, single operator, no external host, no hosting/TLS/CI-CD (architecture §10). **Never calls Claude / no LLM or agentic component** (ADR-0001, CLAUDE.md golden rule). This context sizes every severity below.
- **Auditor:** security-auditor agent (READ-ONLY — audit + report only; fixes are the next spawn's job).
- **Relation to prior report:** re-audit over `docs/reviews/security-2026-07-05.md`. All prior findings re-verified against current source; no new Critical/High; no regression. Findings below are the standing set.

## Verdict

**No Critical or High findings. Release NOT blocked.** All findings are Low/Medium, accepted-with-reason under the internal-tool threat model. No fixes are required for release; the two dependency advisories and the two path/exec defense-in-depth items are recorded as non-blocking hardening follow-ups.

## Findings

### F-1 (Low, accepted) — `discardChangeAction` / `readBoardDoc` trust a client-supplied base path
- **Evidence:** `src/app/projects/[slug]/_actions/discard-change.ts:35` — `discardChangeAction(projectPath, id, slug)` passes the client-supplied `projectPath` straight into `discardChange(projectPath, id)` (`src/lib/changes/discard-change.ts:49-50`) which joins `<projectPath>/.pandacorp/inbox/changes/<id>.md` and writes. `src/app/board/actions/read-doc.ts:20` — `readBoardDoc(project, relPath)` → `resolveProjectPath(project)` (`src/lib/config/config.ts:34-39`, absolute paths returned verbatim). Neither validates the base against the portfolio allow-list.
- **Risk:** a Server Action is a callable POST reachable from any process on the loopback interface (and, via the browser, from a malicious page — DNS-rebinding class). A caller can name an arbitrary base directory: flip `status: discarded` on any pre-existing change item that is currently `ready`/`draft`, or read a doc-named markdown file under any directory. Both are tightly bounded — the `id`/`relPath` are traversal-guarded (`discardChange` uses `path.resolve` prefix check + `lstatSync` symlink rejection; `readDoc` validates `relPath` against the discovered allow-list), writes toggle exactly one frontmatter field on a pre-existing specifically-typed file, and reads are limited to discovered doc filenames.
- **Remediation (non-blocking, defense-in-depth):** resolve `projectPath`/`project` against the portfolio allow-list before use — the exact pattern `/api/live` already applies via `resolveProjectDir` (`src/app/api/live/route.ts:90-101`). Reject any base not in the portfolio. Not blocking: no network exposure, single operator, no privilege boundary crossed, no secret readable.

### F-2 (Low, accepted) — `execSync` with string interpolation in git-backed readers
- **Evidence:** `src/lib/achievements/read-model/seal.ts:36` (`git log -1 --format=%H -- ${SEAL_PATHSPECS.join(" ")}` — static constant pathspecs); prior report also cites `phaseTransitions.ts` / `flowSeries.ts` interpolating discovered work-order filenames and git SHAs. `src/lib/achievements/report/scalars.ts:49` is a static command (no interpolation). Contrast: `src/lib/pendingMerge/pendingMerge.ts:47` already uses the safe `execFileSync("git", [...args])` arg-array form.
- **Risk:** the interpolated values are trusted — static path constants, factory-created work-order filenames (`wo-NN-MMM-slug.md`), and git-provided 40-hex SHAs — never HTTP request input. `seal.ts` interpolates only a hardcoded readonly constant, so it has no injection surface today.
- **Remediation (non-blocking, defense-in-depth):** migrate the remaining interpolating call sites to `execFile("git", [...args])` (no shell) to make the safety structural rather than dependent on input provenance. Not blocking: inputs are trusted, server-side only, fail-soft (`try/catch`).

### F-3 (Medium, accepted — non-blocking) — 2 moderate transitive dependency advisories
- **Evidence:** `pnpm audit --prod`: `postcss <8.5.10` (GHSA-qx2v-qp2m-jg93, XSS via unescaped `</style>` in stringify) via `next>postcss`; `js-yaml <3.15.0` (GHSA-h67p-54hq-rp68, quadratic-complexity DoS via repeated merge-key aliases) via `gray-matter>js-yaml`.
- **Risk:** both **moderate**, both transitive, both unreachable in this app's usage. postcss runs only at build time on the app's own trusted CSS (no untrusted stylesheet is ever stringified). js-yaml parses only the factory's own trusted frontmatter/status YAML (no attacker-supplied YAML), so the DoS vector has no untrusted entry point.
- **Remediation (non-blocking):** clears when `next` / `gray-matter` publish patched transitive ranges; a `pnpm.overrides` bump (`postcss>=8.5.10`, `js-yaml>=3.15.0`) is the low-risk follow-up. Recorded, not blocking (no Critical/High; DR-085 blocks only on Critical/High).

## Cleared checks (no finding)

- **Secrets:** none in tracked code/config or git history (added-line scans for AWS/OpenAI/GitHub/Slack tokens, PEM headers, quoted credential assignments — all empty; "secret(s)" matches are the Achievements-Hall feature name, not credentials). `.env` / `.env.*` gitignored with `!.env.example`; only `.env.example` is tracked and it documents solely the non-secret `PANDACORP_FACTORY_ROOT` / `PANDACORP_EVENTS_FILE` paths ("needs no secrets"). No `NEXT_PUBLIC_*` secret in the client bundle.
- **Write surface (path traversal):** the bounded writes — `discardIdea`, `restoreIdea`, `setFavorite`, `discardChange`, `discardBacklogItem`, `snapshotGamificationLedger` — all guard `slug`/`id` with a `path.resolve` prefix check + `lstatSync` regular-file/symlink rejection, and rewrite exactly one frontmatter field, body preserved verbatim. Adversarial tests present (`layout.guard.test`, symlink groups).
- **Injection sinks:** no `eval` / `new Function`; `react-markdown` used without `rehype-raw` (raw HTML rendered inert). `child_process` calls are read-only git probes.
- **Auth:** no home-grown auth (none at all, deliberate — ADR-0001: 127.0.0.1-bound, single operator). Not a home-grown-auth finding.
- **Routes:** `/api/live` (SSE) and `/api/plugin-sync` are read-only, `force-dynamic` (no cache leak). `/api/live` resolves project dirs via the portfolio allow-list (never trusts a client path for `fs.watch` targets) and cleans up watchers/timers on abort. `/api/plugin-sync` shells `git` read-only via `execFileSync` (arg array — safe). Both return 200-only, no mutation.
- **SSRF:** no server-side `fetch` of a user-influenced URL anywhere; the single client `fetch` (`plugin-sync-banner.tsx:71`) hits a hardcoded same-origin constant `/api/plugin-sync`. The app only reads the local filesystem and shells local git.
- **Personal data / PII-in-logs:** no third-party personal data collected; the only owner-personal artifact is the gitignored `factory/gamification-ledger.json` (counters, no PII). No PII values in `console`/log call sites.
- **Security headers / CSP:** N/A for an internal `127.0.0.1` tool (`docs/rules/README.md` explicitly excludes `web-security.md` — "internal, not public"). `deploy_target: internal` → the CSP enforce ratchet (external-only, DR-085 3b) does not apply. Adding baseline headers via `next.config.ts` `headers()` remains an optional defense-in-depth hardening, not a finding.
- **Agentic (ASI01–ASI10):** N/A — no agent, no LLM, no tool invocation; the app never calls Claude (core value hypothesis, architecture §1).

## Method note

`gitleaks` is not installed in this environment; secret detection used manual regex patterns over tracked files and `git log --all -p` (scoped to `mission-control/` paths). Installing `gitleaks` for a future automated sweep is a low-priority tooling follow-up.
