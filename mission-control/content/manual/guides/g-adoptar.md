# Adoptar un proyecto que ya tienes

¿Tienes un proyecto que NO nació del handoff de Pandacorp —uno externo, brownfield— y quieres meterlo bajo la fábrica? Para eso está `/pandacorp:adopt`. Es el espejo brownfield de `scaffold` (greenfield): en vez de crear un proyecto vacío, **envuelve lo que ya existe**, para que desde entonces los skills `/pandacorp:*`, Mission Control y el write-gate apliquen sobre él. Se corre **DENTRO** de la carpeta del proyecto existente.

> El cuerpo paso a paso se compone en React (`GuideAdoptar`); este markdown respalda el índice del Manual.

- **Tocar un proyecto existente = gate humano.** adopt presenta el **plan de adopción** y espera tu OK antes de escribir nada (DR-045, DR-038).
- **Lee el proyecto para entenderlo (sin escribir aún).** Inspecciona stack, estructura, `git log` y `git remote`. El código da solo un **techo candidato** (una cota superior, no la fase): deploy vivo / CI a prod / URL de producción → `release`; sin deploy, adopt no emite work orders → el techo es `architecture` (nunca `implementation`). Declara el techo **y su evidencia**.
- **La fase se DERIVA de gates de calidad documental, no del código (DR-119).** Antes adopt leía la madurez del código y **estampaba** la fase; ahora corre —una vez, de abajo arriba hasta el techo— las **mismas baterías** que corren los skills de cada fase: la de **producto** (verificador de contradicciones DR-116 + sin `[NEEDS CLARIFICATION]` + campos agent-critical DR-095, con un agente fresco JUDGE) y la de **diseño** (verificación de Step 5 —axe-core + Playwright a 375/1280— más el chequeo de entregables de Step 9 y la lectura anti-omisión §1c). La **primera batería que falla topa la fase** en el escalón anterior. La readiness de arquitectura NO se chequea al adoptar (necesita work orders + Build Plan que adopt no produce) → se difiere a `/pandacorp:architecture`. Los huecos se reportan **por fase** (`## Estado derivado` / `## Huecos por fase` / `## Siguiente paso`).
- **Cierre según la fase derivada.** Fase **manual** (product/design/architecture) → escribe `phase` + `advance_pending: true` y **para** (adopt nunca estampa una fase que entra sola a un build, DR-032). App **viva** (`release`) → **mantiene `phase: release`** (un deploy vivo es un hecho observable: no se degrada el tablero) y cada batería fallida se archiva como **tarjeta de `/change`**, enrutando a `/pandacorp:design` + `/pandacorp:review-launch`. Tras tu confirmación, adopt puede **ofrecerte lanzar ese siguiente skill en la misma conversación** (un solo salto, nunca directo a `/pandacorp:implement`) — solo lo lanza si respondes que sí.
- **Inyecta el overlay SIN pisar.** Copia las plantillas (`CLAUDE.md`, `AGENTS.md`, `.pandacorp/`) pero nunca sobrescribe: ficheros nuevos se copian; si ya tienes `CLAUDE.md`, **añade** las líneas de import sin tocar tu contenido; `.pandacorp/guide.md` siempre se (re)escribe (es regenerable).
- **Reconstruye docs as-built mínimas** para que el proyecto tenga la documentación base que el resto de skills esperan.
- **Registra en la fábrica — la ficha (tablero) Y la fila de portafolio, ambas SIEMPRE.** Crea **siempre** una **ficha de idea retroactiva** `factory/ideas/<slug>.md` (`status: in-pipeline`, `project:` con la ruta que entiende `resolveProjectPath` —relativa a la raíz de la fábrica: `../<slug>` para un repo hermano, o `<slug>` si vive *dentro* de la fábrica como Mission Control— o absoluta; `return_type`, nota "proyecto externo adoptado") y la copia a `.pandacorp/idea-origin.md`. **Esa ficha es lo que hace que el proyecto aparezca en el TABLERO** (se congela como puntero; el tablero deriva su columna de la fase del `status.yaml`). Añade también la fila al `factory/portfolio.md` **en el momento de adoptar, sea cual sea la fase inferida** — igual que `scaffold`/`spec` la registran al nacer (fase `product`); un proyecto todavía en `architecture` recibe su fila ya, no cuando arranque `/pandacorp:implement`.
- **Hace el handoff a los skills normales.** Desde ahí operas el proyecto como cualquier otro: `/pandacorp:iterate`, `/pandacorp:implement`, `/pandacorp:release`…

**Enlaces:** tras adoptar, el proyecto aparece de inmediato en el **tablero** de Mission Control (por su ficha) y también en el **Portfolio** (por su fila), con su fase inferida, igual que un proyecto nacido del handoff. El tablero (fichas) y el portafolio (`portfolio.md`) son dos lecturas distintas; por eso todo proyecto necesita su ficha para salir en el tablero, y ambos resuelven la ruta del proyecto con el mismo helper (`resolveProjectPath`) para no desincronizarse.
