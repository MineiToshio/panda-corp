## 2026-06-15 — Language policy: committed = English / gitignored = Spanish
**What:** Change to the language rule for artifacts. **The state in git decides the language**: everything committed goes in ENGLISH —code, commits, **file and folder names**, and product/technical documents (PRD, FRD, blueprint, ADR, README, tests, and the project's `docs/decision-log.md`)—; everything gitignored goes in SPANISH —the communication with Pandacorp (project summary, `docs/decisions.md`, logs, activity, Mission Control feed, `docs/iteration.md`) and the owner's personal data—. `docs/status.yaml` is committed with only machine state in English (keys/enums/counters/SHAs) and its human-readable prose moves to the gitignored Spanish layer; Mission Control shows it in Spanish. The user's UI continues via i18n, Spanish by default.
**Why:** The owner wants to keep the industry standard (anyone who clones the GitHub repo sees everything in English) while still operating Pandacorp in Spanish. The previous rule ("documents in Spanish") was the opposite and would have forced on-the-fly translation in the interface; with this policy each artifact is born already in the correct language, without a translation layer. Resuming on another machine still works because what matters (FRD/PRD/work orders/`status.yaml`) is committed and in English; the Spanish layer is a regenerable local view (closing/advancing the phase before jumping to another machine protects the in-flight feedback in `iteration.md`).
**Context:** Owner's decisions in the conversation: (1) file names ALWAYS in English, in every project, regardless of the content's language; (2) the PROJECT decision log is the real history of the product → English + committed (the factory's is documented today in Spanish and will be translated in the sweep); (3) `status.yaml` committed with machine state in English + the prose to the Spanish layer. The Spanish layer is local-only: losing the old history when switching machines is acceptable as long as the durable parts are in the English docs.
**Impact:** Canonical docs updated TODAY (in Spanish, to be translated in the sweep): `CLAUDE.md` (rule #1), `factory/standards/conventions.md` (Language section), `factory/decisions/registry.yaml` (DR-009 rewritten). **PENDING** (large sweep, not yet executed): translate ~100 committed docs to English, rename ~40 files/folders with Spanish names, separate the prose from `status.yaml` into the gitignored layer, set up the convention for the local Spanish layer, and propagate to the templates (`CLAUDE.md.tpl`, `AGENTS.md.tpl`, the scaffold's `.gitignore`). Mission Control: its UI copy stays in Spanish by design (it is the owner's Spanish tool).

## 2026-06-15 — App rename: "cockpit" → "Mission Control" (and the RPG panel → "Party")
**What:** Owner's decision: the factory app is no longer called "cockpit" and is now called **Mission Control**; the per-project RPG panel previously called "Mission Control" is now **Party**. The folder `cockpit/` → `mission-control/` was renamed (with `git mv`) and the vocabulary was updated across the whole repo: paths (`cockpit/` → `mission-control/`), prose ("the cockpit" → "Mission Control"), the panel's canonical docs (`MISSION-CONTROL.md` → `PARTY.md`, `frd-06-*`), `.claude/launch.json`, and `.claude/hooks/decision-log-reminder.sh`. **Why:** By convention, "Mission Control" names the center that supervises and commands the entire operation (NASA, macOS), not a sub-panel; the name was applied backwards and was being confused with the whole app. It stands as: Pandacorp (organization) → Mission Control (control center = the app) → Party (the agent crew of each mission).
**Context:** The owner requested the rename after noticing the inconsistency; he chose "Party" (an RPG term) for the panel, consistent with the UI (the neighboring subtab was already "Work orders"). The non-visible internal identifiers of the prototype were preserved (`cockpitDataDiagram`, slug `pandacorp-cockpit`, id `c-cockpit`, functions `mc*`/`missionBody`). The historical file name `docs/proposals/05-interfaz-cockpit.md` was renamed to `05-mission-control-interface.md`.
**Impact:** `CLAUDE.md`, `README.md`, `DECISION-LOG.md`, `factory/standards/{infra,external-services}.md`, `factory/decisions/registry.yaml` (DR-031), `factory/portfolio.example.md`, `plugin/skills/{implement,decide,recommend,codify,bug,blueprint}/SKILL.md`, `plugin/scripts/emit-event.sh`, `plugin/templates/shared/docs/status.yaml.tpl`, `docs/**`, the entire `mission-control/` folder, `.claude/launch.json`, `.claude/hooks/decision-log-reminder.sh`. ## 2026-06-14 — The factory opens as a public template (depersonalization + profile + portability)
**What:** We prepared the repo to upload it to **public** GitHub so that anyone can clone it and use it as their own factory. Three structural changes: (1) **depersonalization** — the hardcoded personal identity was removed (the owner's own name, ~167 occurrences in ~110 files), replacing it with the generic role "the owner"; (2) **owner profile** — a new skill `/pandacorp:onboarding` that interviews the owner and generates `factory/profile.md` (personal, gitignored), which the rest of the factory reads to personalize itself; versioned seeds `profile.example.md` and `portfolio.example.md`; (3) **portability** — the absolute paths to the factory were removed from skills/scripts/Mission Control; the root is resolved dynamically (cwd / repo root). Personal data (profile, portfolio, idea cards) is left gitignored.
**Why:** The owner wants to share the know-how without leaking what is his, and to have the repo work as-is on any machine. A factory written around one person and with absolute paths is not reusable.
**Context:** Owner's decisions — remove the own name (don't even leave it nor parameterize it with tokens); the onboarding interview as the personalization mechanism; exclude only idea cards + `portfolio.md` (the decision logs and `docs/` are kept as the "why" of the framework); history cleaned only of the personal cards (filter-repo); no LICENSE for now.
**Impact:** `README.md` (new), `.gitignore`, `CLAUDE.md`, `factory/profile.example.md` and `factory/portfolio.example.md` (new), `plugin/skills/onboarding/` (new), `plugin/scripts/scan-ideas.sh` and several skills (relative paths), `mission-control/PLAN.md`, `factory/decisions/registry.yaml` (DR-033). ## 2026-06-14 — Documenting = canonical doc + decision log (two layers)
**What:** We refined the "document everything" discipline: each relevant change is documented in **two places**. (1) The **canonical doc** that owns the fact is updated to reflect the current truth (app behavior → the FRD; technical → blueprint/ADR; design → DESIGN/tokens; plugin → the skill/agent file; idea → its card). (2) The area's **decision log** records what changed and why, linking that doc.
**Why:** The owner clarified that a log is not enough: the documentation has to go **where it belongs**. The canonical doc answers "what is true now?" and the decision log "how did we get here and why?". The doc alone loses the why; the decision log alone leaves the doc out of date.
**Context:** Three distinct things remain and are not mixed — canonical doc (truth), decision log (history), `registry.yaml` (policy). Routing map of change→doc in `CLAUDE.md`.
**Impact:** `CLAUDE.md` (Decision Log section rewritten with the map and the two-layer rule), `DECISION-LOG.md` (How it is used).

## 2026-06-14 — Decision-log discipline: document everything, by area
**What:** We adopted a **per-area decision log** (Mission Control, plugin, ideas, factory) with an index in `DECISION-LOG.md` (root). Each relevant decision is noted with a date, with *what* and *why*. It is codified as a rule in `CLAUDE.md` + a reminder hook.
**Why:** Until now the *why* of decisions was scattered (commit messages, `registry.yaml`, or it was lost). The owner wants **everything** we decide to be recorded and visible, organized by area, so the thread can be picked up again even as time passes or the conversation is lost.
**Context:** The owner chose: scope = *decisions + the why* (not raw transcripts); organization = *one decision log per area* (not global nor hybrid); enforcement = *rule in CLAUDE.md + reminder hook*. Discarded: a single global chronological log (mixes topics) and dumping entire conversations (noise).
**Impact:** `DECISION-LOG.md`, `CLAUDE.md`, `mission-control/docs/decision-log.md`, `plugin/docs/decision-log.md`, `factory/ideas/decision-log.md`, `factory/decision-log.md`, hook in `.claude/settings.json`.
