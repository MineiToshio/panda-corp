## 2026-06-16 — Feature-centric docs structure evaluated → proposal drafted (pending owner gate)
**What:** Evaluated whether to replace the factory's current *by-type* docs model (`docs/` with flat `frds/`, one global `blueprint.md`, one flat `work-orders/`) with a *by-feature* model where each FRD is a self-contained module folder (`frds/frd-NN/` carrying `frd.md` + conditional `fdd.md` + per-FRD `blueprint.md` + `mocks/` + `work-orders/`), above a thin product layer (`product/prd.md` + a global `product/architecture.md` split from per-FRD blueprints + a global design system). Benchmarked three real systems on disk: ours, **JobLeap** (`0-jobleap-company-context`, 17-FRD/3-repo, the reference the owner liked), and **PandaTrack** (deeper nesting, rejected). Wrote the full proposal in [docs/proposals/10-feature-centric-doc-structure.md](../docs/proposals/10-feature-centric-doc-structure.md). **No standard or skill changed yet** — this is a proposal pending the owner's gate.
**Why:** The owner observed (correctly) that the flat model is comfortable for small/medium MVPs but has a hard ceiling — a single global blueprint becomes a monolith every feature must edit, and growth forces a mid-project flat→nested migration. The by-feature model gives *locality of change* (a new feature = one new folder, nothing else moves), letting a product grow from MVP to corporate platform without restructuring. Key insight from the benchmark: JobLeap separates **two layers of architecture** (global `architecture.md` vs per-FRD `blueprint.md`) which we conflate — that split is why theirs scales. It applies the factory's existing "code by feature" / locality principle (`structure.md`) to `docs/`.
**Context:** Owner chose, via the evaluation, to (a) draft the formal proposal first (not touch skills yet) and (b) adopt the **full package — structure + stable-ID traceability** (`REQ→AC→CMP/IF→WO→tests` with a `FRD > FDD > tokens > blueprint > WO` source-of-truth hierarchy), not folders alone. The proposal resolves his three questions: PRD/FRD as **siblings, not nested** (JobLeap, not PandaTrack); **two design layers** (global design system = our PDD + conditional per-FRD `fdd.md`); **mocks inside the FRD folder**. Governed by **progressive disclosure** (constant folder shape, depth scales with project size, folders on demand) so it doesn't crush small MVPs. On approval: rewrite `factory/standards/structure.md`, add the source-of-truth hierarchy to `documentation.md`, a DR + registry default, a MAJOR plugin bump (6 skills incl. `implement`'s WO-glob → per-FRD walk, + Mission Control + Manual/DR-046), pilot on one project, migrate the rest via `/pandacorp:upgrade`.
**Impact:** `docs/proposals/10-feature-centric-doc-structure.md` (new). Pending owner gate before any change to `factory/standards/`, the plugin, or Mission Control. Owner resolved (2026-06-16): **ID scheme = numeric folder-derived** (`frd-05` → `REQ-05-001`; JobLeap 2-letter code rejected as cryptic); **pilot deferred** (Mission Control is built first regardless, launch-on-new-structure decided later — document fully, don't force a pilot). Clarified for the owner that small vs large are *not two structures* — one shape, two fill levels; growth is additive (a feature = a new `frds/frd-NN/` folder, nothing moves), the only rename ever is `product/prd.md` → `product/prds/`. Further owner refinements (2026-06-16): a heavy per-FRD blueprint splits into a **`blueprints/` folder by coherent concern (NOT "pillars")**, not flat pillar files; the **PDD = the `docs/design/` folder** (there is no separate PDD file — the global design system *is* the PDD); the standard is an **extensible base skeleton** (projects may add their own folders like `runbooks/`, `db/`; only the contract parts — `frds/frd-NN/`, `product/`, `design/`, the ID convention — stay fixed). Still open: analytics global-vs-per-FRD, single PRD vs `prds/` folder.

## 2026-06-15 — Self-learning loop: cross-project memory + harvest/retrieve/eval-gate (DR-047, alternative B)
**What:** Defined the factory's self-learning system and shipped its foundation (Phase 0). New policy **DR-047**: the factory accumulates durable engineering lessons in a new **`factory/memory/`** store (committed English know-how, like `factory/standards/`; typed: problem-solution | library-verdict | pattern | gotcha | anti-pattern), **harvested** at the existing ~15 capture points and **retrieved** by agents before design/blueprint/implement. Autonomy is **tiered by risk (alternative B, semi-autonomous)**: low-risk (store a lesson) auto-applies after a light eval-gate as a *candidate*; medium-risk (a SHOULD standard) auto only if it ships with its verifier; high-risk (a MUST standard, a skill/agent, a DR, any deletion, anything touching a project) always goes through the owner. `learn` becomes the *apply* stage of a closed loop (harvest → store → retrieve → propose → gate → promote → prune), no longer a manual point-tool. Phase 0 shipped: `factory/memory/` (README + lesson template + first seed lesson), DR-047, and the proposal `docs/proposals/09-self-learning-factory.md`.
**Why:** The owner wants the factory to get smarter and faster with use — documenting solved problems, recording which libraries worked/failed, growing skills/rules, self-suggesting improvements in Mission Control, and self-questioning/pruning what stops working ("solito solito"). A deep-research pass (25 claims, 24 at 3-0, primary sources) showed the proven path is **gradient-free externalized memory** (A-MEM, Letta/MemGPT, Voyager, ExpeL, Self-Refine) + **eval-gated self-modification** (DGM, AlphaEvolve), and that the governance edge (pruning stale lessons, anti-poisoning, a library-verdict memory) is **unsolved in the literature** — so an eval-gate + human gate is the responsible design. The factory already had the three knowledge planes (registry=policy, decision-logs=history, standards=know-how) and the capture points, but no cross-project memory and no closed loop; `learn` was 100% manual.
**Context:** The owner chose **alternative B** (semi-autonomous, eval-gated) over A (everything human-gated) and C (full DGM-style self-evolution, deferred as a north star). Key design choices: the store is committed English engineering know-how (not personal data, DR-033) so it propagates like a standard; harvested lessons are *candidates*, not confirmed truth (anti-poisoning, grounded in the ExpeL finding that reflections-on-reflections poison insight extraction); deletion is deprecation, never removal (DR-011/DR-007); the eval-gate reuses existing machinery (`learn` already requires a verifier, `skill-creator` already benchmarks skills). DR-047 operationalizes the long-aspirational DR-030 (recurring pattern → standard) by giving it a substrate and a loop. The autonomy/eval/human-gate shape is the constitution itself (§3 adversarial verification, §7 each intervention reduces future ones) applied to the factory's own evolution. The "Hermes Agent" the owner referenced is real (NousResearch: Memory/Skills/Crons pillars, self-documenting) but its claims are secondary/unverified — used as directional inspiration only.
**Impact:** `factory/decisions/registry.yaml` (DR-047), `factory/memory/` (new: `README.md`, `_lesson-template.md`, `LESSON-0001-*.md`), `docs/proposals/09-self-learning-factory.md` (new), `CLAUDE.md` ("What's here" gains a `factory/memory/` line). Pending (later phases, NOT this change): the harvester agent ("cronista") + capture hooks (Phase 1), retrieval injection into the agents (Phase 2), Mission Control **FRD-17** "Propuestas" inbox (Phase 3/5), the prune job (Phase 4). Mission Control's Manual decision-rules reference (FRD-08) will pick up DR-047 by derivation (DR-046); the prototype's hand-reconciled `CONFIG.rules` + a self-learning Concept page are pending until the system is built.

## 2026-06-15 — Skill write-gate + brownfield adoption (factory↔project boundary hardened)
**What:** Two new policies + a guard subsystem to harden the boundary between the factory and a project. (1) **Write-gate (DR-044)**: changes to a Pandacorp product project that touch app behavior, a canonical doc (PRD/FRD/blueprint/ADR/DESIGN) or state (`status.yaml`/work-orders) flow through the `/pandacorp:*` skills, not ad-hoc free-chat edits — with a PRAGMATIC frontier (reading/debugging and micro non-product edits stay exempt) and a ROUTER that classifies the owner's request and AUTO-INVOKES the right skill, announcing it (entry only; the skill's internal human-gates stay intact). (2) **Brownfield adoption (DR-045)**: the factory stays greenfield-first but now ACCEPTS external projects via the new `/pandacorp:adopt`; every mutating skill gained a PREFLIGHT that stops on a non-Pandacorp folder and points to adopt/spec instead of failing silently or hallucinating.
**Why:** Analysis of "what happens if I run a Pandacorp skill on a project that isn't in the factory" found two gaps: (a) no skill verified "is this a Pandacorp project?" — `implement` hit an empty queue, `iterate` invented PRD/FRDs and queued work-orders into a non-existent queue; (b) nothing stopped free-chat changes from bypassing the documented process → docs/state drift. The owner wanted changes to a Pandacorp project to always flow through the skills, plus a supported path for projects built outside the factory.
**Context:** Owner's design choices — frontier = pragmatic (not "everything by skill"); router = auto-invoke and announce (not propose-and-confirm); enforcement = instruction + a non-blocking warning hook (not a hard block, which would false-positive against the skills themselves). The Pandacorp marker is `docs/status.yaml` + "Origin — Pandacorp" in `CLAUDE.md` (reuses the scope check already in `block-dangerous.sh`; a shared `is-pandacorp-project.sh` formalizes it, distinguishing a product project from the factory, which has no `status.yaml`). Auto-invoke covers entering the skill only; the production (DR-004), money (DR-005), delete (DR-007) and external-comms (DR-008) gates still ask.
**Impact:** `factory/decisions/registry.yaml` (DR-044, DR-045). Plugin (same date, `plugin/docs/decision-log.md`, v5.1.0): `CLAUDE.md.tpl`/`AGENTS.md.tpl` (write-gate section), preflight in `implement`/`iterate`/`release`/`bug`/`new-version`, new `scripts/is-pandacorp-project.sh` + `scripts/warn-adhoc-write.sh` + the `Write|Edit` hook in `hooks/hooks.json`, new skill `plugin/skills/adopt/`. ## 2026-06-15 — Language policy: committed = English / gitignored = Spanish

The Spanish layer is local-only: losing the old history when switching machines is acceptable as long as the durable parts are in the English docs.
**Impact:** Canonical docs updated TODAY (in Spanish, to be translated in the sweep): `CLAUDE.md` (rule #1), `factory/standards/conventions.md` (Language section), `factory/decisions/registry.yaml` (DR-009 rewritten). **PENDING** (large sweep, not yet executed): translate ~100 committed docs to English, rename ~40 files/folders with Spanish names, separate the prose from `status.yaml` into the gitignored layer, set up the convention for the local Spanish layer, and propagate to the templates (`CLAUDE.md.tpl`, `AGENTS.md.tpl`, the scaffold's `.gitignore`). ## 2026-06-14 — Documenting = canonical doc + decision log (two layers)
**What:** We refined the "document everything" discipline: each relevant change is documented in **two places**. (1) The **canonical doc** that owns the fact is updated to reflect the current truth (app behavior → the FRD; technical → blueprint/ADR; design → DESIGN/tokens; plugin → the skill/agent file; idea → its card). (2) The area's **decision log** records what changed and why, linking that doc.
**Why:** The owner clarified that a log is not enough: the documentation has to go **where it belongs**. The canonical doc answers "what is true now?" and the decision log "how did we get here and why?". The doc alone loses the why; the decision log alone leaves the doc out of date.
**Context:** Three distinct things remain and are not mixed — canonical doc (truth), decision log (history), `registry.yaml` (policy). Routing map of change→doc in `CLAUDE.md`.
**Impact:** `CLAUDE.md` (Decision Log section rewritten with the map and the two-layer rule), `DECISION-LOG.md` (How it is used).

## 2026-06-14 — Decision-log discipline: document everything, by area
**What:** We adopted a **per-area decision log** (Mission Control, plugin, ideas, factory) with an index in `DECISION-LOG.md` (root). Each relevant decision is noted with a date, with *what* and *why*. It is codified as a rule in `CLAUDE.md` + a reminder hook.
