## 2026-06-17 — Source of truth for build state: the work order; the FRD is a derived rollup
**What:** Codified in `build-orchestration.md` §1 + `registry.yaml` DR-050: the **work order** `implementation_status` is the atomic source of truth; an FRD's/blueprint's `implementation_status` is a **DERIVED rollup** of its work orders (`VERIFIED` iff all; else the worst non-`VERIFIED`: `BLOCKED` > `PLANNED` > `IN_PROGRESS` > `IN_REVIEW`), never an independent state. **On any FRD↔WO discrepancy, the work orders win** — the FRD field is a cache, trustworthy on its own only at `VERIFIED`. "Done" = `VERIFIED`, measured on work orders. The engine keeps the cache honest (re-derives on plan + after each gate); exact readers (Mission Control) derive from the WOs and prefer a breakdown ("7/11 verified") over a single rolled-up word.
**Why:** A stale `frd.md` (FRD-06 said `PLANNED` while its 11 WOs were `IN_REVIEW`) made even the agent misreport whether a feature was done. Needed one unambiguous rule for "what determines done", since docs and WOs can drift (a crash mid-build, an interrupted run, a manual edit).
**Impact:** `factory/standards/build-orchestration.md` (§1), `factory/decisions/registry.yaml` (DR-050). The engine that enforces it: plugin v8.10.0 (`plugin/docs/decision-log.md`).

## 2026-06-17 — The overnight brake is maxAgents (counted in the engine), not maxSpend
**What:** `build-orchestration.md` §6: the real run-stopping budget ceiling is now **`maxAgents`** (a hard cap on the subagents the engine spawns), not `maxSpend` (`budget.spent()`). `budget.spent()` under-counts the heavy subagent work and isn't enforced if the supervisor dies — an MC run hit ~6M tokens against a 2M `maxSpend` after an app restart orphaned it. `maxAgents` is counted inside the engine, so it brakes reliably and survives a dead supervisor; `maxSpend` is demoted to a secondary ceiling.
**Why:** Owner flagged that `maxSpend` could never reliably fire — the orchestrator/verifier writes little, and the engine wasn't capping the real work the subagents do. A spawn-count proxy maps directly to spend AND to the engine's own control flow, so it's both honest and enforceable.
**Impact:** `factory/standards/build-orchestration.md` (§6). Engine + skill + history: plugin v8.9.0 (`plugin/docs/decision-log.md`).

## 2026-06-17 — Build speed & scale: powerful default, focused gates, trimmed cold start (plugin v8.7.0)
**What:** Owner flagged mid-run that the build was slow and the per-FRD gates re-ran the whole test suite (3700+ and growing) — not scalable. Three fixes: `powerful` is the default mode now (more parallelism); the cold start is trimmed (skip baseline `verify.sh` when `HEAD==last_green`; planner reads WO frontmatter only, not full bodies); and the per-FRD gate is **focused** — `verify.sh --since <last_green>` runs biome+tsc globally + only the vitest tests affected since the last green, with the full suite once at close-out. Standards updated: `build-orchestration.md` §5 (focused gate) + `architect.md`/DR-019 (verify.sh gains a `--since` mode).
**Why:** A gate that re-runs every test on every FRD is O(total tests) and collapses past mid-project; focused gates are O(touched). Owner-approved trade-off: an unrelated-feature regression could slip to close-out, mitigated by global tsc/biome each gate + full suite at close-out. Speed is now a first-class concern alongside the rigor we already had.
**Impact:** `factory/standards/build-orchestration.md` (§5). Plugin side (engine + skill + architect + bump 8.7.0, OVERLAY 8.7.0): `plugin/docs/decision-log.md` (same date). MC adopts it next run (engine + verify.sh `--since` copied when the current run ends).

## 2026-06-16 — Worktrees for build parallelism: evaluated, NOT adopted
**What:** Evaluated the owner's idea of running build subagents in isolated git worktrees (`isolation:'worktree'`) for more parallelism / to dodge same-file collisions. Confirmed the harness mechanism: **no automatic merge** — a worktree's changes stay on its own branch and the orchestrator must merge them with raw git; conflicts are manual; `node_modules` is NOT shared (each worktree needs its own `pnpm install`, 30–90s on Next.js); plus open harness bugs (stale-branch reuse on id collision, worktrees forking from `main` not the caller's HEAD). **Decision: do NOT use worktrees in the build engine.** Reasons: (1) it doesn't solve the same-file case — 4 agents writing one file in 4 worktrees = a merge conflict the script can't safely auto-resolve; merging the WOs is strictly better; (2) per-worktree `pnpm install` eats the parallelism gain; (3) script-resolved merges in an unattended overnight build are exactly the silent-breakage risk to avoid; (4) the engine's intra-FRD waves over disjoint files (the new no-same-file-siblings rule) already give real parallelism. **Reserved for:** isolated snapshot-testing (1 read-only worktree on `last_green_sha`, already in the implement skill) and, as a future measured experiment, building fully-independent FRDs in parallel (one worktree per FRD) IF speed ever becomes the measured bottleneck.
**Why:** Owner asked to evaluate worktrees to finish builds faster. Honest verdict: the cost (install per worktree + manual merge + harness bugs) outweighs the benefit for an autonomous JS build, and the problem that motivated it (same-file collisions) is better solved by coarser WOs + the no-same-file rule (v8.4.0). Recorded so a future agent doesn't re-litigate it.
**Impact:** No code change — decision only. Related: `factory/standards/build-orchestration.md` §2 (the real fix), plugin v8.4.0.

## 2026-06-16 — Build freno: health+budget, repair before block, BLOCKED with a reason (DR-050 refined, plugin v8.2.0)

A future Pandacorp Mission Control mobile app is the owner's preferred notification channel (noted, not built).
**Impact:** Canonical: `factory/standards/build-orchestration.md` (§6 "How a run stops"), `factory/decisions/registry.yaml` (DR-050 refined). **Plugin v8.2.0 (MINOR)** — see `plugin/docs/decision-log.md` (same date): engine `maxFrds`→Infinity default + `attemptRepair` + health breaker + `blocked_reason` + desktop-only NOTIFY; the `implement` skill aligned; `work-order-template.md` gains `blocked_reason`; `OVERLAY_VERSION` → 8.2.0. ## 2026-06-15 — Language policy: committed = English / gitignored = Spanish

The Spanish layer is local-only: losing the old history when switching machines is acceptable as long as the durable parts are in the English docs.
**Impact:** Canonical docs updated TODAY (in Spanish, to be translated in the sweep): `CLAUDE.md` (rule #1), `factory/standards/conventions.md` (Language section), `factory/decisions/registry.yaml` (DR-009 rewritten). **PENDING** (large sweep, not yet executed): translate ~100 committed docs to English, rename ~40 files/folders with Spanish names, separate the prose from `status.yaml` into the gitignored layer, set up the convention for the local Spanish layer, and propagate to the templates (`CLAUDE.md.tpl`, `AGENTS.md.tpl`, the scaffold's `.gitignore`). ## 2026-06-14 — Documenting = canonical doc + decision log (two layers)
**What:** We refined the "document everything" discipline: each relevant change is documented in **two places**. (1) The **canonical doc** that owns the fact is updated to reflect the current truth (app behavior → the FRD; technical → blueprint/ADR; design → DESIGN/tokens; plugin → the skill/agent file; idea → its card). (2) The area's **decision log** records what changed and why, linking that doc.
**Why:** The owner clarified that a log is not enough: the documentation has to go **where it belongs**. The canonical doc answers "what is true now?" and the decision log "how did we get here and why?". The doc alone loses the why; the decision log alone leaves the doc out of date.
**Context:** Three distinct things remain and are not mixed — canonical doc (truth), decision log (history), `registry.yaml` (policy). Routing map of change→doc in `CLAUDE.md`.
**Impact:** `CLAUDE.md` (Decision Log section rewritten with the map and the two-layer rule), `DECISION-LOG.md` (How it is used).

## 2026-06-14 — Decision-log discipline: document everything, by area
**What:** We adopted a **per-area decision log** (Mission Control, plugin, ideas, factory) with an index in `DECISION-LOG.md` (root). Each relevant decision is noted with a date, with *what* and *why*. It is codified as a rule in `CLAUDE.md` + a reminder hook.
