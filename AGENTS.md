# Pandacorp ‚Äî 100% AI software factory

This repo is **the factory**: the company's know-how. Product code NEVER lives here.

This file is the **canonical instruction set for ANY coding agent** (Claude Code, OpenAI Codex, Cursor, OpenCode‚Ä¶). Claude Code reads it through `CLAUDE.md` (which imports it and adds the Claude-specific layer: plugin lifecycle, build engine, hooks). Every other runtime reads this file natively. If you are not Claude Code, also read the **Runtime portability** section at the end before doing factory work.

## What's here

- `factory/constitution.md` ‚Äî non-negotiable principles and standards. **Read it before any factory work.**
- `factory/standards/` ‚Äî engineering standards injected into every project (conventions, structure, patterns, quality, **build orchestration** ‚Äî how a build is planned/run: states, coarse work orders, the Build Plan, how a run stops; DR-050 ‚Äî, recommended stack, external services/accounts/secrets, **agent portability** ‚Äî how any runtime operates the factory; DR-113). The stack is a suggestion; it is chosen and approved in the blueprint. Default services, the account model, secrets (SOPS+age) and payments (Polar) live in `external-services.md`.
- `factory/ideas/` ‚Äî idea base (one .md card per idea, frontmatter with `status`). **Cards are personal (gitignored)**; `_idea-template.md` is versioned.
- `factory/profile.md` ‚Äî the owner's profile. **Personal (gitignored)**; generated by the `onboarding` skill. The factory reads it to personalize itself.
- `factory/portfolio.md` ‚Äî index of created projects and their state (pointers, not content). **Personal (gitignored)**.
- `factory/decisions/registry.yaml` ‚Äî decision registry with pre-approved defaults (**policy**).
- `factory/memory/` ‚Äî cross-project engineering memory: durable lessons harvested from projects. Committed know-how (NOT personal data); policy DR-047. **Memory is durable knowledge, NOT an action queue** ‚Äî a defect/change to the factory's own tooling goes to `factory/backlog/` (DR-103).
- `factory/backlog/` ‚Äî the factory's **actionable work queue** for its OWN tooling: closeable `BL-*` items (`open ‚Üí doing ‚Üí done`). Policy DR-103. Worked via the `implement-backlog` skill.
- `plugin/` ‚Äî the `pandacorp` capability pack: **skills** (`plugin/skills/<slug>/SKILL.md`, Agent Skills open standard ‚Äî also exposed at `.agents/skills/`), **agents** (`plugin/agents/*.md`, canonical; Codex mirrors generated at `.codex/agents/*.toml`), hooks, templates. Under Claude Code it installs as a plugin; see `CLAUDE.md` for that lifecycle.
- `docs/` ‚Äî vision, research and proposals.
- `DECISION-LOG.md` ‚Äî index of decision logs (one per area). See **Decision log** below.
- `ideas.base` ‚Äî Obsidian kanban view over the idea base.
- `mission-control/` ‚Äî the Next.js dashboard app (a product project living in-repo).

## How the factory is operated

Every lifecycle action is a **skill**: a `SKILL.md` at `plugin/skills/<slug>/SKILL.md`. Under Claude Code they are invoked as `/pandacorp:<slug>`; under other runtimes they are discovered via `.agents/skills/` or read directly (see Runtime portability).

| Action | Skill slug |
|---|---|
| Configure the factory (first boot) | `onboarding` |
| Explore/clarify a fuzzy idea (discovery) | `explore` |
| Capture/crystallize an idea | `new-idea` |
| Search the internet for monetizable pains | `discover` |
| Idea ranking/recommendation | `recommend` |
| Create the project + document an idea's MVP (handoff) | `spec` |
| Adopt an EXISTING external project (brownfield) | `adopt` |
| Sync the portfolio | `sync-portfolio` |

`spec` is the **handoff**: run FROM the factory with the idea's name; creates the project folder/repo and documents the MVP (research + PRD + FRDs). The other phases run **inside the project folder**: `design` (mockups + design system), `architecture` (blueprint + work orders), `implement` (the build ‚Äî **its LAST step is the hardening pass**: security audit + quality close-out + telemetry/metrics verification, moved here from `release`, DR-085; see Runtime portability for what the build means outside Claude Code), `release` (deploy/launch the product, **internal or external** ‚Äî the `deploy_target` field ‚Äî with the return-aware launch plan + the human gate for an external production deploy; sets `phase: release`, the launched/terminal phase; the audit is NOT here anymore ‚Äî it closes construction), `review-launch` (post-launch: real metrics vs the PRD's value hypothesis ‚Üí kill/hold/double-down verdict, DR-043; on demand or as a recurring job), `change` (**the single owner front door for ANY change** ‚Äî it classifies feature/change vs bug, assigns a class of service (urgent vs normal) and files it in the project's `.pandacorp/inbox/changes/` queue for the build to drain at its next safe point; DR-069 ‚Äî never route the owner to `iterate`/`bug` directly), `decide` (answer a pending decision in `.pandacorp/inbox/decisions.md`), `iterate` (feature/change engine behind `change`), `sync` (reverse flow: reconcile docs from hand-made code changes, exhaustive cascade + intent gate, DR-081), `upgrade` (re-sync a project's `.pandacorp/` overlay to the latest factory version, DR-048 ‚Äî **also runs automatically from a skill's preflight when it detects drift**; the owner can run it directly to force a re-sync).

Internal engines (marked `user-invocable: false`; not offered to the owner directly, but readable/executable by an agent when a flow calls for them): `bug` (the bug-case engine behind `change`), `new-version` (large milestone with its own mini-PRD), `scaffold` and `work-orders` (steps normally invoked by `spec`/`architecture`), `iterate`.

**Iterate without advancing (DR-032).** No manual phase (`explore`, `new-idea`, `spec`, `design`, `architecture`) auto-advances a column: it produces its output, sets `advance_pending: true` and waits for the owner's "ok, advance". Re-running the same phase = keep polishing (it refines; it doesn't regenerate or repeat what was discarded). The back-and-forth essence persists in the project's `.pandacorp/comms/iteration.md` (ideas without a project: `factory/ideas/_drafts/<slug>.md`). It does NOT apply to `implement` (autonomous, already resumable) and does not change the production gate.

## Changing the factory itself ‚Äî route by plane (DR-103)

| What the owner wants | Plane ¬∑ home | How |
|---|---|---|
| A durable rule/standard, a decision default, or a new skill/capability | know-how ‚Üí `factory/standards/` ¬∑ `factory/decisions/registry.yaml` ¬∑ `plugin/` | `learn` skill |
| An actionable defect/change to the factory's own tooling | actionable ‚Üí `factory/backlog/` (`BL-*`) | file a `BL-*` item; `learn` step 0 also routes here |
| A durable lesson (verdict, gotcha, pattern) | knowledge ‚Üí `factory/memory/` | jot to `_inbox.md` ‚Üí refine via `memory` skill |
| A research/strategy exploration | narrative ‚Üí `docs/proposals/` | write a proposal |

A change to a **product project** (Mission Control, siblings) never uses any of these: it goes to THAT project's `.pandacorp/inbox/changes/` via the `change` skill.

## Decision log ‚Äî document everything

Documenting a change is **two things, always**: (1) **update the canonical doc** (the current truth) and (2) **record the decision in the area's decision log** (the history: date, what, why, linking the doc touched). The canonical doc answers *"what is true now?"*; the decision log, *"how did we get here and why?"* ‚Äî you need both.

**Canonical doc ‚Äî where each type of change is updated:**

| Change | Owning doc |
|---|---|
| App behavior/feature (what it does, acceptance criteria) | the corresponding **FRD** (`docs/frds/`); new feature ‚Üí new FRD |
| Product scope/goal, metrics | the **PRD** (`docs/product/prd.md`) |
| Architecture, stack, data model, technical decision | the **blueprint** + an **ADR** |
| Visual design, tokens, components | **DESIGN.md** / design tokens |
| Plugin skill/agent/hook | the **file in `plugin/`** |
| An idea (content/state) | its **card** `factory/ideas/<slug>.md` |
| Engineering convention/standard | `factory/standards/` |
| Build orchestration (build state, WO granularity, Build Plan, how a run stops) | `factory/standards/build-orchestration.md` (DR-050) |
| Recurring rule with a default | `factory/decisions/registry.yaml` |

**Decision logs (one per area):** Mission Control ‚Üí `mission-control/docs/decision-log.md`; plugin ‚Üí `plugin/docs/decision-log.md`; ideas ‚Üí `factory/ideas/decision-log.md`; factory ‚Üí `factory/decision-log.md`; index in `DECISION-LOG.md`. Most recent on top; don't record trivia obvious from the commit. **Three distinct things:** canonical doc = current truth; decision log = history; registry = policy. For **product projects** this discipline is injected as a propagable standard (`factory/standards/documentation.md`): every project is born with its `docs/decision-log.md` and the rule in its `CLAUDE.md`/`AGENTS.md`. Each project's `status.yaml` is written by the skills/CI, never by hand.

**Mission Control's Manual reflects the factory (DR-046).** Every change to the factory's operable surface (skill, agent, flow, gate, rule, standard) must also surface in Mission Control's Manual: Reference catalogs are derived from the canonical sources automatically; hand-authored Guides/Concepts are updated in the same change.

## Rules

1. **Language ‚Äî the git state decides (DR-009).** Committed ‚Üí English (code, commits, file names, technical docs). Gitignored ‚Üí Spanish (owner communication, personal data). `.pandacorp/status.yaml` is committed machine-state in English. **The interaction with the owner is ALWAYS in Spanish** ‚Äî everything any agent says to the owner in chat, under any runtime, whatever the artifact language.
2. Every recurring decision is resolved by consulting `factory/decisions/registry.yaml`. Not covered ‚Üí escalate to the owner ONCE and codify the answer as a new rule.
3. **Human gates (the owner):** idea selection, design choice, release to production, spending money, external communications, deleting data. The idea board is read-only for humans: states are written by the skills.
4. Agents never check off their own checks: verification is done by scripts/CI.
5. **One source of truth per state, with an explicit bridge.** A card's `status` covers pre-project + terminal (`discovered ‚Üí recommended ‚Üí in-pipeline ‚Üí shipped | discarded`); once `in-pipeline` the card freezes as a pointer and the project's state lives ONLY in its `.pandacorp/status.yaml` (`phase`). The board derives a card's **column** from both axes (pre-project ‚Üí card status; `in-pipeline` ‚Üí project `phase`: product‚ÜíDocumentada, design‚ÜíDise√±o, architecture‚ÜíArquitectura, implementation‚Üíbuilding, release‚Üíshipped ‚Äî DR-085: `release` is the launched/terminal phase; the old `operation` phase no longer exists). It never expects `design`/`architecture`/`building` to be a card status. The portfolio only keeps pointers and summaries.
6. Document everything per the **Decision log** section ‚Äî before closing the turn.
7. **Framework vs owner's data (DR-033).** Never mix: the framework (versioned: `plugin/`, `factory/constitution.md`, `factory/standards/`, registry, Mission Control, templates and `*.example.md` seeds); the owner's data (gitignored: `factory/profile.md` ‚Äî generated by `onboarding`, the **single source** of user configuration: name, goals, interests, assets, GitHub account, project path ‚Äî, `factory/portfolio.md`, `factory/ideas/*.md`; secrets never in the repo, DR-037); product projects (sibling folders/repos, own docs in their `docs/`).
8. **Capture lessons as you work (DR-047) ‚Äî IN THE SAME TURN.** The moment a capture event happens (owner correction, fix after failure, library verdict, non-obvious gotcha), jot one line to `factory/memory/_inbox.md` (tag `(owner-stated)` or `(agent-inferred)`). Never defer to session end.
9. **Keep the owner's profile alive (DR-053).** When the owner reveals something durable about themselves (an interest, a taste/dislike, a goal, an asset/lever, how they like to work), update `factory/profile.md` (personal, gitignored, Spanish): read it first, add only what's new, never duplicate. In a product project the same rule writes to the factory's profile via the path stamped in its `.pandacorp/guide.md`. Don't ask permission.
10. **Interaction style (DR-110/CONV-11).** Owner-facing replies: mark where the real answer begins after progress narration (`---` + short bold label); concise, scannable, high-level by default; storytelling/analogies over jargon; visualize complex flows; red-team proposals that require judgment ‚Äî never rubber-stamp; stay human, disagree substantively.
11. **Subagent model selection (DR-111/CONV-12, generalized by DR-113).** When delegating without an explicit model, compute the tier from the SUBTASK's complexity ‚Äî never inherit the parent's tier: **MECH** (mechanical/zero-judgment), **STANDARD** (default floor for real work), **JUDGE** (genuine judgment/adversarial). Per-runtime mapping in `factory/standards/agent-portability.md` (Claude: haiku/sonnet/opus; Codex: gpt-5.4-mini/gpt-5.5 (medium effort)/gpt-5.5 (high effort)). Escalate upward, never downward. The most expensive tier (Fable-class) is NEVER chosen automatically.
12. **Evidence before assertion (CONV-13).** Every factual claim to the owner about the state of code/data/systems anchors to something observed with a tool in the CURRENT session. Recorded state ‚Äî a flag, a cached count, a subagent's report, a prior audit ‚Äî is a claim to re-verify against live signals, not evidence. What isn't verified is said as unverified, never asserted. Full rule: `factory/standards/conventions.md` (CONV-13).

## Runtime portability (DR-113)

Full operative standard: `factory/standards/agent-portability.md`. The essentials any non-Claude runtime must know:

- **Skill invocation protocol.** When the owner asks for a phase (`/pandacorp:x`, "ejecuta la fase x", `$x`): read `plugin/skills/<x>/SKILL.md` and follow it, applying the tool translation table below. Human gates, language rules and documentation discipline are runtime-independent ‚Äî they NEVER degrade.
- **Tool translation** (Claude tool referenced in a skill ‚Üí what to do elsewhere): `Agent`/`Task` ‚Üí an explicit subagent from the **active runtime only** (Codex: the generated `.codex/agents/` incl. `tier-mech`/`tier-standard`/`tier-judge`), else do the work inline sequentially ¬∑ `Workflow` (Dynamic Workflows) ‚Üí **no equivalent**: never imitate or take over its live run; use the runtime's certified executor only after its portability gates pass ¬∑ `Monitor`/`ScheduleWakeup`/`PushNotification` ‚Üí none until a runtime-specific equivalent is certified: stay attended and report in chat ¬∑ `AskUserQuestion` ‚Üí ask in chat ¬∑ `EnterWorktree`/`ExitWorktree` ‚Üí `git worktree` by hand ¬∑ `DesignSync` (Claude Design canvas) ‚Üí the documented HTML-mockup fallback (DR-058 Plan B) ¬∑ `$ARGUMENTS` in a skill body ‚Üí the free text of the owner's request.
- **Build safety (`implement`).** Codex may write normal project build state only through PORT-5's exact promoted `EXPERIMENTAL/attended_foreground/targeted-only` profile: one FRD or one ready change, foreground, cumulative duration `<=7200`, zero automatic restarts, ending in `implementation`. Bare/global, multiple-FRD, background/unattended/overnight, hardening/release and cross-runtime switching remain denied while their R10/R11 gates are pending; every other non-Claude runtime remains read/review-only until separately promoted. Runtime execution is strict-local, never live takeover or simultaneous build. Full contract: `factory/standards/agent-portability.md` PORT-5.
- **Enforcement note.** Claude Code enforces some rules with hooks (dangerous-command gate, verify-before-stop, lesson-capture backstop). Other runtimes don't get that enforcement ‚Äî the RULES still bind you as instructions: never run destructive commands without the owner's gate, never declare work done with a red `verify.sh`, capture lessons in the same turn.
- **PROTECTED STATE PATHS ‚Äî never delete, recreate or bulk-clean (BL-0035 incident, 2026-07-04).** Any `.pandacorp/` directory (in any project) and the factory's `factory/{ideas,memory,profile.md,portfolio.md}` hold gitignored owner state with **no git history ‚Äî a delete is permanent data loss**. Treat them as append-only/managed: completed queue cards are MOVED to `done/`, never `rm`'d; test artifacts you create there are removed file-by-file (never the directory); NEVER run `rm -r`, `git clean -x` or a "reset/recreate the folder" step against them. If a task seems to require it, stop and ask the owner. This state is consolidated out of the repo in the **`pandacorp-vault`** sibling (`/Users/Shared/Proyectos/pandacorp-vault/`: the `personal.git` overlay that versions the factory's gitignored personal files + daily `backups/`); secrets/keys stay in `~/.config/pandacorp/`. Canonical: `factory/standards/infra.md` ‚Üí "pandacorp-vault".
- **Modifying the factory from any runtime.** Every piece has ONE source of truth (skills are symlinked, the manual is imported, the Codex agent TOMLs are the only generated copy). Before touching skills/agents/manifests, read the modification guide in `factory/standards/agent-portability.md` ¬ßMaintenance so the change lands in both worlds.
