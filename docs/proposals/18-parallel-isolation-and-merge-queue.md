# Proposal 18 — Parallel-agent isolation + merge queue (the gate stops flapping)

> Status: IN_DRAFT (2026-06-26).
> Origin: owner-driven. The owner routinely fans out **several agents in parallel, by hand, OUTSIDE `/implement`**, to advance multiple things at once. They keep hitting a flapping gate: agent A finishes clean, but its Stop-hook `verify.sh` goes RED because of agent B's in-flight WIP — and A has no context on B, sometimes doesn't even know B exists, and tries to "fix" code it never wrote. This proposal was designed and **verified live**: while we discussed it, this very session's Stop hook failed on a `documentacion`-mobile visual diff caused entirely by a parallel session's uncommitted `globals.css`/`Banner` changes — zero edits from this session. That incident is the spec.

## The one insight

The shared gate flaps because the gate is **whole-program** (`tsc`, `knip`, `madge`, the Playwright visual baselines all read the *entire* tree), so any in-flight WIP from any session contaminates everyone's gate. You cannot scope your way out of this in TypeScript: `tsc --noEmit` is a whole-program check, so a broken type B introduced anywhere fails A's gate even when A's *target* files are disjoint. **The only thing that makes "my gate sees only my work" true is isolation — B's file literally not on A's disk.**

`/implement` already has this guarantee **by construction**: its workers run on one shared tree but are kept disjoint by declared `artifacts` and serialized by a single-writer commit (DR-060/DR-086), and the per-turn Stop gate stands aside while the build is active (DR-063). The owner's pain is the **other** parallelism: *uncoordinated, manually-launched sessions* on one repo (DR-093). This proposal operationalizes DR-093 into a turnkey flow — **worktree-per-agent + a serialized merge queue + a "not-mine → report, don't fix" rule** — and lifts it into a **propagable standard** so every Pandacorp project inherits it, not just Mission Control.

## What is ALREADY shipped (reuse, do not rebuild)

Verified against the repo this session:

- **`factory/ports.yaml`** — a 10-port block per project, and its offset map *already anticipates worktrees*: `+0 app (agent worktree) · +1 app (review worktree) · +2 Postgres agent · +3 Postgres review · +4 Redis agent · +5 Redis review · +6..+9 spare`. The scheme was built worktree-aware for **one** agent + one review slot.
- **`build.lock` freshness pattern** — `verify-before-stop.sh:31` no-ops while `.pandacorp/run/build.lock` is fresh (`find -mmin -10`); the supervisor touches it on a ~120 s heartbeat (`supervisor.sh:28`). This is the exact mechanism to clone for a **`merge.lock`**.
- **`verify.sh --since <sha>`** — the gate already runs `vitest` scoped to changed tests while `biome/tsc/knip/madge` stay global; the full suite runs at close-out.
- **`PANDACORP_FACTORY_ROOT` override** — `lib/config/config.ts:16-25` resolves the factory root from the env var, falling back to `cwd/..`. This is what lets a worktree at a non-standard path still read real factory data.
- **SOPS+age secrets live OUTSIDE any repo** (`~/.config/pandacorp/secrets/…`, `external-services.md:41,95`). They are **never** in the tree, so they don't need to "travel" to a worktree — its bootstrap runs the same `sops exec-env` as the main checkout.
- **DR-093** already states the policy ("isolate or serialize concurrent sessions; append-only the shared logs"); **DR-090/DR-089** fix where derived worktrees live. This proposal turns the policy into tooling.

## Verified root cause (the live incident)

`git status` this session: 18 modified + 5 untracked files (`globals.css`, `Banner.tsx`, `IdeaPitch/`, `SpecDigest/`, `overlay-freshness/`, `pitch.ts`, the `styling-and-ui.md` rule, the `documentacion-mobile` baseline) — **all from a parallel session**, none from this one. The failing test was `visual · documentacion (/manual)` at a 0.03 px diff. `/manual`'s own content was untouched; the diff came from the parallel session's **global** `globals.css`/`Banner` edits bleeding into every page's render. The Stop hook ran the full strict gate (correct: no build active, so DR-063 doesn't apply) and blamed the session that happened to finish — not the one that owns the WIP. **This is whole-program-gate cross-contamination, and it is unfixable without isolation.**

## The improvements

### P0 — highest leverage

- [ ] **P0.1 — Self-isolation as the default session behavior (NO orchestrator).** The owner's real workflow is *N independent natural conversations* ("make a plan… ya, ejecuta"), not one launcher fanning out agents — so isolation must be a **per-session reflex, not an upfront parametrized call**. Encode a **standing rule in the project overlay** (CLAUDE.md / guide / AGENTS): *a session about to make a non-trivial code change works in a worktree* — triggered on the **same frontier as the existing write-gate** (a change significant enough to route through a skill is significant enough to isolate; micro-edits — typo/comment/config — stay in-tree, reusing a boundary the overlay already defines, not a new one). The agent calls **`EnterWorktree` itself** the moment it transitions from planning to executing ("ya, ejecuta" → enter the worktree *first*, then edit, so no uncommitted work is stranded by the branch-from-a-commit base) — exactly the native tool's sanctioned path ("project instructions direct it"). Bootstrap on entry: `pnpm install` (sub-second hardlink from the shared store; **never** symlink `node_modules`; `.next` per-worktree) → worktree-local `.claude/launch.json` on **autoPort** → secrets via the same out-of-repo `sops exec-env` → `PANDACORP_FACTORY_ROOT` (MC) / per-worktree DB (stateful, P1.3). Ephemeral worktrees may use Claude Code's native `.claude/worktrees/` (they are *meant* to be swept); the merge-back (P0.2) removes them. **Why default-on, not detect-then-isolate:** start-time detection is racy (two sessions each see "solo", then collide), and — decisively — a whole-program gate (`tsc`/visual) only goes **green** for a session whose foreign WIP isn't on disk, so isolation is what lets each parallel session actually *reach done*, not merely "be blamelessly blocked." Practical ceiling ~4–5 concurrent agents before machine/rate-limit pressure dominates (external research). The "not-mine" rule (P0.3) is the **fallback** for the in-tree micro-edit case, not the primary mechanism.
- [ ] **P0.2 — Serialized merge queue (`merge.lock`).** Reunite branches through one serialized writer, mirroring the single-writer commit of DR-060 but at *merge* granularity. It runs **automatically when the session's worktree work greens** (the agent's own gate passes → it joins the queue), so the owner just sees "done and merged" — no manual step. Per branch, behind an atomic, freshness-stamped `merge.lock` (atomic `mkdir`/`O_EXCL`, the portable lock; clone the `build.lock` `-mmin` reclaim): **rebase onto the current `main` tip → run the integration gate (`verify.sh`) on the combined result → fast-forward merge if green → `ExitWorktree remove`.** Because each merge advances `main`, the next holder rebases onto the *new* tip — the merge-queue invariant ("frozen + up-to-date target"). A textual conflict git's 3-way merge can't resolve (`git merge`/`rebase` exits non-zero, `<<<<<<<` markers) → **hand back, never auto-force**; the agent may auto-resolve only when it can read both diffs/intents **and** the gate then passes.
- [ ] **P0.3 — "Not-mine → report, don't fix" rule (the rule that would have saved this session).** Stamp the gate state at session/worktree start; on a Stop-hook RED, the agent attributes failures to **its own changed files vs. a foreign/pre-existing red** (diff against the start stamp + `git status` ownership). A failure outside the session's scope is **reported and the agent stops — it never edits or `--update-snapshots`-masks another session's WIP** (DR-093's no-sweep rule, made executable). Inside a worktree this is mostly moot (the foreign WIP isn't on disk); it is the safety net for the shared-tree fallback.

### P1 — bring to standard (propagable to every project)

- [ ] **P1.1 — The per-project parallel contract (3 declarations).** A project becomes parallel-safe by declaring, in its blueprint/overlay: **(a) ports** — which services need a *fixed* reserved port (OAuth callbacks, webhooks, CORS allowlists, the canonical/always-on instance, stateful backing services) vs. which go **autoPort** (every ephemeral/parallel copy); **(b) a reproducible bootstrap** — one command that reconstitutes everything not tracked (install + secrets + migrate + worktree launch.json); **(c) per-worktree state isolation** — the DB strategy (P1.3) or "stateless, none" (the MC case). Injected like any other standard; lives as a new section in `build-orchestration.md`.
- [ ] **P1.2 — autoPort for ephemeral instances; reserved offsets stay for the canonical/stateful/callback ones.** Extend `ports.yaml` semantics: the reserved block keeps the canonical agent+review+DB+Redis slots; **N parallel ephemeral agents use autoPort** (OS-assigned `PORT`, app must read it — remove hardcoded `--port` flags). Fix MC's `launch.json`: the ephemeral entries (`mc-preview`) drop `--port 3100` and set `autoPort: true`; the canonical `mission-control` (internal tool, no callbacks) may also go autoPort to end the port-3000 collision class.
- [ ] **P1.3 — DB isolation default for stateful projects: `CREATE DATABASE … TEMPLATE`.** On one local Postgres, each worktree points `DATABASE_URL` at `<app>_<slug>` cloned in sub-seconds from a connectionless `<app>_template` (no migration re-run, full isolation, far lighter than a container per worktree). Hard caveat to encode: the template must have **zero active connections** during the clone, so branch-creation serializes. `testcontainers`/`pg_tmp` own the *test* layer; Neon branching is the *CI/preview* option. **MC is exempt — stateless: it references real factory data via `PANDACORP_FACTORY_ROOT` instead of copying it** (external state is *referenced*, owned state is *isolated*).

### P2 — reinforcements

- [ ] **P2.1 — Active-worktree manifest (the "see it" part).** A lightweight registry under `.pandacorp/run/` (gitignored) listing live worktrees — branch, port, task, started-at — written on spawn, read by each agent on start (awareness: "2 other agents active, here is their scope"), and surfaced in Mission Control. Turns "I have to tell each agent another one exists" into something automatic.
- [ ] **P2.2 — Stop-hook attribution, not just phase-awareness.** Today the Stop hook stands aside only during an active build (`build.lock`). Extend it: a session that changed nothing relevant must not be blamed for a foreign red — pair with P0.3's start-stamp so the gate attributes RED to the owning session.
- [ ] **P2.3 — The honest semantic-conflict backstop.** The integration gate on the merged result is the **arbiter** of textual+type+test correctness, but it cannot close the **semantic-conflict** gap (two changes green alone, green merged, wrong together — CI checks code against *your tests*, not an independent oracle). Encode the standard mitigations: strong typing (free fast semantic check), **seam tests on the contracts two changes share**, small/frequent merges (shrink the divergence window), and optional human review of the merged diff on same-region overlaps. Auto-merge holds for disjoint+green; it **escalates** on same-region or gate-red. State this as an accepted limit, not a gap to engineer away.

## How each enters the workflow

- **DR + standard** → `/pandacorp:learn` + owner gate promotes the new **DR-096** (draft below) and the new `build-orchestration.md` section. (Not written in this proposal: a parallel session is currently active and `registry.yaml`/`decision-log.md`/`build-orchestration.md` are append-heavy shared files — per DR-093 they are touched one session at a time. The drafts live here, ready to promote when the tree is quiet.)
- **Tooling** → **no orchestrator skill.** P0.1 is a **standing self-isolation rule in the project overlay** — the agent calls `EnterWorktree` itself on the write-gate frontier; the merge-back (P0.2) runs automatically when its worktree work greens, reusing the `build.lock` pattern as a `merge.lock`; the "not-mine" rule (P0.3) is wired into the implementer/agent prompt + `verify-before-stop.sh`. An optional thin `/pandacorp:merge` escape hatch covers a manual re-run (e.g. resolving an escalated conflict).
- **Per-project** → the 3-declaration contract (P1.1) injected at `/pandacorp:scaffold`/`:blueprint`, which already materialize `ports.yaml` → `launch.json`/`.env`.

## The through-line

Isolation makes **"my gate sees only my work"** literally true (B's WIP isn't on A's disk). The merge queue makes **"all our work composes"** a controlled, owner-free step — the only place cross-session conflicts surface, exactly where you want them. The "not-mine → report, don't fix" rule makes a foreign red **harmless** instead of a confused agent's emergency. It is the same guarantee `/implement` enjoys by construction, extended to *uncoordinated* parallel sessions, and shipped as a standard every project inherits — with the genuinely hard spots named honestly: **DB isolation has no local free lunch**, and **the semantic-conflict ceiling cannot be automated away.**
