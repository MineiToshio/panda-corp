# Proposal 14 — Fidelity-to-prototype gate + the app shell as a first-class, never-orphaned FRD (DR-075)

> Status: DRAFT for owner review (red-team pending). Author: factory supervisor, 2026-06-21.
> Origin: Mission Control shipped with **no global navigation menu / app shell** AND it passed the
> visual gate GREEN. Two factory-level root causes, both the highest-leverage gaps left after DR-074.
> Continuation of `mc-design-fidelity-failure` + `mc-session-2026-06-21`; builds on DR-054/055/056/072/074.

## The problem (evidenced in MC)

The prototype is a SPA with a **persistent shell**: a topbar + primary nav (Inicio · Tablero · Portfolio · Documentación), rendered on every screen ("header/nav SIEMPRE primero", prototype `index.html:646,649,1464`). The shipped Mission Control has **none of it** — `layout.tsx` mounts only `GuildBar`(XP) + `ProposalsBadge` + `children`; there is no nav component, no FRD, no work order for the app shell. The whole guild menu is simply absent. And it shipped **112/112 VERIFIED GREEN**. Two distinct, factory-level root causes:

1. **The sharding drops the chrome (structural).** `design` Step 8 (DR-056) shards the approved whole-app prototype into one self-contained mock **per UI FRD** ("give each work order only its own screen"). The **global shell** — topbar, primary nav, footer, the layout frame every screen mounts into — belongs to **no single FRD**, so it was never extracted into any mock, never written as an FRD, never assigned a work order. The per-FRD reviewer then judged each surface against its own shard (which also has no shell) and never asked "does the whole app shell match the prototype?". The chrome falls through the cracks of a per-FRD decomposition.

2. **The gate proves CONSISTENCY, not FIDELITY (enforcement).** The deterministic Layer-A visual gate (`e2e/visual.spec.ts`, `toHaveScreenshot`) matches each route against its **own committed baseline**. The build rendered every page menu-less → the baselines were menu-less → every diff was green. A whole missing nav (and, separately, the non-responsiveness DR-074 addressed) sailed through. `toHaveScreenshot` is *structurally* a self-consistency regression guard — it can never be a fidelity check, because you cannot pixel-diff a Next.js render against a hand-authored HTML prototype. The only fidelity judgment today is the reviewer's Layer-B VLM lens — but DR-072 made it advisory for everything except a *gross* structural mismatch, and it operates **per-surface against the shard**, never **whole-app against the prototype**. So "the whole shell is missing" was neither mechanically caught nor judged.

These are the same class as `mc-design-fidelity-failure`: a passing gate proved the build was self-consistent, not that it matched the approved design. DR-056/072 fixed *per-surface* fidelity; the **global shell** and the **consistency-not-fidelity** gate were never addressed.

## The change — two load-bearing pieces (+ a small responsive tie-in)

### Piece B — the app shell is an explicit, never-orphaned FRD (the structural fix)

When `design` shards a **whole-app** prototype, the **persistent shell is captured as its own explicit FRD**, built FIRST, into which every per-FRD screen mounts — never assumed to emerge from the surface FRDs.

- **`design` Step 8 — new sub-step 8a "extract the shell BEFORE sharding surfaces".** Identify the prototype's persistent chrome (what renders on EVERY screen: header/topbar, primary nav + active-state, footer, the global layout frame) and emit it as an explicit reserved FRD `frd-00-app-shell`:
  - `docs/frds/frd-00-app-shell/frd.md` — the user contract: a persistent global nav reaching **every top-level destination**, the active-section state, header/footer, and the responsive behaviour for the declared `target_platforms` (topbar on desktop; a declared drawer/hamburger on mobile).
  - `docs/frds/frd-00-app-shell/mocks/` — the global chrome extracted from `docs/design/prototype/`.
  - frontmatter `ui: true`, `visual_source: prototype`.
  - The nav's destination list is **derived from the route manifest** (`e2e/routes.ts` `SURFACES` → one nav entry per top-level surface) so nav ↔ routes stay coherent by construction.
- **Per-surface mocks show the screen as it mounts INSIDE the shell** (context), but the shell itself is owned by `frd-00`. Maps cleanly onto Next.js App Router: `app/layout.tsx` = the shell (FRD-00), `app/<route>/page.tsx` = each surface FRD.
- **`blueprint` / Build Plan — the shell builds right after the foundation wave, before any surface fans out.** The shell depends on the foundation primitives (`Button`, `Nav`, …) and **everything depends on the shell** (it's the layout each surface mounts into). So: foundation primitives → **shell FRD** → surface FRDs. The surface FRDs declare a `Dependency` on the shell FRD.
- **"This app has a shell" is detected, not forced.** A whole-app prototype with persistent chrome → emit `frd-00`. A single-screen tool, a landing page, an API/scraper, a prototype with **no** persistent nav → **no shell FRD** (and the mechanical gate below no-ops). Author-declared via the presence/absence of a non-empty shell mock + nav contract — never a heuristic that forces a menu onto an app that has none.

### Piece A — verify FIDELITY to the prototype, not self-baseline (the enforcement fix), in two layers

**A1 — a MECHANICAL shell-presence gate (shippable, fail-closed, deterministic).** A new VERBATIM stack template `e2e/shell.spec.ts` (joins the DR-059/074 byte-diffed/conformance-checked set) + a PER-PROJECT seed `e2e/shell.ts` (the **nav contract**: the persistent-shell landmark selector + the list of `{label, href}` top-level destinations + an optional `data-nav-toggle` selector for mobile). On **every** route in `routes.ts`, it asserts:
  - the persistent shell landmark is present (e.g. `[data-app-shell]` / `<nav>`), AND
  - it contains a link to **every** declared destination, AND
  - at the mobile width, **either** the nav is visible **or** the declared `data-nav-toggle` reveals it (the author-declared escape hatch — the exact DR-074 `data-scroll-x="intentional"` pattern, never a heuristic).

  This catches the precise MC failure — "a whole nav is missing on every page" — **deterministically and fail-closed**: it **cannot** pass green the way `toHaveScreenshot`-against-a-menu-less-baseline did. It ships and retrofits exactly like the DR-074 responsive gate: installed by `blueprint`, byte-diff-conformance-checked by `upgrade`; `shell.ts` is per-project (created-if-missing, NEVER byte-diffed, like `routes.ts`); `verify.sh` gains a fail-closed `test:shell` shim. When the project has **no** shell (empty `shell.ts`), it's a **vacuous pass** (fail-closed harness, no premature lock) — the same pattern as `routes.ts`/`SURFACES`.

**A2 — a SEMANTIC whole-app fidelity check (the reviewer, gross-structural blocks).** The reviewer's runtime/visual lens gains an explicit **WHOLE-APP** comparison against the canonical prototype `docs/design/prototype/` (not only per-surface shards): is the persistent shell present and recognizably the prototype's chrome? is every top-level destination in the prototype's nav reachable? does each route mount **inside** the shell? A **missing global chrome / a missing top-level destination / a route that renders outside the shell** is a **GROSS structural mismatch → BLOCKS** (DR-072 preserved exactly: gross-structural blocks, pixel/spacing/shade nits stay advisory on the punch-list). The shell FRD's own gate compares the rendered shell to the prototype's chrome; the end-of-build Visual QA pass adds a whole-app shell sweep.

Net for Piece A: we do **not** undo DR-072 by making pixel-fidelity block again. We (i) ship a **mechanical** shell-presence assertion so the specific class of failure (an entire region of the prototype missing) is fail-closed and can't pass green; (ii) widen the reviewer's *gross-structural* judgment from per-surface to **whole-app against the prototype**.

### Piece C — desktop/mobile/both, finished (the small DR-074 tie-in)

DR-074 shipped `target_platforms` + the responsive gate but (per its red team) **dropped** proposal 13's "decide at definition" enforcement and the ADOPT-path coverage check. The shell is the natural place to finish it, because **the shell is the one surface that is on every page** — and MC's shell was both missing AND non-responsive:

- The **shell FRD carries the responsive AC** for the declared `target_platforms` (desktop topbar vs mobile drawer/hamburger); its mock covers the declared platforms. The shell is the cross-cutting responsive surface.
- **`design` runs the prototype-covers-target check** at the start of sharding (proposal 13 risk 4, deferred by DR-074): if `target_platforms` includes mobile but the prototype is desktop-only, **FLAG it** (the prototype/shell need mobile layouts before sharding) — a **warn**, not a hard block.

## Feasibility

- **A1 (mechanical shell gate): LOW-MEDIUM** — a small verbatim `shell.spec.ts` + a seed `shell.ts`, the exact DR-059/074 e2e mechanism that already ships and retrofits.
- **A2 (semantic whole-app reviewer): LOW** — reviewer prose; the prototype is already committed at `docs/design/prototype/`.
- **B (shell FRD): MEDIUM** — the largest change: a new shard sub-step, a reserved FRD, Build-Plan ordering. But it reuses the foundation-wave machinery (DR-057) and the FRD module structure.
- **C (responsive shell): LOW** — additive ACs on the shell FRD + a design-phase warn check.

## Pre-identified risks (input for the red team)

1. **Over-engineering / wrong unit.** Is the shell a full **FRD** (with its own gate) the right unit, or should it be a **foundation work order** in the existing DR-057 foundation wave (lighter, no new FRD-numbering surface)? Is the mechanical A1 gate **load-bearing**, or ceremony on top of the A2 reviewer? Does any of this repeat the DR-074 lesson ("the failure was non-enforcement, not a missing artifact")? Note the distinguishing fact: unlike DR-074, there is **no existing rule** that the shell be built — the sharding genuinely *drops* it — so this is a real structural gap, not a non-enforcement of an existing rule. Pressure-test that claim.
2. **Apps with no global shell (false-positive surface).** A single-screen tool / landing / API / scraper has no persistent nav. The shell FRD and the A1 gate must **no-op gracefully** and never force a menu onto an app that has none. Is "presence of a non-empty shell mock + nav contract" the right author-declared switch, or will it mis-fire (a one-page app the prototype still wraps in a header)?
3. **Does A2 re-introduce the DR-072 churn?** A2 makes a *gross* whole-app structural mismatch block. The line between "missing shell/destination = gross = block" and "shell slightly off = nit = advisory" must be crisp and non-gameable, or we resurrect the build-never-finishes failure DR-072 fixed. Is the boundary defensible, or does "the shell looks a bit off" leak into blocking?
4. **A1 robustness — false-positives & false-negatives (the DR-074-B lens).** Nav present but links point to `/` / to nowhere; nav present on desktop but absent on mobile; nav rendered client-side and not yet mounted at assertion time (SSR/hydration timing); a legit app whose nav genuinely lives in a non-`<nav>` element. Will `shell.spec.ts` false-red a correct app or false-green the MC failure's variants? Compare against DR-074's settle-first/author-declared hardening.
5. **Retrofit (the DR-074-C lens).** MC + brownfield have no shell FRD and no nav contract. Migration: seed an **empty** `shell.ts` → vacuous pass (no red-lock), filled when MC is re-anchored (next step), exactly the conservative DR-074 back-fill. Does `adopt` need to capture the shell? Does `upgrade` install `shell.spec.ts` + seed `shell.ts` cleanly into a project (MC) whose build is mid-flight / already shipped?
6. **FRD-numbering / spec-flow collision.** A **design-emitted** `frd-00-app-shell` lands *after* `spec` already numbered the FRDs from the PRD. Does `frd-00` collide with the PM's FRD list, the traceability spine (`REQ-NN-MMM` → `WO-NN-MMM`), the status rollups, the change queue, or the engine's plan? Is a reserved `frd-00` (or a clearly-namespaced id) safe, or does the shell need to originate in `spec`/the PRD instead of `design`?

---

# RED-TEAM RESULTS (3 adversarial agents, opus) + v2 design (DR-075)

Lenses: **A** = minimalism/over-engineering, **B** = gate reliability (false +/−), **C** = ship/retrofit/integration. The three converged hard. **Unanimous: v1 over-engineered the same way DR-074-v1 was** — the load-bearing fix is ONE mechanical gate (A1) + putting the shell in the EXISTING foundation wave; the rest (full FRD, A2-as-blocker, Piece C) is ceremony to cut, and A1's spec had reliability holes that would let the MC failure slip through or false-red correct apps.

## Findings

1. **Shell as a full FRD is the WRONG UNIT (A+C, decisive).** The factory ALREADY has the right primitive: the **foundation wave** (DR-057), and `build-orchestration.md` §3 *explicitly names* "the layout shells" as foundation. A design-emitted `frd-00` is an orphan in the traceability spine (`spec` roots every FRD in a PRD `REQ-NN-MMM`; `frd-00` traces to nothing; `work-orders` requires every `AC-NN-MMM.K` covered by a WO), and the engine has **no FRD-level "build between foundation and surfaces" primitive** — foundation-first is keyed off `foundation: true` on **work orders** (`pandacorp-build.js` foundation wave + completeness gate), not FRDs. **Fix:** the shell is a `foundation: true` **work order** (`artifacts: [app/layout.tsx, components/.../Nav…]`), one row in `docs/design/components.md`, built first by the existing foundation-first wave + completeness gate — **zero engine change, no new numbering, no spine collision** (kills risk 6 entirely). Surfaces mount into `app/layout.tsx`. Better-enforced than a sibling FRD (the completeness gate refuses surfaces until the foundation is green), and cheaper.
2. **The "no existing rule" claim is half-true (A).** DR-062 already presupposes a nav ("the page H1 **equals its nav label**", "the menu item the owner clicked"); the reviewer was already told to check H1↔nav-label cohesion — and missed that MC's pages had H1s with **no nav to match**. So the enforcement gap is partly non-enforcement (the A1 mechanical gate fixes it). The genuinely-new gap is narrow: **the sharding drops the chrome from the artifact set** → the one structural fix is putting the shell in the foundation wave. Not a new FRD class + a fourth fidelity standard.
3. **A2 (semantic whole-app blocking gate) is mostly redundant + re-opens DR-072 churn (A+B).** A2's crisp blocking cases (missing chrome / missing destination / route-outside-shell) are EXACTLY what A1 proves deterministically; A2's non-mechanical remainder ("is it *recognizably* the prototype's chrome?") is the soft VLM judgment DR-072 deliberately demoted to advisory — making it a blocker resurrects the WO-07-005 non-convergence. **Fix:** A2 is NOT a new blocking layer. The reviewer gets ONE sentence folded into its existing (advisory) whole-app/DR-062 lens; the **block** is delegated entirely to A1 (deterministic). DR-072 untouched.
4. **A1's nav contract must be PROTOTYPE-anchored + author-declared, NOT derived from `routes.ts` (B, decisive).** "One nav entry per SURFACE" is provably wrong on MC's own data: `routes.ts` has **8 SURFACES**, the prototype's `topbar()` has **6 nav tabs** — `workspace` (reached via Portfolio) and `configuration` (fused into the Manual) are surfaces but **not** top-level nav destinations. Deriving the contract from SURFACES → (a) false-reds the correct app (demands a topbar link to `/projects/…`), or (b) forces the app to ADD links the prototype lacks → the gate *causes* infidelity; and (c) anchors the contract to the app's own routes = the **consistency-not-fidelity trap moved up one level** (a build that drops "Documentación" and trims `routes.ts` to match passes green). **Fix:** `shell.ts` carries author-declared, prototype-anchored `NAV_DESTINATIONS` (`{label, path}`, seeded by `design` from the prototype's persistent-nav block, frozen like a baseline) + `SHELL_EXEMPT` (routes legitimately without the shell). `routes.ts` is used ONLY to enumerate routes-to-assert and to confirm each path 200s — never as the source of the expected nav.
5. **A1 must iterate `SURFACES` (minus exempt), NOT `BLESSED` (C, decisive — "the whole ballgame").** Every shipped spec iterates `BLESSED` (`SURFACES.filter(s=>s.blessed)`). The MC failure is a route blessed (shipped green) *while the shell is absent* — so a `BLESSED`-iterating shell gate inherits the exact blind spot. **Fix:** iterate `SURFACES`, gated by the **non-empty `shell.ts`** switch (the `_target.ts`/`TARGETS_MOBILE` pattern, NOT the `BLESSED` filter): empty contract ⇒ vacuous pass (no premature red-lock mid-build), non-empty ⇒ assert on every non-exempt route (the missing-shell route goes RED even before blessing — fail-closed against the MC failure).
6. **A1 false-NEGATIVES — presence-in-DOM ≠ reachable (B).** A nav with every `href="/"`/`#`/404, an empty landmark, or a `display:none`/occluded/off-canvas nav all pass a naïve "landmark present + a link per destination". **Fix:** assert per destination, **scoped INSIDE the shell landmark** (never page-wide), a **visible** link (`toBeVisible`) whose `href` **equals the expected path** (not `/`/`#`), and each path **2xx-resolves** once per suite.
7. **A1 false-POSITIVES — hydration/SSE timing (B).** Asserting at `domcontentloaded` on a client-rendered nav flakes; `networkidle` **hangs forever** on MC's live SSE routes (DR-071). **Fix (mirror `_responsive-helper.ts` exactly):** per route `page.route('**/api/live**', abort)` → `goto(domcontentloaded)` → `fonts.ready` → `main` visible → **then** web-first auto-retrying `toBeVisible` assertions (absorb hydration). Never `networkidle`, never bare `.count()`, never `waitForTimeout`.
8. **Piece C is scope creep DR-074 deliberately cut (A+C).** The "prototype-covers-target" warn check was cut by DR-074's own red team as non-enforcing ceremony; smuggling it back under DR-075 re-litigates a settled decision. The shell's responsiveness is **already** covered — once the shell exists it's one more `SURFACES` route the shipped responsive gate (DR-074) asserts no-overflow/tap-target/occlusion on. **Fix:** DROP Piece C. Keep ONLY the shell-specific "mobile nav reachable (visible or via the declared toggle)" assertion, which rides INSIDE A1 reading `_target.ts` `TARGETS_MOBILE` (the responsive gate does NOT check nav reachability).
9. **Documentation footprint was missing (C).** v1's feasibility named no decision-logs/registry/Manual. **Fix:** registry DR-075 + `plugin/`+`factory/` decision-logs + `build-orchestration.md` (foundation wave names the shell) + `quality.md` (the shell gate) + the MC Manual narrative (deferred + named, like prior DRs) + `OVERLAY_VERSION`/`plugin.json` MINOR bump.

## v2 — the minimal, robust design (DR-075)

**1. The shell is a `foundation: true` WORK ORDER, not a new FRD — captured in BOTH the greenfield (`design`) AND the brownfield (`adopt`) shard paths.** The shell-capture sub-step is the SAME logic, wired into every place a whole-app prototype is sharded "never the whole app" (the exact line that drops the chrome):
- **`design` Step 8 → new sub-step 8a "capture the persistent shell BEFORE sharding"** (greenfield): identify the prototype's chrome (topbar/nav + active-state, header/footer, the layout frame) and (a) add `AppShell`+`Nav` to `docs/design/components.md` (the foundation the build implements first; `artifacts` include `app/layout.tsx`), and (b) seed `e2e/shell.ts` by reading the prototype's persistent-nav block. `blueprint`'s Build Plan schedules it in the existing foundation wave (DR-057). Surfaces mount into `app/layout.tsx`. No `frd-00`, no engine change.
- **`adopt` Step 6 (the ADOPT-VISUAL chain) → the SAME shell-capture, before/with the shard (6.3)** (brownfield — *this is where MC actually lost it*: `adopt` shards "only that FRD's screens, never the whole app" with no shell-capture step). Because `adopt` documents and does not build code, it branches on the existing code:
  - existing code **has** a real shell (a `layout.tsx`/header with nav) → capture it as the foundation component in `components.md` **with its real path** + seed `shell.ts` **filled** from the prototype's nav (the gate then verifies the existing shell).
  - existing code **lacks** the shell the prototype shows (the MC case) → record it as an **owed** foundation component + a **Group-B functional-reconciliation gap** (DR-064: "FRD/prototype wants it, code lacks it") so it gets built on the re-anchor, and seed `shell.ts` **empty** (vacuous pass — no red-lock; it is filled in the SAME change that builds the shell).
  - no prototype / no persistent chrome → empty `shell.ts`, no-op.

If the prototype has **no** persistent chrome (single-screen tool / landing / API / scraper) in either path → no shell WO, empty `shell.ts`, the gate no-ops. **This is the fix for the specific reason MC lost the shell** — the brownfield adopt path, not just the greenfield design path, now forces the shell into the foundation/inventory and can't silently shard it away.

**2. A1 — the mechanical shell-presence gate (the load-bearing fix), hardened per findings 4–7:**
- NEW VERBATIM `e2e/shell.spec.ts` → joins the DR-059/074 byte-diffed conformance set (now **six** verbatim e2e files). Iterates `SURFACES` minus `SHELL_EXEMPT`, gated by the non-empty `shell.ts` switch. Settle-first (abort live → domcontentloaded → fonts.ready → main visible) then web-first assertions: shell landmark visible on every non-exempt route; every `NAV_DESTINATION` a visible in-shell link with `href` === its path; each path 2xx-resolves; mobile (via `_target.ts` `TARGETS_MOBILE`) nav visible or the declared toggle reveals it.
- NEW PER-PROJECT seed `e2e/shell.ts` (created-if-missing, NEVER byte-diffed, like `routes.ts`/`_target.ts`): author-declared, prototype-anchored `SHELL_SELECTOR`, `NAV_DESTINATIONS` (`{label, path}` — from the prototype, NOT SURFACES), `SHELL_EXEMPT`, `NAV_TOGGLE`. Empty `NAV_DESTINATIONS` ⇒ vacuous pass.
- `verify.sh` gains a fail-closed `test:shell` shim; `test:shell` script added.

**3. A2 → ONE advisory reviewer sentence, not a gate.** The reviewer's existing whole-app/DR-062 lens: "verify the prototype's persistent shell/nav is present app-wide and every top-level destination is reachable." The **block** is A1 (deterministic); shell *resemblance* (placement/order/active-state/branding) is advisory punch-list. DR-072 preserved.

**4. Piece C dropped.** The shell rides the shipped responsive gate as a normal route; the mobile-nav-reachable check lives inside A1.

**5. Retrofit, concrete + sequenced.** `blueprint` step 6/7: install `shell.spec.ts` (verbatim) + seed `shell.ts` (if missing) + the `test:shell` script/shim. `upgrade`: add `shell.spec.ts` to the six-file byte-diffed set, `shell.ts` to create-if-missing-never-byte-diff, `test:shell` to the required scripts; **back-fill an EMPTY `shell.ts`** (vacuous pass — never filled; the conservative analogue of `target_platforms: desktop`). `adopt`: the shell-capture branch in v2-point-1 (filled `shell.ts` if the code already has a shell; empty + an owed Group-B gap if it doesn't). **Sequencing (MC):** the re-anchor that FILLS `shell.ts` must land **together with** the shell foundation WO's first build — a filled contract on an app with no shell red-locks all commits.

**6. Documentation:** registry DR-075; `plugin/`+`factory/` decision-logs; `factory/standards/build-orchestration.md` (§3 foundation wave names the app shell) + `quality.md` (the shell gate in the gates list); MC Manual narrative (deferred, named); `OVERLAY_VERSION`→bump, `plugin.json`→v8.36.0 (MINOR — new e2e capability).

Net: fixes the MC failure (a whole nav missing, passed green) at the root — the shell becomes a built, gated foundation artifact AND a deterministic fail-closed presence gate anchored to the prototype — using the existing foundation-wave + DR-059/074 e2e machinery, with no engine change, no new FRD class, and DR-072 untouched. Moving parts down from v1's ~10 to: one e2e template + one seed + one shim, one design sub-step, one reviewer sentence, the standard/registry/log/Manual updates.
